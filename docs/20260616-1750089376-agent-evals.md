# 解密 AI Agent 的评估

## 引言

好的评估能帮助团队更自信地发布 AI agent。没有评估，团队很容易陷入被动的循环——只在生产环境中发现问题，修复一个故障又引发另一个。评估让问题和行为变化在影响用户之前变得可见，而且它们的价值会在 agent 的整个生命周期中不断累积。

正如我们在[构建有效 agent](https://www.anthropic.com/engineering/building-effective-agents)中所述，agent 在多轮交互中运行：调用工具、修改状态、并根据中间结果进行适应。正是这些使 AI agent 有用的能力——自主性、智能性和灵活性——也使它们更难被评估。

通过我们的内部工作以及与处于 agent 开发前沿的客户合作，我们学会了如何为 agent 设计更严谨、更有用的评估。以下是在各种 agent 架构和用例的实际部署中行之有效的方法。

## 评估的结构

**评估**（"eval"）是对 AI 系统的测试：给 AI 一个输入，然后对其输出应用评分逻辑来衡量成功与否。在本文中，我们关注的是可以在开发阶段无需真实用户即可运行的**自动化评估**。

**单轮评估**比较直接：一个提示、一个响应和评分逻辑。对于早期的 LLM，单轮非 agent 评估是主要的评估方法。随着 AI 能力的进步，**多轮评估**变得越来越普遍。

![简单评估和多轮评估示意图](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2Fbd42e7b2f3e9bb5218142796d3ede4816588dec0-4584x2834.png&w=3840&q=75)

*在一个简单的评估中，agent 处理一个提示，评分器检查输出是否符合预期。对于更复杂的多轮评估，一个编码 agent 接收工具、任务（在此例中是构建一个 MCP 服务器）和环境，执行"agent 循环"（工具调用和推理），并更新环境中的实现。然后评分使用单元测试来验证可工作的 MCP 服务器。*

**Agent 评估**则更加复杂。Agent 在多轮中使用工具，在环境中修改状态并不断适应——这意味着错误可以传播和累积。前沿模型还能找到超越静态评估限制的创造性解决方案。例如，Opus 4.5 通过[发现](https://x.com/siyuan_says/status/1922725584447524928)策略中的漏洞，解决了一个 [𝜏2-bench](https://arxiv.org/abs/2505.16064) 关于预订航班的题目。它按照既定规则"失败"了评估，但实际上为用户找到了一个更好的解决方案。

在构建 agent 评估时，我们使用以下定义：

- **任务**（又称**问题**或**测试用例**）是一个具有定义输入和成功标准的单次测试。
- 每次尝试完成任务就是一个**试验**。由于模型输出在运行之间会有所不同，我们运行多次试验以产生更一致的结果。
- **评分器**是对 agent 某方面表现进行评分的逻辑。一个任务可以有多个评分器，每个评分器包含多个断言（有时称为**检查**）。
- **转录**（也称为**追踪**或**轨迹**）是一次试验的完整记录，包括输出、工具调用、推理、中间结果和任何其他交互。对于 Anthropic API，这是评估运行结束时的完整 messages 数组——包含评估期间对 API 的所有调用和所有返回的响应。
- **结果**是试验结束时环境中的最终状态。一个航班预订 agent 可能在转录结尾说"您的航班已预订"，但结果则是环境的 SQL 数据库中是否存在预订记录。
- **评估框架**是端到端运行评估的基础设施。它提供指令和工具，并发运行任务，记录所有步骤，评分输出并汇总结果。
- **Agent 框架**（或**脚手架**）是使模型能够作为 agent 行事的系统：它处理输入，编排工具调用，并返回结果。当我们评估"一个 agent"时，我们评估的是框架*和*模型协同工作的效果。例如，[Claude Code](https://docs.anthropic.com/en/docs/claude-code) 是一个灵活的 agent 框架，我们通过 [Agent SDK](https://github.com/anthropics/anthropic-sdk-python) 使用其核心原语来构建我们的[长时间运行 agent 框架](https://github.com/anthropics/claude-code/blob/main/docs/long-running-agents.md)。
- **评估套件**是一组旨在衡量特定能力或行为的任务集合。套件中的任务通常共享一个广泛的目标。例如，客户支持评估套件可能测试退款、取消和升级。

## 为什么要构建评估？

当团队刚开始构建 agent 时，通过手动测试、[dogfooding](https://en.wikipedia.org/wiki/Eating_your_own_dog_food)和直觉的组合，可以走得惊人的远。更严谨的评估甚至看起来像是拖慢发布速度的开销。但在早期原型阶段之后，一旦 agent 进入生产环境并开始扩展，没有评估的开发就开始力不从心了。

转折点通常出现在用户报告 agent 在变更后感觉变差了，而团队却"盲飞"，除了猜测和验证外无法确认。缺少评估时，调试是被动的：等待投诉、手动复现、修复 bug，然后祈祷没有其他回归。团队无法区分真正的回归和噪音，无法在发布前自动测试数百个场景，也无法衡量改进。

我们多次目睹这种进展。例如，Claude Code 最初基于 Anthropic 员工和外部用户的反馈快速迭代。后来，我们添加了评估——首先是针对简明性和文件编辑等狭窄领域，然后是过度工程等更复杂的行为。这些评估帮助识别问题、指导改进、并聚焦研究-产品协作。结合生产监控、A/B 测试、用户研究等，评估为持续改进 Claude Code 提供了信号。

编写评估在 agent 生命周期的任何阶段都有用。早期，评估迫使产品团队明确 agent 的成功标准；后期，它们帮助维持一致的质量标准。

[Descript](https://www.descript.com/)的 agent 帮助用户编辑视频，因此他们围绕成功编辑工作流的三个维度构建评估：不破坏东西、按要求做事、做好事情。他们从手动评分演进到由产品团队定义标准的 LLM 评分器并进行定期人工校准，现在定期运行两个独立的套件进行质量基准测试和回归测试。[Bolt](https://bolt.new/) AI 团队在已经有广泛使用的 agent 之后才开始构建评估。在 3 个月内，他们构建了一个评估系统，用静态分析运行 agent 并对输出评分，使用浏览器 agent 测试应用，并采用 LLM 评判器来评估指令遵循等行为。

一些团队在开发开始时就创建评估；另一些则在规模扩展后、当评估成为改进 agent 的瓶颈时才添加。评估在 agent 开发初期特别有用，可以明确编码预期行为。两位工程师阅读相同的初始规格，可能对 AI 应如何处理边缘情况得出不同的理解。评估套件解决了这种模糊性。无论何时创建，评估都有助于加速开发。

评估还影响你采用新模型的速度。当更强大的模型发布时，没有评估的团队面临数周的测试，而有评估的竞争对手可以快速确定模型的优势、调整提示并在几天内升级。

一旦评估存在，你就免费获得了基线和回归测试：延迟、token 使用量、每任务成本和错误率可以在静态任务库上跟踪。评估还可以成为产品团队和研究团队之间最高带宽的沟通渠道，定义研究人员可以优化的指标。显然，评估的好处远不止跟踪回归和改进。它们的复合价值很容易被忽视，因为成本是前期可见的，而收益是后期累积的。

## 如何评估 AI Agent

我们看到当今大规模部署的几种常见 agent 类型，包括编码 agent、研究 agent、计算机使用 agent 和对话 agent。每种类型可以部署在广泛的行业中，但可以使用类似的技术进行评估。你不需要从零开始发明评估。以下部分描述了几种 agent 类型的经过验证的技术。将这些方法作为基础，然后扩展到你的领域。

### Agent 的评分器类型

Agent 评估通常组合三种类型的评分器：基于代码的、基于模型的和人工的。每个评分器评估转录或结果的某些部分。有效评估设计的一个关键组成部分是为工作选择合适的评分器。

**基于代码的评分器**

| 方法 | 优势 | 劣势 |
|------|------|------|
| • 字符串匹配检查（精确、正则、模糊等）<br>• 二元测试（fail-to-pass、pass-to-pass）<br>• 静态分析（lint、类型、安全）<br>• 结果验证<br>• 工具调用验证（使用的工具、参数）<br>• 转录分析（轮次数、token 用量） | • 快速<br>• 便宜<br>• 客观<br>• 可重现<br>• 易于调试<br>• 验证特定条件 | • 对不完全匹配预期模式的有效变体过于脆弱<br>• 缺乏细腻度<br>• 在评估更主观的任务时受限 |

**基于模型的评分器**

| 方法 | 优势 | 劣势 |
|------|------|------|
| • 评分准则评分<br>• 自然语言断言<br>• 成对比较<br>• 基于参考的评估<br>• 多评判器共识 | • 灵活<br>• 可扩展<br>• 捕捉细微差别<br>• 处理开放式任务<br>• 处理自由格式输出 | • 非确定性<br>• 比代码更昂贵<br>• 需要与人工评分器校准以确保准确性 |

**人工评分器**

| 方法 | 优势 | 劣势 |
|------|------|------|
| • 领域专家评审<br>• 众包评判<br>• 抽查采样<br>• A/B 测试<br>• 标注者间一致性 | • 黄金标准质量<br>• 匹配专家用户判断<br>• 用于校准基于模型的评分器 | • 昂贵<br>• 缓慢<br>• 通常需要大规模的人工专家资源 |

对于每个任务，评分可以是加权的（组合评分器分数必须达到阈值）、二元的（所有评分器必须通过）或混合的。

### 能力评估 vs. 回归评估

**能力评估**或**"质量"评估**问的是"这个 agent 擅长做什么？"它们应该从低通过率开始，针对 agent 感到困难的任务，给团队一座要攀登的山。

**回归评估**问的是"agent 是否仍然能处理所有以前能处理的任务？"，应该有接近 100% 的通过率。它们防止退化，因为分数下降意味着出了问题需要改进。当团队在能力评估上攀升时，同时运行回归评估以确保变更不会在其他地方引起问题也很重要。

在 agent 发布并优化之后，高通过率的能力评估可以"毕业"成为持续运行的回归套件来捕捉任何漂移。曾经衡量"我们能做到这个吗？"的任务现在衡量"我们还能可靠地做到这个吗？"

### 评估编码 Agent

**编码 agent**编写、测试和调试代码，像人类开发者一样导航代码库和运行命令。有效的现代编码 agent 评估通常依赖于明确的任务、稳定的测试环境和针对生成代码的全面测试。

确定性评分器对编码 agent 来说是天然的，因为软件通常易于评估：代码能运行吗？测试能通过吗？两个广泛使用的编码 agent 基准，[SWE-bench Verified](https://openai.com/index/introducing-swe-bench-verified/)和[Terminal-Bench](https://www.terminal-bench.com/)，遵循这种方法。SWE-bench Verified 给 agent 来自热门 Python 仓库的 GitHub issue，通过运行测试套件来评分解决方案；解决方案只有在修复失败测试而不破坏现有测试时才通过。LLM 在一年内将该评估从 40% 进步到 >80%。Terminal-Bench 走了不同的路：它测试端到端的技术任务，例如从源代码构建 Linux 内核或训练 ML 模型。

一旦你有了一套通过/失败的测试来验证编码任务的关键**结果**，通常也有用对**转录**进行评分。例如，基于启发式的代码质量规则可以基于超越通过测试的标准来评估生成的代码，带有清晰评分准则的基于模型的评分器可以评估 agent 如何调用工具或与用户交互等行为。

**示例：编码 agent 的理论评估**

考虑一个编码任务，agent 必须修复身份验证绕过漏洞。如下面的示意性 YAML 文件所示，可以使用评分器和指标来评估此 agent。

```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token
```

注意此示例展示了可用评分器的完整范围仅作说明。在实践中，编码评估通常依赖单元测试进行正确性验证和 LLM 评分准则来评估整体代码质量，仅在需要时添加额外的评分器和指标。

### 评估对话 Agent

**对话 agent**在支持、销售或辅导等领域与用户交互。与传统的聊天机器人不同，它们维护状态、使用工具，并在对话中采取行动。虽然编码和研究 agent 也可能涉及与用户的多轮交互，但对话 agent 呈现出一个独特的挑战：交互本身的质量就是你要评估的一部分。有效的对话 agent 评估通常依赖可验证的最终状态结果和同时捕捉任务完成度和交互质量的评分准则。与大多数其他评估不同，它们通常需要第二个 LLM 来模拟用户。我们在[对齐审计 agent](https://www.anthropic.com/engineering/alignment-advances-from-anthropic)中使用这种方法通过扩展的对抗性对话来对模型进行压力测试。

对话 agent 的成功可以是多维的：工单是否已解决（状态检查），是否在 <10 轮内完成（转录约束），语气是否恰当（LLM 评分准则）？两个纳入多维性的基准是 [𝜏-Bench](https://arxiv.org/abs/2505.16064) 及其继任者 [τ2-Bench](https://arxiv.org/abs/2505.16064)。这些模拟零售支持和航空预订等领域的多轮交互，一个模型扮演用户角色，agent 在真实场景中导航。

**示例：对话 agent 的理论评估**

考虑一个支持任务，agent 必须为沮丧的客户处理退款。

```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics:
      - n_turns
      - n_toolcalls
      - n_total_tokens
  - type: latency
    metrics:
      - time_to_first_token
      - output_tokens_per_sec
      - time_to_last_token
```

与编码 agent 示例一样，此任务展示了多种评分器类型仅作说明。在实践中，对话 agent 评估通常使用基于模型的评分器来评估沟通质量和目标完成度，因为许多任务——如回答问题——可能有多个"正确"解决方案。

### 评估研究 Agent

**研究 agent**收集、综合和分析信息，然后产生答案或报告等输出。与编码 agent 不同，编码 agent 可以通过单元测试提供二元通过/失败信号，而研究质量只能相对于任务来判断。什么算作"全面"、"来源充分"甚至"正确"取决于上下文：市场扫描、收购的尽职调查和科学报告各自需要不同的标准。

研究评估面临独特的挑战：专家可能对综合是否全面存在分歧，随着参考内容不断变化，真实标准也在偏移，更长、更开放的输出创造了更多出错的空间。例如，[BrowseComp](https://arxiv.org/abs/2504.12516)等基准测试 AI agent 能否在开放网络的大海捞针——问题设计为易于验证但难以解决。

构建研究 agent 评估的一种策略是组合评分器类型。依据性检查验证主张是否得到检索来源的支持，覆盖度检查定义好的答案必须包含的关键事实，来源质量检查确认咨询的来源是权威的，而不仅仅是首先检索到的。对于有客观正确答案的任务（"X 公司第三季度营收是多少？"），精确匹配可行。LLM 可以标记无依据的主张和覆盖度缺口，还可以验证开放式综合的连贯性和完整性。

鉴于研究质量的主观性，基于 LLM 的评分准则应经常与专家人工判断校准，以有效评分这些 agent。

### 计算机使用 Agent

**计算机使用 agent**通过与人类相同的界面与软件交互——截图、鼠标点击、键盘输入和滚动——而不是通过 API 或代码执行。它们可以使用任何具有图形用户界面（GUI）的应用程序，从设计工具到遗留企业软件。评估需要在真实或沙盒环境中运行 agent，使其可以使用软件应用程序，并检查是否达到了预期结果。例如，[WebArena](https://webarena.dev/)测试基于浏览器的任务，使用 URL 和页面状态检查来验证 agent 是否正确导航，以及对修改数据的任务进行后端状态验证（确认订单确实已下，而不仅仅是确认页面出现）。[OSWorld](https://os-world.github.io/)将此扩展到完整的操作系统控制，评估脚本在任务完成后检查各种产物：文件系统状态、应用程序配置、数据库内容和 UI 元素属性。

浏览器使用 agent 需要在 token 效率和延迟之间取得平衡。基于 DOM 的交互执行快速但消耗大量 token，而基于截图的交互较慢但更省 token。例如，当让 Claude 总结维基百科时，从 DOM 中提取文本更高效。当在亚马逊上找新的笔记本电脑包时，截图更高效（因为提取整个 DOM 是 token 密集型的）。在我们的 Claude for Chrome 产品中，我们开发了评估来检查 agent 是否为每个上下文选择了正确的工具。这使我们能够更快、更准确地完成基于浏览器的任务。

### 如何看待 Agent 评估中的非确定性

无论 agent 类型如何，agent 行为在运行之间会有所不同，这使得评估结果比乍看之下更难解读。每个任务都有自己的成功率——可能一个任务 90%，另一个 50%——在一次评估运行中通过的任务可能在下一次失败。有时，我们想衡量的是一个 agent 成功完成任务的*频率*（试验的比例）。

两个指标有助于捕捉这种细微差别：

[pass@k](https://arxiv.org/abs/2102.03316)衡量 agent 在 *k* 次尝试中至少得到一个正确解决方案的可能性。随着 *k* 增加，pass@k 分数上升：更多的"射门"意味着至少一次成功的更高概率。50% pass@1 的分数意味着模型在第一次尝试时成功完成评估中的一半任务。在编码中，我们通常最感兴趣的是 agent 在第一次尝试时就找到解决方案——pass@1。在其他情况下，提出多个解决方案是有效的，只要有一个有效即可。

[pass^k](https://arxiv.org/abs/2404.06374)衡量*所有 k* 次试验都成功的概率。随着 *k* 增加，pass^k 下降，因为要求更多试验的一致性是更难达到的门槛。如果你的 agent 每次试验的成功率是 75% 并运行 3 次试验，三次全部通过的概率是 (0.75)³ ≈ 42%。这个指标对于面对用户、期望每次可靠行为的 agent 特别重要。

![pass@k 和 pass^k 随试验次数增加而分叉](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F3ddac5be07a0773922ec9df06afec55922f8194a-4584x2580.png&w=3840&q=75)

*pass@k 和 pass^k 随试验次数增加而分叉。在 k=1 时，它们相同（都等于每次试验的成功率）。到 k=10 时，它们讲述了相反的故事：pass@k 接近 100%，而 pass^k 降至 0%。*

两个指标都有用，使用哪个取决于产品需求：pass@k 用于一次成功就重要的工具，pass^k 用于一致性至关重要的 agent。

## 从零到一：构建优秀 Agent 评估的路线图

本节列出了我们从零开始构建可信赖评估的实用、经过实战检验的建议。将此视为评估驱动 agent 开发的路线图：尽早定义成功、清晰地衡量、并持续迭代。

### 收集初始评估数据集的任务

**步骤 0. 尽早开始**

我们看到团队推迟构建评估，因为他们认为需要数百个任务。实际上，从真实故障中提取的 20-50 个简单任务就是一个很好的起点。毕竟，在 agent 开发早期，对系统的每次变更通常有明显的、可注意的影响，这种大效应量意味着小样本量就够了。更成熟的 agent 可能需要更大、更难的评估来检测更小的效应，但最好在开始时采用 80/20 方法。评估等待时间越长越难构建。早期，产品需求自然地转化为测试用例。等太久，你就得从运行中的系统逆向工程成功标准。

**步骤 1. 从你已经手动测试的内容开始**

从你在开发期间运行的手动检查开始——每次发布前验证的行为和终端用户尝试的常见任务。如果你已经在生产环境中，查看你的 bug 跟踪器和支持队列。将用户报告的故障转换为测试用例确保你的套件反映实际使用；按用户影响优先排序帮助你将精力投入到最重要的地方。

**步骤 2：编写明确的任务并附带参考解决方案**

把任务质量做对比看起来更难。一个好的任务是两个领域专家能独立得出相同通过/失败结论的任务。他们能自己通过任务吗？如果不能，任务就需要改进。任务规格中的模糊性会变成指标中的噪音。基于模型的评分器标准也是如此：模糊的评分准则会产生不一致的判断。

每个任务都应该能被正确遵循指令的 agent 通过。这可能很微妙。例如，审计 Terminal-Bench 发现，如果任务要求 agent 编写脚本但未指定文件路径，而测试假设特定的文件路径，agent 可能并非因自身过错而失败。评分器检查的所有内容都应该在任务描述中清晰可见；agent 不应因模糊的规格而失败。对于前沿模型，0% 的通过率（即 0% pass@100）通常是任务本身有问题的信号，而不是 agent 能力不足，这意味着你需要重新检查任务规格和评分器。对于每个任务，创建一个参考解决方案很有用：一个已知能通过所有评分器的有效输出。这证明任务是可解决的，并验证评分器配置正确。

**步骤 3：构建平衡的问题集**

同时测试行为*应该*发生和*不应该*发生的情况。单方面的评估造成单方面的优化。例如，如果你只测试 agent 是否在应该搜索时搜索，你最终可能得到一个几乎搜索一切的 agent。尽量避免[类别不平衡](https://en.wikipedia.org/wiki/Class_imbalance)的评估。我们在为 [Claude.ai](https://claude.ai/) 构建网页搜索评估时深有体会。挑战在于防止模型在不应该搜索时搜索，同时保留其在适当时进行深入研究的能力。团队构建了涵盖两个方向的评估：模型应该搜索的查询（如查找天气）和应该从现有知识回答的查询（如"谁创立了苹果？"）。在欠触发（应该搜索时不搜索）和过触发（不应该搜索时搜索）之间找到正确平衡很困难，需要对提示和评估进行多轮优化。随着更多示例问题出现，我们继续向评估中添加内容以改善覆盖率。

### 设计评估框架和评分器

**步骤 4：构建稳健的评估框架和稳定的环境**

评估中的 agent 应该与生产中使用的 agent 大致相同地运行，环境本身也不应引入额外的噪音，这至关重要。每次试验应通过从干净环境开始来"隔离"。运行之间不必要的共享状态（残留文件、缓存数据、资源耗尽）可能因基础设施不稳定而非 agent 性能导致相关故障。共享状态还可能人为地提高性能。例如，在一些内部评估中，我们观察到 Claude 通过检查之前试验的 git 历史在某些任务上获得了不公平的优势。如果多个不同试验因环境的相同限制（如有限的 CPU 内存）而失败，这些试验不是独立的，因为它们受相同因素影响，评估结果对于衡量 agent 性能变得不可靠。

**步骤 5：精心设计评分器**

如上所述，优秀的评估设计涉及为 agent 和任务选择最佳评分器。我们建议在可能的情况下选择确定性评分器，在需要额外灵活性时使用 LLM 评分器，并审慎使用人工评分器进行额外验证。

有一种常见的直觉是检查 agent 是否遵循了非常具体的步骤，比如按正确顺序调用工具。我们发现这种方法过于僵化，导致测试过于脆弱，因为 agent 经常找到评估设计师未曾预见的有效方法。为了不必要地惩罚创造力，通常最好评估 agent 产生了什么，而不是它走了什么路径。

对于包含多个组件的任务，**建立部分积分**。一个正确识别问题并验证客户但未能处理退款的客服 agent，明显比一个立即失败的 agent 更好。在结果中表现这种成功的连续性很重要。

模型评分通常需要仔细迭代来验证准确性。LLM 即评判的评分器应与人工专家密切校准，以确信人工评分和模型评分之间几乎没有分歧。为避免幻觉，给 LLM 一个退路，例如提供在信息不足时返回"Unknown"的指令。创建清晰、结构化的评分准则来评估任务的每个维度也有帮助，然后用独立的 LLM 即评判评估每个维度，而不是用一个来评估所有维度。一旦系统稳健，偶尔使用人工审查就足够了。

一些评估有微妙的失败模式，即使在 agent 表现良好的情况下也会导致低分，因为 agent 因评分 bug、agent 框架约束或模糊性而无法解决任务。即使经验丰富的团队也可能错过这些问题。例如，[Opus 4.5 在 CORE-Bench 上最初得分 42%](https://x.com/siyuan_says/status/1922725584447524928)，直到一位 Anthropic 研究员发现了多个问题：僵硬的评分在期望"96.124991…"时惩罚了"96.12"，模糊的任务规格，以及不可能精确重现的随机任务。修复 bug 并使用限制较少的脚手架后，Opus 4.5 的分数跃升至 95%。类似地，[METR 发现](https://metr.org/blog/2025-05-02-ai-r-d-bench-issues/)他们的时间范围基准中有几个配置错误的任务，要求 agent 优化到声明的分数阈值，但评分要求超过该阈值。这惩罚了 Claude 等遵循指令的模型，而忽略声明目标的模型获得了更好的分数。仔细检查任务和评分器可以帮助避免这些问题。

让你的评分器抗绕过或抗黑客攻击。agent 不应该能轻易"作弊"评估。任务和评分器应设计为通过真正需要解决问题而不是利用意外漏洞。

### 长期维护和使用评估

**步骤 6：检查转录**

除非你阅读许多试验的转录和评分，否则你不会知道评分器是否运作良好。在 Anthropic，我们投资了查看评估转录的工具，并定期花时间阅读它们。当任务失败时，转录告诉你 agent 是犯了真正的错误还是你的评分器拒绝了有效的解决方案。它还经常揭示关于 agent 和评估行为的关键细节。

失败应该看起来是公平的：agent 做错了什么以及为什么应该是清楚的。当分数不上升时，我们需要确信这是由于 agent 性能而不是评估。阅读转录是你验证评估是否衡量了真正重要的东西的方式，这是 agent 开发的一项关键技能。

**步骤 7：监控能力评估饱和**

100% 的评估跟踪回归但不提供改进信号。**评估饱和**发生在 agent 通过了所有可解决的任务，没有留下改进空间时。例如，SWE-Bench Verified 的分数今年从 30% 开始，前沿模型现在正接近饱和的 >80%。随着评估接近饱和，进展也会放缓，因为只剩下最困难的任务。这可能使结果具有欺骗性，因为大的能力改进表现为分数的小幅增长。例如，代码审查初创公司 [Qodo](https://www.qodo.ai/) 最初对 Opus 4.5 不以为然，因为他们的单次编码评估没有捕捉到更长、更复杂任务上的增益。作为回应，他们开发了新的 agent 评估框架，提供了更清晰的进展图景。

作为规则，在有人深入研究评估细节并阅读一些转录之前，我们不会将评估分数视为理所当然。如果评分不公平、任务模糊、有效解决方案被惩罚或框架限制了模型，评估就应该被修订。

**步骤 8：通过开放贡献和维护长期保持评估套件健康**

评估套件是一个活的工件，需要持续的关注和明确的所有权才能保持有用。

在 Anthropic，我们尝试了各种评估维护方法。最有效的是建立专门的评估团队来拥有核心基础设施，而领域专家和产品团队贡献大多数评估任务**并自己运行评估**。

对于 AI 产品团队，拥有和迭代评估应该像维护单元测试一样日常。团队可能浪费数周在早期测试中"有效"但未能满足精心设计的评估会及早暴露的未说明期望的 AI 功能上。定义评估任务是压力测试产品需求是否足够具体以开始构建的最佳方式之一。

我们建议实践评估驱动开发：在 agent 尚未满足能力之前构建评估来定义计划的能力，然后迭代直到 agent 表现良好。在内部，我们经常构建今天"够好"但押注模型几个月后能做到的功能。以低通过率开始的能力评估让这一点可见。当新模型发布时，运行套件可以快速揭示哪些押注得到了回报。

最接近产品需求和用户的人最有资格定义成功。在当前的模型能力下，产品经理、客户成功经理或销售人员可以使用 Claude Code 以 PR 的形式贡献评估任务——让他们来！或者更好的是，主动赋能他们。

## 评估如何与其他方法结合以全面理解 Agent

自动化评估可以在数千个任务上对 agent 运行，而无需部署到生产环境或影响真实用户。但这只是理解 agent 性能的众多方式之一。完整的图景包括生产监控、用户反馈、A/B 测试、手动转录审查和系统性人工评估。

| 方法 | 优势 | 劣势 |
|------|------|------|
| **自动化评估**<br>*无需真实用户即可编程运行测试* | • 更快的迭代<br>• 完全可重现<br>• 无用户影响<br>• 可在每次提交时运行<br>• 无需生产部署即可大规模测试场景 | • 需要更多前期投资来构建<br>• 随产品和模型演进需要持续维护以避免漂移<br>• 如果不匹配真实使用模式可能产生虚假信心 |
| **生产监控**<br>*在实时系统中跟踪指标和错误* | • 大规模揭示真实用户行为<br>• 捕捉合成评估遗漏的问题<br>• 提供 agent 实际表现的真实基准 | • 被动的；用户先遇到问题你才知道<br>• 信号可能有噪音<br>• 需要仪器化投资<br>• 缺乏评分的真实基准 |
| **A/B 测试**<br>*用真实用户流量比较变体* | • 衡量实际用户结果（留存、任务完成）<br>• 控制混淆因素<br>• 可扩展且系统化 | • 慢；需要数天或数周才能达到显著性且需要足够流量<br>• 只测试你部署的变更<br>• 在无法彻底审查转录的情况下，对指标变化背后的"原因"信号较少 |
| **用户反馈**<br>*明确信号如踩或 bug 报告* | • 揭示你未曾预期的问题<br>• 来自真实人类用户的实际例子<br>• 反馈通常与产品目标相关 | • 稀疏且自我选择<br>• 偏向严重问题<br>• 用户很少解释*为什么*失败<br>• 非自动化<br>• 主要依赖用户来发现问题可能对用户产生负面影响 |
| **手动转录审查**<br>*人工阅读 agent 对话* | • 建立对失败模式的直觉<br>• 捕捉自动化检查遗漏的微妙质量问题<br>• 帮助校准什么是"好的"并掌握细节 | • 耗时<br>• 不可扩展<br>• 覆盖率不一致<br>• 审查者疲劳或不同审查者可能影响信号质量<br>• 通常只给出定性信号而非清晰的定量评分 |
| **系统性人工研究**<br>*由训练有素的评分者对 agent 输出进行结构化评分* | • 来自多个人工评分者的黄金标准质量判断<br>• 处理主观或模糊的任务<br>• 为改进基于模型的评分器提供信号 | • 相对昂贵且周转缓慢<br>• 难以频繁运行<br>• 评分者间分歧需要协调<br>• 复杂领域（法律、金融、医疗）需要人工专家进行研究 |

这些方法映射到 agent 开发的不同阶段。自动化评估在发布前和 CI/CD 中特别有用，在每次 agent 变更和模型升级时运行，作为质量问题的第一道防线。生产监控在发布后启动以检测分布漂移和未预期的真实世界故障。A/B 测试在有足够流量后验证重大变更。用户反馈和转录审查是填补空白的持续实践：不断分类反馈，每周抽样阅读转录，并根据需要深入挖掘。保留系统性人工研究用于校准 LLM 评分器或评估主观输出，其中人工共识作为参考标准。

![Swiss Cheese Model](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2Fb77b8dbb7c2e57f063fbc8a087a853d5809b74b0-4584x2580.png&w=3840&q=75)

*就像安全工程中的[瑞士奶酪模型](https://en.wikipedia.org/wiki/Swiss_cheese_model)一样，没有单一评估层能捕捉每个问题。多种方法组合后，穿过一层的失败会被另一层捕捉。*

最有效的团队组合这些方法：自动化评估用于快速迭代，生产监控用于真实基准，定期人工审查用于校准。

## 结论

没有评估的团队陷入被动的循环——修复一个故障又创造另一个，无法区分真正的回归和噪音。及早投资的团队发现恰恰相反：随着故障变成测试用例、测试用例防止回归、指标取代猜测，开发加速。评估给整个团队一座清晰的山要攀登，将"agent 感觉变差了"转化为可操作的东西。价值会复合，但前提是你将评估视为核心组件，而非事后之想。

模式因 agent 类型而异，但此处描述的基础是恒定的。尽早开始，不要等待完美的套件。从你看到的故障中获取真实任务。定义明确、稳健的成功标准。精心设计评分器并组合多种类型。确保问题对模型来说足够难。迭代评估以提高其信噪比。阅读转录！

AI agent 评估仍是一个新兴、快速发展的领域。随着 agent 承担更长的任务、在多 agent 系统中协作，并处理越来越主观的工作，我们将需要调整我们的技术。我们将继续在学到更多时分享最佳实践。

## 附录：评估框架

几个开源和商业框架可以帮助团队实现 agent 评估，而无需从零构建基础设施。正确的选择取决于你的 agent 类型、现有技术栈，以及是否需要离线评估、生产可观测性，或两者兼需。

[Harbor](https://harbor.run/)专为在容器化环境中运行 agent 而设计，具有跨云提供商大规模运行试验的基础设施和定义任务与评分器的标准化格式。热门基准如 Terminal-Bench 2.0 通过 Harbor 注册表发布，使运行既有基准和自定义评估套件变得容易。

[Braintrust](https://www.braintrust.dev/)是一个结合离线评估与生产可观测性和实验追踪的平台——对于需要同时在开发期间迭代和生产中监控质量的团队很有用。其 `autoevals` 库包括预构建的事实性、相关性和其他常见维度的评分器。

[LangSmith](https://smith.langchain.com/)提供追踪、离线和在线评估，以及与 LangChain 生态系统紧密集成的数据集管理。[Langfuse](https://langfuse.com/)为有数据驻留要求的团队提供类似功能，作为自托管的开源替代方案。

[Arize](https://arize.com/)提供 Phoenix，一个用于 LLM 追踪、调试和离线或在线评估的开源平台，以及 AX，一个扩展 Phoenix 以实现规模、优化和监控的 SaaS 产品。

许多团队组合多个工具、自建评估框架，或仅使用简单的评估脚本作为起点。我们发现虽然框架可以是加速进度和标准化的有价值方式，但它们只与你通过它们运行的评估任务一样好。通常最好快速选择适合你工作流程的框架，然后将精力投入到评估本身——迭代高质量的测试用例和评分器。
