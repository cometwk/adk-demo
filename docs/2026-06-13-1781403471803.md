AI 领域发展迅猛，工具和概念层出不穷……还记得 **Context Engineering** 刚出现的时候吗，接着是 **Harness Engineering**……现在我们又有了 **Loop Engineering**。

> 把它想象成三层，每一层解决不同的问题……

![三层架构图](https://substackcdn.com/image/fetch/f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F930403f7-e0e8-46f0-848c-9924c1329b13_1200x625.png)

Harness 装备的是单次 Agent 运行；而 Loop 则是持续按调度驱动 Agent、生成子 Agent 并自我维持的系统。

![Harness 与 Loop 对比图](https://substackcdn.com/image/fetch/f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F780648c3-6b02-427b-bc06-b27c06730c95_2080x1098.png)

Loop Engineering 是这样一种转变：从你亲自逐轮向编码 Agent 发出提示，转变为你设计一个系统（即 Loop）——它自动发现工作、将任务分配给 Agent（通常是子 Agent）、验证结果、持久化状态，并决定下一步行动……按调度运行，或直到目标达成。

![Loop Engineering 概览](https://substackcdn.com/image/fetch/f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fb03d90c3-e1d2-4b4f-a3d6-b9fa1ca59501_1200x632.png)

> **Addy 的框架：** "Loop Engineering 就是取代你自己成为那个提示 Agent 的人。你设计系统来替你做这件事。"

他引用了两位关键实践者的话：

**Peter Steinberger**，OpenClaw 的创建者……"你不应该再亲自提示编码 Agent 了。你应该设计 Loop 来提示你的 Agent。"

> 工具的趋同令人瞩目：Claude Code 和 OpenAI Codex 都采用了非常相似的原语，因此"Loop 形态"正在变得与工具无关。

以及 **Boris Cherny（Anthropic Claude Code 负责人）**："我不再亲自提示 Claude 了。我有 Loop 在运行，它们提示 Claude 并自行决定做什么。我的工作就是写 Loop。"

旧的工作流（*写提示 > 读输出 > 写下一条提示*）正在被构建使用 Agent 的小型自主控制系统所取代。Addy 将此称为比"Agent Harness Engineering"更高一层。

## 五大构建模块 + 记忆

一个真正能无人值守运行的 Loop 并不是一条长提示。

它是一个包含六个部分的小系统。其中五个是能力组件，第六个是在运行之间维持状态的骨架。

## 自动化 / 调度是心跳

没有调度，你只有一次性的 Agent 会话。有了调度，你就有了按节奏运行的发现和分流机制。

这就是将"我应该每天早上检查 CI"变成无论你是否打开终端都会自动发生的事情的组件。

在 Claude Code 中，这可能是 `/loop`、`/schedule` 或 `/goal`——运行直到满足可验证的条件，由一个独立的模型检查"是否完成"，这样工作者不会自己给自己打分。

Hooks 和 GitHub Actions 将同样的理念带入了聊天之外的持久化场景。

在 Grok 中：`/loop [interval] <prompt>` 以及底层调度工具（`scheduler_create`、`scheduler_list`、`scheduler_delete`）。支持循环、持久化、立即触发。

> 心跳不需要很聪明，但需要可靠。

### Worktree 实现安全并行执行

两个 Agent 同时编辑相同文件是等待发生的合并灾难。

隔离的 git worktree 或等效的检出，为每个 Agent 提供独立的工作目录，同时共享历史。

两大主要编码 Agent 工具都提供了这个功能，子 Agent 可以启动到新的检出中，这样并行工作不会冲突。

在 Grok 中：生成子 Agent 时传入 `isolation: "worktree"`，或使用专用会话。清理很重要。一个留下孤立 worktree 的 Loop 是一个你会后悔的 Loop。

### Skills——持久化的项目知识

每次会话，Agent 都是从零开始。

约定、构建命令、审查标准，以及那个让你学到"我们不是这么做的"的事故——所有这些都必须重新推导，除非你将其外化。

> Skills 是你偿还意图债务的方式。

一个 SKILL.md（加上可选的脚本和参考资料）保存着应该跨运行保留的知识。Claude Code 使用 CLAUDE.md 和 skills，打包为插件以便共享。Grok 使用相同的模式。

没有 skills，每次 Loop 运行都是第一天。

### 插件与连接器——触达真实工具

一个只能读取文件系统的 Loop 只能提出建议。

基于 MCP 的连接器让 Loop 能够行动：创建 PR、更新 Linear 工单、发 Slack 消息、查询数据库、触发运维手册。Loop 从评论者变成了操作者。

MCP 已成为通用基础，因此为一个工具编写的连接器通常可以移植到另一个。Claude Code 和 Grok 都支持它。

### 子 Agent 是制造者/检查者的分离

编写代码的 Agent 不适合评判自己的工作。这不是模型的局限，而是结构性的问题。

一个 Agent（或团队）探索和实现。另一个不同的 Agent，有时是更强的模型，总是带着不同的指令……根据规范、skills 和测试进行验证。

在无人值守的 Loop 中，验证器是让你能放心离开的关键。多个工具中的 `/goal` 应用了同样的原则：一个全新的模型决定停止条件是否已满足。

### 记忆是持久化的骨架

以上任何一项都无法独自跨越会话边界……

Loop 必须读取和写入外部的东西：一个 STATE.md、一个 LOOP-STATE.json、一个 Linear 看板列、一个 GitHub Project 视图。好的状态回答三个问题：

- 我们现在在做什么？
- 上次我们尝试了什么，结果如何？
- 什么在等待人工处理？

对于多天或多运行的 Loop，这是不可妥协的。状态文件通常是 Loop 产生的最重要的产物。

## 没有人应该跳过的现实

Loop 还处于早期阶段。

经济性和验证挑战是真实存在的。

Token 成本差异巨大。

子 Agent 和高频调度会快速倍增。

一个每 5 分钟运行一次、每次都生成实现者 + 验证器的 Loop，会在早餐前就耗尽你的有限额度。

分流应该成本低廉；子 Agent 只应在状态表明可操作时才生成。

"完成"是一个主张，直到你确认它。无人值守的 Loop 会犯无人值守的错误。验证器有帮助，但它不能替代你审查进入仓库的内容。

理解债务在好的 Loop 下增长得更快。

速度可以掩盖已有内容和你的实际理解之间的差距。Loop 发布了它，但这并不意味着你知道它是怎么工作的。

> 认知投降是一个舒适的陷阱。

同样的 Loop 设计可以加速一个保持工程师角色的人，也可以让一个人完全放弃判断。像打算继续当工程师的人那样构建 Loop——而不是仅仅做那个按下启动按钮的人。

## 最后

直接提示仍然很强大，而且通常是正确的工具。

但杠杆的支点已经移动了。

Peter Steinberger 说得很直白……你不应该再亲自提示编码 Agent 了……你应该设计 Loop 来提示你的 Agent。

Anthropic 的 Boris Cherny 从内部说了同样的话：他的工作是写 Loop，而不是写单独的提示。

这就是 Loop Engineering。不是一个更大的提示，而是一个能发现、分配、验证、持久化并知道何时交接给你的系统。
