# 起源故事

在我打开笔记本电脑之前，代码就已经自我审查完毕了。

我打开 GitHub，准备按照惯例开始——标记命名不当的变量、删减过度的测试覆盖、建议更简洁的错误处理方式。然而，我看到的是 Claude Code 给出的几条有力的评论：

> "变量命名已按照 PR #234 的模式进行了更改，根据 PR #219 的反馈删除了过度的测试覆盖，添加了类似于 PR #241 中已批准方案的错误处理。"

Claude 从过去三个月的代码审查中学习，并在没有被要求的情况下应用了这些经验。它完全掌握了我的偏好，就像一位敏锐的新队友——而且有据可查。

这感觉像是在作弊，但实际上并非如此——这是复合效应。

## 构建 Cora

我是在构建 Cora 的过程中发现复合工程的。

Cora 是 Every 的 AI 邮件助手。每周我都在发布功能、修复 bug、处理邮件解析中的边缘情况、添加集成。这些都很正常。只是我和 Claude 一起在做这些事，而感觉有所不同。

起初，我像其他人一样：让 AI 写一些代码，复制粘贴，逐行审查，修复错误。标准工作流程。但代码越来越好。不仅仅是因为我在提示方面变得更好——AI 在学习我的代码库、我的模式、我的偏好。

有一天，我注意到那周我实际上没有写任何代码。我把所有时间都花在了规划功能、审查 Claude 产出的内容、记录模式上。功能照样发布。测试通过。用户满意。

就在那时，我恍然大悟。

## 顿悟

真正的实力不在于你写的代码。而在于你围绕写代码构建的系统。

每次我修复问题时，我都会把上下文添加到 CLAUDE.md。每次我注意到一个模式，我就为它创建一个 agent。每次我遇到一个反复出现的问题，我就记录解决方案。第二天，Claude 就不会再犯同样的错误。

这不仅仅是 AI 辅助。这是复合效应。

## 复合意味着什么

在金融领域，复利意味着你的收益会产生自己的收益。投资的一美元变成两美元，然后四美元，然后八美元。增长会加速。

在工程领域，复合工作意味着每个解决方案都能为未来的问题生成解决方案。记录的模式可以预防十个未来的 bug。审查清单可以在问题变成事故之前捕捉它们。在你的代码库上训练的 agent 会像你一样思考。

传统工程是线性的。你解决问题 A，然后问题 B，然后问题 C。每个问题大约花费相同的精力。

复合工程是指数级的。你解决问题 A，然后你教系统你是如何解决的。问题 B 花费一半的时间。问题 C 花费四分之一的时间。

**典型的 AI 工程** 关注短期收益。你提示，它编码，你发布。然后重新开始。

**复合工程** 关注构建有记忆的系统，每个 pull request 都在教系统，每个 bug 都变成永久的教训，每个代码审查都在更新默认值。

AI 工程让你今天更快。复合工程让你明天更快，以及之后每一天都更快。

## 一个真实案例：挫折检测器

以下是复合工程在实践中如何运作。

我正在为 Cora 构建一个"挫折检测器"——目标是让我们的 AI 助手注意到用户何时对应用的行为感到恼火，并自动提交改进报告。

传统方法：编写检测器，手动测试，调整，重复。需要在像用户一样思考和像开发者一样思考之间频繁切换上下文。

复合方法：我从一个示例对话开始，我在其中表达了挫折感——比如反复问同一个问题，语言越来越简短。然后我把它交给 Claude："这段对话显示了挫折感。写一个测试来检查我们的工具是否能捕捉到它。"

Claude 编写测试。测试失败——这就是测试驱动开发。Claude 编写检测逻辑。它仍然不能完美工作。美妙的部分来了：我让 Claude 迭代挫折检测提示，直到测试通过。

Claude 调整提示并再次运行测试。它读取日志，看到为什么错过了挫折信号，再次调整。经过几轮迭代后，测试通过了。

但 AI 输出不是确定性的——一次有效的提示下次可能会失败。所以我让 Claude 运行测试 10 次。当它只在 10 次中的 4 次识别出挫折感时，Claude 分析了它在另外 6 次中失败的原因。它发现了一个模式：它错过了像"嗯，不太对"这样的委婉语言，当与重复请求配对时，这表明了挫折感。Claude 更新了提示，专门寻找礼貌但感到挫折的语言。

下一次迭代：9/10。足够好，可以发布了。

我们将这整个工作流程——从识别模式到迭代提示到验证——固化在 CLAUDE.md 中。下次我们需要检测用户的情绪或行为时，不需要从头开始。我们说："使用挫折检测器的提示工作流程。"系统已经知道该做什么。

## 从个人工具到共享系统

我最初是为我自己创建的。我的代码库、我的模式、我的偏好。

然后我把它展示给 Dan。他开始在 Every 使用它。然后我们展示给几个朋友。然后我们开源了它。

现在成千上万的开发者在使用它。他们添加了自己的 agent、自己的模式、自己的工作流程。系统不仅在单个代码库内复合，而且在整个社区中复合。

奇怪的是？对他们也有效。核心理念——每个工作单元都应该让后续工作更简单——结果证明是普遍适用的。不同的代码库、不同的语言、不同的问题。同样的原则。

## 结果

在 Cora 上进行三个月的复合工程后：

- **交付时间** 从每个功能超过一周下降到 1-3 天
- **生产前捕捉的 bug** 大幅增加
- **PR 审查周期** 曾经拖上几天，现在几小时内完成
- **我的三栏工作流程**：一个终端规划，一个构建，一个审查

在 Cora，我们使用复合工程来：

- **将生产错误转化为永久修复** 通过让 AI agent 自动调查崩溃、从系统日志重现问题、生成解决方案和防止复发的测试
- **从协作工作会议中提取架构决策** 通过记录设计讨论，然后让 Claude 记录为什么选择了某些方案
- **构建具有不同专业知识的审查 agent**——一个"Kieran 审查者"强制执行我的风格选择，一个"Rails 专家审查者"负责框架最佳实践，一个"性能审查者"负责速度优化
- **自动化视觉文档** 通过部署一个检测界面变化的 agent，并捕获不同屏幕尺寸和主题的前后截图
- **并行化反馈解决** 通过为每条审查者反馈创建专门的 agent 同时工作——十个问题的解决时间原来只能解决一个

# 理念：复合工程宣言

> 每个工程工作单元都应该让后续单元更简单——而不是更难。

这是核心原则。一切由此衍生。

传统开发累积债务。每个功能增加复杂性。每个捷径创造未来工作。代码库变得更难理解、更难修改、更难信任。十年后，你花更多时间对抗系统而不是基于它构建。

复合工程颠覆了这一点。每个功能教系统。每个 bug 修复预防未来的 bug。每个你固化的模式成为未来工作的工具。代码库变得更容易理解、更容易修改、更容易信任。

问题不是"我如何发布这个功能？"而是"我如何以让下一个功能更简单的方式发布这个功能？"

## 你需要放弃的信念

你被训练以某种方式思考软件开发。其中一些训练现在是错误的。以下是需要放弃的：

### "我必须自己写代码"

不。你必须确保好的代码被写出来。是你打字还是 AI 打字无关紧要。重要的是结果：干净、经过测试、可维护的代码，解决了正确的问题。

### "我必须审查每一行"

不。你必须确保代码符合你的标准。这可能意味着审查每一行。也可能意味着拥有能捕捉你会捕捉到的东西的系统，然后信任这些系统。

如果你不信任系统，改进它们。不要通过永久手动审查来补偿。

### "我必须想出解决方案"

不。你必须确保选择了好的解决方案。AI 可以研究方法、分析权衡、推荐选项。你的工作是品味：知道哪个解决方案适合这个代码库、这个团队、这个上下文。

### "代码是最重要的产物"

不。产生好代码的系统才是最重要的产物。一个出色的实现不如一个持续产出好实现的流程有价值。

### "写代码就是工作"

不。交付价值才是工作。代码是一种手段。规划是另一种。审查是另一种。教系统是另一种。最好的复合工程师写的代码比以前少——但交付的价值更多。

## 你需要采纳的信念

### 将你的品味提取到系统中

你有偏好。你知道这个代码库中好的代码是什么样的。你对架构、命名、错误处理、测试有观点。

现在，这种品味存在于你的头脑中。每次 AI 写代码时，你通过审查施加你的品味。这不可扩展。

提取它。写下来。把它变成 agents、prompts、CLAUDE.md 指令。当系统分享你的品味时，它产生你喜欢的代码，而不需要你修复它。

### 多花时间固化，少花时间编码

这是 50/50 规则：花 50% 的时间改进系统，50% 构建功能。

传统工程颠倒了这一点——90% 构建，10% 其他所有。这就是传统代码库累积债务的原因。

当你投资系统时，你不是在放慢速度。你在构建一个永远产生回报的资产。花一小时创建一个审查 agent，可以在明年节省十小时的审查时间。

### 100 美元规则

当某个应该被预防的问题失败时，我会罚自己 100 美元，并把它花在永久修复上——一个测试、一条规则、一个评估。

例如：用户报告他们从未收到每日邮件 Brief——一个关键故障。我们编写了捕捉类似交付失误的测试，更新了监控规则以标记 Brief 未发送的情况，并构建了持续验证交付管道的评估。

现在系统永远关注这类问题。一个故障让我们的工具永久变得更聪明。

100 美元规则创造了强制函数。感受一次刺痛，永远修复它。每个失败都变成预防投资。

### 多花时间规划，少花时间实现

一个好的计划配合平庸的执行，胜过一个平庸的计划配合出色的执行。

原因是：AI 可以出色地执行。它快速编写代码，不会疲劳，不会打字错误。它不能做的是知道要构建什么。那是你的工作。

投资规划。彻底研究。考虑替代方案。把计划做对。然后告诉 AI 实现它，不要干预。

### 信任流程，构建安全网

如果你在审查每一行，你无法扩展 AI 辅助。你需要信任流程。

但信任不是盲目信仰。它意味着构建能捕捉问题的系统。意味着有失败时能报错的测试。意味着有能标记问题的审查 agents。

如果你不信任某个步骤，不要通过手动审查来补偿。相反，添加一个让该步骤值得信任的系统。然后信任系统。

### 为未来模型构建，而非当前局限

今天的模型有局限。它们犯错误。它们需要指导。

不要为这些局限构建复杂的补偿。构建假设模型会改进的系统。

这实际上意味着什么？倾向于 agent 架构而非刚性工作流程。工作流程在模型变化时会崩溃。agents 会适应。为你六个月后会有模型构建，而非你今天有的模型。

### 让你的环境 agent-native

作为开发者你能做的事情，agent 也应该能做。你能看到的东西，agent 也应该能看到。

你能运行测试吗？agent 也应该能运行测试。
你能查看生产日志吗？agent 也应该能查看生产日志。
你能用截图调试吗？agent 也应该能用截图调试。
你能创建 pull request 吗？agent 也应该能创建 pull request。

你从 agent 中扣留的每个能力都需要你的手动干预。Agent-native 架构意味着 agent 能做你能做的一切。

### 做成你自己的

没有通用的复合工程设置。只有你的设置。

你的代码库不同。你的团队不同。你的偏好不同。对你有效的系统不会对其他人完全相同。

实验。找出什么有效。适应你从他人那里学到的东西。但不要照搬别人的系统。构建你自己的。

### 并行化是你的朋友

旧约束是人类注意力。你一次只能处理一件事。

新约束是计算机资源。你可以并行运行十个 agents。你可以同时开发三个功能。你可以同时审查、测试和记录。

并行思考。当你在一件事上被阻塞时，不要等待——开始另一件事。让你规划下一件事的时候 agents 工作。

**任务控制设置**

我的显示器现在看起来像任务控制中心：

- **左栏：规划。** 一个 Claude 实例读取问题、研究方法、编写详细实现计划。
- **中栏：构建。** 另一个 Claude 接收这些计划、编写代码、创建测试、实现功能。
- **右栏：审查。** 第三个 Claude 对照 CLAUDE.md 审查输出、建议改进、捕捉问题。

起初感觉很别扭——就像在学杂耍时还要杂耍——但一周内就变得自然了。

### 你的工作不再是打代码

你的工作是设计设计系统的系统。

公司每月支付 400 美元，原来每年要花 400,000 美元。个人初创公司在与有资金的团队竞争。AI 不仅在民主化编码，还在民主化整个工程系统。杠杆正在转移到那些比打字更快教会这些系统的人。

如果你是一个打代码的工程师，打代码的价值很快就会归零。开始关注其他部分——品味、愿景、系统——成为一个复合工程师。

### 把复合思维推向每个地方

复合工程不只是用于写代码。它适用于：

- **研究**：记录你学到的东西，这样你以后不需要重新研究
- **设计**：固化设计模式，这样 AI 可以应用它们
- **QA**：构建能捕捉你手动会捕捉的东西的测试系统
- **产品营销**：从代码变化生成公告
- **安全**：构建能捕捉漏洞的审查 agents
- **调试**：记录解决方案，这样同一个 bug 永远不会让你花两次时间
- **复盘**：将事故转化为预防系统

在任何你做重复工作的地方，问：我如何做一次，让系统永远做下去？

## 飞轮

当你遵循这些原则时，会发生什么：

你的第一次代码审查很慢。你在教 agent 看什么。

你的第二次代码审查更快。agent 记住了你教它的东西。

你的第十次代码审查捕捉到你曾经错过的东西。agent 学到了你没有意识到的模式。

你的第一百次代码审查与五个其他审查并行运行。你在审查发现的问题，而不是代码。

这就是飞轮。每轮让下一轮更好。工作复合。

## 不适感

这会感觉不舒服。

当你不打代码时你会感觉懒惰。你没有懒惰——你在利用杠杆。

当 agent 自主工作时你会感觉失控。你没有失控——你在信任你构建的系统。

当一个功能发布而你没有触碰实现时你会感觉内疚。不要。你规划了它、审查了它、确保它符合你的标准。这比打代码更有价值。

不适感是一个信号。它意味着你在改变工作方式。挺过去。

## 复合工程师的誓言

我将让每个工作单元让后续工作更简单。

我将把我的品味提取到系统中，而非通过手动审查来强制执行。

我将花更多时间教系统，而非自己做工作。

我将信任流程，构建安全网，而非通过手动审查来补偿。

我将让我的环境 agent-native。

我将把复合思维推向我工作的每个部分。

我将拥抱放手的不适感。

我将交付更多价值，同时打更少代码。

# AI 开发阶梯

并非所有 AI 辅助开发都一样。有一个阶梯——从基本聊天辅助到完全自主工程的进阶。你在阶梯的位置决定了什么复合工程实践对你有意义。

你不能跳过阶段。从阶段 0 跳到阶段 5 太极端了。会感觉不舒服，你不会信任流程。你需要一级一级爬上阶梯。

## 阶段 0：不使用 AI

这是每个人开始的地方。你写每一行。你通过读文档和 Stack Overflow 研究解决方案。你通过读代码和添加 print 语句调试。

阶段 0 没有错。几十年来伟大的软件就是这样构建的。但如果你在 2025 年还在这里，你在比需要的工作更努力。

**特征：**
- 纯手动编码
- 无 AI 辅助
- 所有研究由人类驱动
- 调试是手动的

**何时这有意义：**
- 学习新语言或框架（有时）
- 安全关键代码，你需要理解每一行
- 当 AI 真的不能帮助时（这些天很少）

## 阶段 1：聊天式 AI（并肩协作）

你在使用 ChatGPT、Claude 或 Cursor 作为助手。你提问、获取片段、复制粘贴代码。AI 是一个智能参考书，可以为你写代码。

你仍在控制。你仍在审查一切。AI 加速研究和样板，但你是掌舵的人。

**特征：**
- 向 AI 请求代码片段
- 复制粘贴到你的编辑器
- 使用 AI 解释代码或调试错误
- 审查 AI 写的每一行
- 仍将 AI 视为工具，而非协作者

**感觉像什么：**
"你能写一个做 X 的函数吗？"
"这段代码有什么问题？"
"我如何使用库 Y？"

**比阶段 0 的好处：**
- 更快的样板
- 即时回答问题
- 更好的错误消息解释

**局限：**
- 大量上下文切换（聊天 → 编辑器 → 聊天）
- 无代码库意识
- 你仍在做所有集成工作

## 阶段 2：Agent 式，逐行审查

你在使用 agent 工具——Claude Code、Cursor Composer、Copilot Chat。AI 可以读你的文件、理解你的代码库、直接做更改。

但你仍在审查一切。agent 提议更改，你读每一行，你批准或拒绝。你是守门人。

**特征：**
- Agent 有代码库访问权限
- Agent 可以读写文件
- 你批准每个单一动作
- 你审查每行更改的代码
- 逐步协作

**感觉像什么：**
"添加用户认证功能" → agent 写代码 → 你审查每个文件 → 批准 → agent 继续

**比阶段 1 的好处：**
- 无上下文切换
- Agent 理解你的代码库
- 单流程中的多文件更改
- Agent 保持一致性

**局限：**
- 慢——你在审查一切
- 瓶颈是你的注意力
- 不能并行化
- 你仍在做所有质量保证

**这是大多数开发者停止的地方。** 感觉有效率。你比阶段 1 发布更快。但你在放弃桌子上巨大的收益。

## 阶段 3：计划和审查 PR 只

这是突破阶段。

你与 AI 创建详细计划。你敲定需求、方法、边缘情况。然后你说"实现这个"并走开。

当你回来时，有一个 pull request。你审查 PR，而非每行更改。你信任计划和流程。如果有什么问题，你在 PR 审查中修复它——你不监督实现。

**特征：**
- 前期彻底规划
- "实现这个计划"作为单个命令
- 审查在 PR 级别，非行级别
- 感觉像与初级开发者工作
- 你可以移交计划给他人，或接手他们的计划

**感觉像什么：**
你："这是用户认证的计划：JWT 令牌、刷新机制、bcrypt 密码哈希、所有边缘情况的测试。"
Agent：*实现 10 分钟*
Agent："PR 准备审查。"
你：*审查 PR，请求一个更改*
Agent：*修复它*
你：*合并*

**比阶段 2 的好处：**
- 快 10 倍——你不是瓶颈
- 可以在 agent 实现时做其他事情
- 更好的质量——彻底规划早期捕捉问题
- 协作性——其他人可以实现你的计划

**局限：**
- 仍然一次一件事
- 仍在你的电脑上
- 仍有一些手动审查

**这是复合工程真正开始的地方。** 你的计划变得更好，因为 agent 学习你的模式。你的审查更快，因为你信任流程。工作复合。

## 阶段 4：想法到 PR（单台电脑）

你有一个想法。你描述它。一个 pull request 出现。

agent 做一切：研究代码库、创建计划、实现、运行测试、自我审查、修复问题、创建 PR。你只是有想法。

仍在你的电脑上，仍然一次一件事。但你的参与降到：想法 → 审查 PR → 合并。

**特征：**
- 端到端自主实现
- Agent 处理规划、实现、测试、审查
- 你只审查最终 PR
- 感觉像与能干的同事工作

**感觉像什么：**
你："当用户收到新评论时添加邮件通知。"
*去喝咖啡*
Agent："PR #247 准备。添加了 NotificationMailer、异步交付的后台作业、偏好设置。测试通过。"
你：*审查 PR，看起来好，合并*

**比阶段 3 的好处：**
- 近零实现参与
- Agent 在你看到之前捕捉问题
- 你专注于构建什么，而非如何

**局限：**
- 仍然顺序
- agent 工作时电脑忙碌
- 不能并行化

## 阶段 5：云端并行

最终阶段。工作在云端发生，并行。

你描述三个功能。三个 agents 启动，每个实现一个。你定期检查审查 PRs。当一个完成时，你给它另一个任务。

在极端情况下：agent 主动监控功能请求、用户反馈、bug 报告。它提议功能，你批准，它实现。你在指挥舰队，而非划船。

**特征：**
- 云端执行
- 多功能并行
- 主动——agent 提议构建什么
- 你是审查者和批准者
- 用户反馈 → 功能，无需你写代码

**感觉像什么：**
周一早上：
- PR #248：邮件通知 *（来自你的想法）*
- PR #249：暗黑模式 *（来自用户反馈）*
- PR #250：性能修复 *（来自监控）*

你：*审查所有三个，合并两个，在一个上请求更改*
Agent：*修复请求的更改，提议下一个功能*

**比阶段 4 的好处：**
- 真正并行化
- 主动解决问题
- 电脑不被占用
- 超越单人类能做的规模

**局限：**
- 需要基础设施
- 需要信任流程
- 需要好的系统以避免混乱

## 你在哪里？

诚实点。大多数读这个的开发者在阶段 1 或 2。

这没关系。路径清晰：

**从阶段 1**：开始使用 agent 工具。让 AI 访问你的代码库。习惯它做更改。

**从阶段 2**：停止审查每行。创建更好的计划。信任流程。在 PR 级别审查。

**从阶段 3**：进一步降低你的参与。让 agent 处理更多端到端。习惯不看每步。

**从阶段 4**：移到云端。并行运行事物。构建主动工作的系统。

每个阶段花时间。每个阶段起初感觉不舒服。这就是你知道你在进步的方式。

# 如何升级

AI 开发阶梯上的每个阶段需要不同方法。以下是如何从你所在的地方攀登。

## 级别 0 → 级别 1：开始协作

你从未真正使用 AI 编码。你听说过，也许试过 ChatGPT 问问几个问题，但你的工作流程仍然是：你 + 编辑器 + 文档。

### 做什么：

1. **选择一个工具。** Cursor 配合 Opus 4.5 是好开始。或 Claude Code 如果你偏好终端。或 GitHub Copilot 如果你想最少设置。

2. **从问题开始，而非代码。** 让 AI 解释你正在看的代码。问它为什么某事不工作。习惯与它交谈。

3. **让它写样板。** 你让 AI 写的第一批代码应该是无聊的：测试、配置文件、重复函数。低风险，高时间节省。

4. **总是审查它写的东西。** 复制粘贴到编辑器。读每行。修复错误。你在学习它做对和做错什么。

### 你会经历什么：

- 对它写代码多快的惊讶
- 当它做错时的挫折
- 是否信任它的不确定
- 随着你学习它的模式逐渐增加信任

### 如何复合：

每个会话后，注意 AI 哪里错了。这些变成你未来审查时要检查的东西。你在构建 AI 行为的心理模型。

### 当你准备好继续：

你每天使用 AI。你信任它一些东西。你有感觉它擅长什么。你准备好让它更直接在你的代码库工作。

---

## 级别 1 → 级别 2：让 Agent 进来

你习惯与 AI 聊天，但你仍在复制粘贴代码。时间让 agent 直接访问你的代码库。

### 做什么：

1. **切换到 agent 工具。** Claude Code、Cursor Composer 或类似。能读写文件的东西。

2. **从小开始。** "为这个函数添加测试。" "修复这个文件的 linting 错误。" 让 agent 做你可以容易验证的有针对性更改。

3. **批准每个动作。** Agent 在做更改前会请求许可。说对时它是对的，说不对时它错的。你仍是守门人。

4. **审查差异，不只是代码。** 看什么变了，不只是那里有什么。Agent 应该解释它在做什么。

### 你会经历什么：

- 不不断上下文切换的解脱
- 让东西修改文件的焦虑
- 偶尔的"它刚才做了什么？"时刻
- 对 agent 更改的逐渐舒适

### 如何复合：

添加上下文到你的项目。创建一个有你偏好的 CLAUDE.md。当 agent 犯错，添加一条关于避免什么的注释。每条上下文 Agent 变得更聪明。

### 当你准备好继续：

你不担心 agent 做更改。你信任差异。你花更多时间审查而非打字。瓶颈是你的注意力，而非 AI 的能力。

---

## 级别 2 → 级别 3：信任计划

你在逐步协作，但它慢。你批准每个动作。时间毕业到计划级别的信任。

### 做什么：

1. **投资规划。** 开始实现前，创建彻底计划。需求是什么？方法是什么？什么边缘情况存在？

2. **让 agent 研究。** 让它读你的代码库、理解模式、建议方法。用它的研究告知你的计划。

3. **使计划明确。** 写下来。具体。计划中细节越多，实现需要监督越少。

4. **说"实现这个"并走开。** 给 agent 计划让它工作。完成时回来。

5. **在 PR 级别审查。** 看最终结果，而非每步。匹配计划吗？工作吗？干净吗？

### 你会经历什么：

- 不看每步的焦虑
- 不在时完成多少的惊讶
- 偶尔的"我不是那个意思"时刻
- 更好计划 = 更好结果的领悟

### 如何复合：

每次实现后，注意计划缺了什么。agent 理解错什么？你应该指定什么？更好计划复合成更好结果。

### 阶段 3的不同：

你不监督。你在指导。Agent 是能独立执行的协作者。它在实现时你可以做其他事情。

### 当你准备好继续：

你习惯不监督实现。你的计划足够详细，结果匹配预期。你在思考还有什么可以自动化。

---

## 级别 3 → 级别 4：描述，不要计划

你擅长计划，但规划仍花你的时间。时间也信任 agent 规划。

### 做什么：

1. **给高层描述。** 非详细计划，描述你想要的。"为新评论添加邮件通知。" 让 agent 想出如何。

2. **信任 agent 研究。** 它知道你的代码库。它能找到类似实现。它能读文档。让它做规划工作。

3. **实现前审查计划。** Agent 应该展示它的计划。你批准方法，非细节。

4. **审查 PR，非过程。** 实现完成时，看结果。Agent 在展示你之前自我审查和修复问题。

### 你会经历什么：

- 对 agent 计划多好的惊讶
- 偶尔的方法纠正
- 你是审查者非实现者的领悟
- 专注于更高层思考的自由

### 如何复合：

记录 agent 应遵循的模式。当它做好的决定，注意为什么。当它做不好的，添加指导。每条固化的模式 Agent 判断改进。

### 阶段 4的不同：

你不规划。你在构思。Agent 处理从想法到 PR的一切。你的工作是有好想法和验证好结果。

### 当你准备好继续：

你信任 agent 的端到端流程。你有好的质量系统（测试、审查、监控）。你想超越一次一件事的规模。

---

## 级别 4 → 级别 5：并行化一切

你有一个 agent 做好工作。时间有多个。

### 做什么：

1. **移到云端执行。** Agent 不该占用你的电脑。使用云端 Claude 实例、GitHub Actions 或专用基础设施。

2. **启动多个工作流。** 三个功能并行。每个功能一个 agent。PRs 进来时审查。

3. **构建队列系统。** 想法、bug 和改进进入队列。Agents 拉取工作。你审查结果。

4. **使它主动。** Agents 监控机会。用户反馈变成功能提议。错误变成 bug 修复。你批准，它们实现。

### 你会经历什么：

- 起初从并行工作流的压倒
- 对更好系统的需要（PR 队列、通知、优先级）
- 发布速度的兴奋
- 从实现者到编排者的身份转变

### 如何复合：

你构建的每个管理并行工作的系统让并行工作更简单。更好 PR 模板。更好通知系统。更好优先级规则。基础设施复合。

### 阶段 5的不同：

你在运行舰队。你的工作是战略——我们该构建什么、什么顺序、什么标准？Agents 处理执行。

---

## 常见陷阱

### 尝试跳过阶段

不要从阶段 1 跳到阶段 4。你没构建信任、系统或直觉。出问题时你会退回。

### 在阶段 2 卡住

阶段 2 感觉有效率。你比以前发布更快。但你在放弃桌子上 90% 的收益。挺过不适感。

### 补偿而非复合

出问题时，不只修复它——预防它。不要审查更仔细；构建自动捕捉问题的系统。

### 应该改进计划时怪工具

坏结果通常意味着坏输入。如果 agent 一直做错，你的计划不够详细。你的上下文不够清楚。改进输入。

# 主循环

复合工程的核心是四步循环：

```
计划 → 工作 → 审查 → 复合 → 重复
```

这四步是可组合的。每步可扩展或收缩，取决于你构建什么。但如果你在阶段 3 或以上做复合工程，你在为每个有意义的工作运行某种版本的这个循环。

80/20 规则适用：80% 时间在计划和审查。20% 在工作和复合。代码写完前后发生大部分思考。

---

## 1. 计划 (PLAN)

计划阶段把想法变成蓝图。计划越好，结果越好。

### 计划中发生什么：

1. **理解需求。** 我们构建什么？为什么？什么约束存在？

2. **研究代码库。** 类似功能如何工作？什么模式存在？什么需要改变？

3. **外部研究。** 框架文档说什么？最佳实践是什么？他人如何解决这个？

4. **设计解决方案。** 方法是什么？什么文件需要改变？什么边缘情况？

5. **验证计划。** 这合理吗？完整吗？有缺口吗？

### 计划保真度：

不是所有计划需要同样深度：

**小（快速任务）：**
- 已知原因的 bug 修复
- 直截的添加
- 单文件更改
- 最少研究，快执行

**中（标准功能）：**
- 新功能
- 多文件更改
- 中等研究
- 清晰但详细的计划

**大（复杂功能）：**
- 架构更改
- 多研究 agents 并行工作
- 框架文档、最佳实践、repo 分析、git 历史
- 有边缘情况和回滚策略的详细计划

`/plan` 命令检测你需要哪个保真度并相应资源。

### 研究策略

每个策略服务不同目的。更大功能使用多个策略：

**最佳实践基础：**
在互联网搜索此业务用例或设计模式的最佳实践。找博客文章、文档、观点性的看法。比通用模型知识更有用，因为它呈现真实世界观点。

**代码库基础：**
查看当前源代码看你是否已经做类似事情。防止重复发明。如果你想添加事件追踪已有事件追踪系统，agent 找到它而非构建新系统。

**库基础：**
查看你安装的 gems/packages 源代码。比读文档好，因为你知道确切什么可用。当你更新包时，agent 自动知道新能力。

**Git 历史：**
看相关文件的过去提交理解方向和意图。对重构尤其有用——你可以看到移除了什么、添加了什么、为什么做更改。

**氛围编码原型：**
有时你不知道你想构建什么。先氛围编码原型——让 AI 生成选项、快速迭代、看什么有效。然后拿那个理解创建正经计划。删除原型，带着知识重新开始。

### 好计划包括：

- **上下文**：为什么做这个？解决什么问题？
- **方法**：如何解决？什么策略？
- **文件**：什么需要改变？什么需要创建？
- **边缘情况**：什么可能出错？如何处理？
- **测试**：什么测试验证这个有效？
- **回滚**：如果坏了，如何回退？

---

## 2. 工作 (WORK)

工作阶段执行计划。在复合工程，你不打字——你在监控。

### 工作中发生什么：

1. **设置隔离。** Worktrees 或分支保持工作分离。

2. **执行计划。** Agent 逐步实现。

3. **运行验证。** 测试、linting、类型检查。

4. **追踪进度。** 完成什么，剩余什么。

5. **处理问题。** 出问题时，适应。

### 实践中的工作：

你说 `/work` 配合计划。Agent：
- 创建 worktree
- 实现计划每步
- 大更改后运行测试
- 完成时创建 PR

你不看每行。你信任计划。如果计划好，执行会好。

### 何时干预：

- 测试反复失败
- Agent 明显卡住
- 计划中根本性的东西错了

干预时，更新计划，而非代码。让 Agent 基于更新计划修复代码。

### 并行工作：

阶段 4+，你可以有多个工作流：
- 功能 A 在 worktree A
- 功能 B 在 worktree B
- Bug 修复在 worktree C

定期检查。PRs 完成时审查。

---

## 3. 审查 (REVIEW)

审查阶段在发布前捕捉问题。但更重要的是，它为下个周期捕捉学习。

### 审查中发生什么：

1. **多 agent 审查。** 多个专门审查者并行检查代码。

2. **优先级发现。** P1（必须修）、P2（应该修）、P3（可选修）。

3. **解决发现。** Agent 基于审查反馈修复问题。

4. **验证修复。** 确保修复正确完整。

5. **捕捉模式。** 注意什么错了，所以不再发生。

### 审查 agents：

运行 `/review` 看十二个 agents 并行分析代码：

**安全：**
- **security-sentinel**：OWASP top 10、注入攻击、认证缺陷

**性能：**
- **performance-oracle**：N+1 查询、缓存机会、瓶颈

**架构：**
- **architecture-strategist**：系统设计、组件边界、模式
- **pattern-recognition-specialist**：设计模式、反模式、代码异味

**数据：**
- **data-integrity-guardian**：迁移、事务、参照完整性
- **data-migration-expert**：ID 映射、回滚安全、生产验证

**质量：**
- **code-simplicity-reviewer**：YAGNI、不必要复杂性、可读性
- **kieran-rails-reviewer**：Rails 约定、Turbo Streams、胖模型
- **deployment-verification-agent**：部署前后检查清单、回滚计划

### 审查输出：

你得到优先级列表：

```
P1 - 关键（必须修复）：
[ ] 搜索查询中的 SQL 注入漏洞 (security-sentinel)
[ ] 用户创建缺少事务 (data-integrity-guardian)

P2 - 重要（应该修复）：
[ ] 评论加载中的 N+1 查询 (performance-oracle)
[ ] 控制器做业务逻辑 (kieran-rails-reviewer)

P3 - 次要（可选修复）：
[ ] 未使用的变量 (code-simplicity-reviewer)
[ ] 可用 guard clause (pattern-recognition-specialist)
```

### 并行解决：

运行 `/resolve_pr_parallel`，agent 同时修复所有发现。先 P1，然后 P2。审查修复。

### 没有工具时的三个问题

不是每个人都有十三个审查 agents。原则仍有效。批准任何 AI 输出前，问三个问题：

1. **"这里你做的最难决定是什么？"**
2. **"你拒绝了什么替代方案，为什么？"**
3. **"你最不自信什么？"**

那两分钟的对话呈现十分钟读会错过的。AI 知道棘手部分在哪。它只是不主动呈现，除非你问。

### 复合效果：

每次审查教系统。五十次审查后，agents 知道你的模式。它们捕捉你不知道你关心的问题，因为它们从你的反馈学习了你的偏好。

## 4. 复合 (COMPOUND)

复合阶段是魔法发生的地方。你把这周期的学习变成下周期的优势。

### 复合中发生什么：

1. **捕捉解决方案。** 我们学到了什么？什么有效？什么无效？

2. **使它可找。** 添加 YAML frontmatter、标签、分类。

3. **更新系统。** CLAUDE.md 中新模式。需要时新 agents。

4. **验证学习。** 系统下次会捕捉这个吗？

### 实践中的复合：

你刚修复了一个棘手的 CORS 问题。不走开：

```
/compound
```

Agent：
- 问你问题是什么
- 记录解决方案
- 标记它以便搜索
- 添加到你的文档

下次某人（或某 agent）搜索"CORS production"，他们在五秒内找到你的解决方案，而非调试三小时。

### 什么要复合：

- **Bug 修复**：尤其是诊断花时间的
- **模式**：可应用其他地方的解决方案
- **错误**：下次要避免的东西
- **偏好**：你对方法的选择

**14 个智能技能。** 即时领域专业知识。DHH 风格 Rails 模式。Gemini 图像生成。Andrew Kane gem 模式。

# 斜杠命令参考

插件包含 23 个斜杠命令。这章覆盖每个的细节。

## 核心工作流命令

这些是主循环命令，你会经常使用。

---

### /plan

**目的：** 把功能想法变成详细实现计划。

**何时使用：** 任何非琐碎工作的开始。功能、重构、需要调查的 bug 修复。

**它做什么：**
1. 分析请求确定保真度（小/中/大）
2. 通过 repo-research-analyst 研究代码库
3. 通过 framework-docs-researcher 收集框架文档
4. 通过 best-practices-researcher 找最佳实践
5. 通过 git-history-analyzer 分析 git 历史
6. 创建有文件、方法、边缘情况的详细计划

**例子：**
```
/plan 当用户收到新评论时添加邮件通知
```

**输出：** 结构化计划文档：
- 上下文和需求
- 提议方法
- 要修改/创建的文件
- 要处理的边缘情况
- 测试策略
- 回滚计划

---

### /work

**目的：** 用 worktrees 和任务追踪执行计划。

**何时使用：** 有批准计划后。

**它做什么：**
1. 为工作创建隔离 worktree
2. 逐步实现计划
3. 更改后运行验证（测试、linting）
4. 追踪进度处理问题
5. 完成时创建 PR

---

### /review

**目的：** 运行穷尽多 agent 代码审查。

**何时使用：** 合并任何 PR 前。尤其是生产部署前。

**它做什么：**
1. 并行启动 12+ 专门 agents
2. 每个 agent 从其视角分析代码
3. 发现被收集优先级化（P1/P2/P3）
4. 结果作为可操作 todos 呈现

---

### /compound

**目的：** 记录已解决问题以复合团队知识。

**何时使用：** 解决任何非琐碎问题后。尤其是诊断花时间的 bug。

**它做什么：**
1. 问你解决了什么
2. 创建有 YAML frontmatter 的可搜索文档
3. 标记它以便未来可找
4. 添加到你的复合文档集合

---

### /lfg

**目的：** 完全自主工作流——想法到 PR。

**何时使用：** 准备阶段 4 复合工程时。你描述想要的，agent 做一切。

**它做什么：**
1. 创建计划（带研究）
2. 展示计划给你批准
3. 实现计划
4. 自我审查修复问题
5. 运行完整 `/review`
6. 修复任何发现
7. 复合学习
8. 创建 PR

**例子：**
```
/lfg 在设置页添加暗黑模式切换
```

## 有效命令使用技巧

1. **不要跳过 /plan。** 即使"简单"东西，快速计划早期捕捉问题。

2. **合并前总是 /review。** Agents 捕捉你错过的。

3. **解决难题后 /compound。** 你未来的自己会感谢。

4. **用 /resolve_pr_parallel，非手动修复。** 让 agent 修复审查发现的。

5. **日常功能用 /lfg。** 信任流程后，让它跑。

6. **为你的工作流自定义命令。** 默认是起点。
