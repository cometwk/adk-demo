# Open Knowledge Format (OKF)

版本 0.1 — 草案

OKF 是一种开放的、对人类和 Agent 友好的知识表示格式——用于表示围绕数据和系统的元数据、上下文和精选洞察。它设计为可由人编写、由 Agent 生成、跨组织交换，并可被两者消费。

该格式刻意保持极简：一个由 Markdown 文件和 YAML frontmatter 组成的目录。没有模式注册中心，没有中央权威，也没有必需的工具链。如果你能 `cat` 一个文件，你就能阅读 OKF；如果你能 `git clone` 一个仓库，你就能发布它。

## 1. 动机

AI Agent 的知识表示领域正在快速演进，许多不兼容的约定正在涌现。OKF 的立场是，知识最好以常见的、可访问的、成熟的格式来表示，这些格式：

- 人类无需工具即可阅读。
- Agent 无需定制 SDK 即可解析。
- 在版本控制中可 diff。
- 可跨工具、组织和时间移植。

该格式最小化地表达意见。它仅标准化使知识语料库自描述所需的那一小部分结构约定——除此之外的一切留给生产者。

### 目标

- 定义一种通用格式，使富化 Agent 可以写入。
- 告知消费 Agent 应如何阅读和遍历它。
- 促进跨系统和组织的知识交换。
- 标准化为使内容能被有意义地消费所必须存在的少量必需字段。

### 非目标

- 定义概念类型的固定分类法。
- 规定存储、服务或查询基础设施。
- 替代领域特定的模式（Avro、Protobuf、OpenAPI 等）——OKF 引用它们，而非取代它们。

## 2. 术语

- **知识包 (Knowledge Bundle)** — 一个自包含的、层次化的知识文档集合。分发的单位。
- **概念 (Concept)** — 包内知识的单个单元。表示为一个 Markdown 文档。可以描述有形资产（表、API）、抽象概念（指标、业务流程），或介于两者之间的任何东西。
- **概念 ID (Concept ID)** — 概念文件在包内的路径，去掉 `.md` 后缀。例如，`tables/users.md` 的概念 ID 为 `tables/users`。
- **Frontmatter** — Markdown 文件顶部由 `---` 界定的 YAML 元数据块。
- **正文 (Body)** — frontmatter 之后文件中的所有内容。
- **链接 (Link)** — 从一个概念到另一个概念的标准 Markdown 链接，用于表达隐式父/子层次结构之外的关系。
- **引用 (Citation)** — 从概念到支持正文中主张的外部来源的链接。

## 3. 包结构

一个包是一个 Markdown 文件的目录树。目录结构独立于领域——生产者可以按照对所捕获知识有意义的方式组织概念。

```
path/to/bundle/
├── index.md                      # 可选。目录列表，用于渐进式披露。
├── log.md                        # 可选。更新的按时间历史记录。
├── <concept>.md                  # 包根目录下的概念。
└── <subdirectory>/               # 子目录将概念组织成组。
    ├── index.md
    ├── <concept>.md
    └── <subdirectory>/
        └── …
```

一个包**可以**以以下方式分发：

- Git 仓库（推荐——提供历史、归属、diff）。
- 目录的 tarball 或 zip 归档。
- 更大仓库中的子目录。

### 3.1 保留文件名

以下文件名在层次结构的任何级别都有定义的含义，**不得**用于概念文档：

| 文件名 | 用途 |
|---|---|
| `index.md` | 目录列表。见 §6。 |
| `log.md` | 更新历史。见 §7。 |

所有其他 `.md` 文件都是概念文档。

标签本身仍然是一等概念——见 §4.1 中的 `tags` frontmatter 字段。OKF 不指定单独的文件格式来按标签聚合文档；想要标签浏览视图的生产者可以在消费时通过扫描 frontmatter 来合成一个。

## 4. 概念文档

每个概念都是一个 UTF-8 Markdown 文件。它有两部分：

1. 一个 YAML frontmatter 块，由文件开头独占一行的 `---` 和独占一行的闭合 `---` 界定。
2. 一个 Markdown 正文，包含自由格式的内容。

### 4.1 Frontmatter

```yaml
---
type: <类型名称>                    # 必需
title: <可选显示名称>
description: <可选单行摘要>
resource: <可选底层资产的标准 URI>
tags: [<tag>, <tag>, …]            # 可选
timestamp: <ISO 8601 日期时间>     # 可选的最后修改时间
# … 其他生产者定义的键/值对
---
```

**必需：**

- `type` — 标识概念类型的短字符串。消费者使用它进行路由、过滤和展示。示例值：`BigQuery Table`、`BigQuery Dataset`、`API Endpoint`、`Metric`、`Playbook`、`Reference`。

类型值不在中心注册。生产者**应该**选择具有描述性和自解释性的值；消费者**必须**优雅地容忍未知类型（通常将其视为通用概念）。

**推荐（按优先级排序）：**

- `title` — 人类可读的显示名称。如果省略，消费者**可以**从文件名派生标题。
- `description` — 总结概念的单句描述。由 `index.md` 生成器、搜索片段和预览使用。
- `resource` — 唯一标识概念所描述底层资产的 URI。对于描述抽象概念而非物理资源的概念，此项不存在。
- `tags` — 用于交叉分类的短字符串 YAML 列表。
- `timestamp` — 最后有意义更改的 ISO 8601 日期时间。

**扩展：** 生产者**可以**包含任何额外的键。消费者在往返传输时**应该**保留未知键，**不应**因无法识别的字段而拒绝文档。

### 4.2 正文

正文是标准 Markdown。生产者**应该**优先使用结构化 Markdown——标题、列表、表格、围栏代码块——而非自由散文，因为结构有助于人类阅读和 Agent 检索。

没有必需的正文章节。以下章节标题具有约定含义，在适用时**应该**使用：

| 标题 | 用途 |
|---|---|
| `# Schema` | 资产列/字段的结构化描述。 |
| `# Examples` | 具体使用示例，通常为围栏代码块。 |
| `# Citations` | 支持正文中主张的外部来源。见 §8。 |

### 4.3 示例：绑定到资源的概念

```yaml
---
type: BigQuery Table
title: Customer Orders
description: One row per completed customer order across all channels.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders, revenue]
timestamp: 2026-05-28T14:30:00Z
---
```

```markdown
# Schema

| Column        | Type      | Description                              |
|---------------|-----------|------------------------------------------|
| `order_id`    | STRING    | Globally unique order identifier.        |
| `customer_id` | STRING    | Foreign key into [customers](/tables/customers.md). |
| `total_usd`   | NUMERIC   | Order total in US dollars.               |
| `placed_at`   | TIMESTAMP | When the customer submitted the order.   |

# Joins

Joined with [customers](/tables/customers.md) on `customer_id`.

# Citations

[1] [BigQuery table schema](https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders)
```

### 4.4 示例：未绑定到资源的概念

```yaml
---
type: Playbook
title: Incident response — data freshness alert
description: Steps to triage a freshness alert on the orders pipeline.
tags: [oncall, incident]
timestamp: 2026-04-12T09:00:00Z
---
```

```markdown

A freshness alert fires when `orders` lags more than 30 minutes behind
its expected SLA. See the [orders table](/tables/orders.md).

# Steps

1. Check the [ingestion job dashboard](https://example.com/dash).
2. …
```

## 5. 交叉链接

概念**可以**使用标准 Markdown 链接链接到其他概念。支持两种形式：

### 5.1 绝对（包相对）链接

以 `/` 开头，相对于包根目录解释。

```markdown
See the [customers table](/tables/customers.md) for the join key.
```

这是推荐的形式，因为当文档在其子目录内移动时它是稳定的。

### 5.2 相对链接

标准 Markdown 相对路径。

```markdown
See the [neighboring concept](./other.md).
```

### 5.3 链接语义

从概念 A 到概念 B 的链接断言了一种关系。关系的具体类型（父/子、引用、连接、依赖等）由周围文本传达，而非链接本身。构建图视图的消费者通常将所有链接视为无类型关系的有向边。

消费者**必须**容忍断链——目标在包中不存在的链接不是格式错误的；它可能仅仅代表尚未编写的知识。

## 6. 索引文件

`index.md` 文件**可以**出现在任何目录中，包括包根目录。它枚举目录的内容以支持渐进式披露——让人类或 Agent 在打开单个文档之前看到可用的内容。

索引文件不包含 frontmatter。正文使用一个或多个章节，每个章节在标题下对概念进行分组：

```markdown
# Section / Group Heading

* [Title 1](relative-url-1) - short description of item 1
* [Title 2](relative-url-2) - short description of item 2

# Another Section

* [Subdirectory](subdir/) - short description of the subdirectory
```

条目**应该**包含被链接概念 frontmatter 中的描述。生产者**可以**自动生成 `index.md`；消费者在没有 `index.md` 时**可以**即时合成一个。

## 7. 日志文件（可选）

`log.md` 文件**可以**出现在层次结构的任何级别，以记录该范围内更改的历史。格式是按日期分组的条目平面列表，最新在前：

```markdown
# Directory Update Log

## 2026-05-22
* **Update**: Added new BigQuery table reference for [Customer Metrics](/tables/customer-metrics.md).
* **Creation**: Established the [Dataplex Playbook](/playbooks/dataplex.md).

## 2026-05-15
* **Initialization**: Created foundational directory structure.
* **Update**: Added progressive-disclosure guidelines to the root [index](/index.md).
```

日期标题**必须**使用 ISO 8601 `YYYY-MM-DD` 格式。日志条目是散文；开头的粗体词（**Update**、**Creation**、**Deprecation** 等）是惯例，不是要求。

## 8. 引用

当概念的正文中的主张来源于外部材料时，这些来源**应该**列在文档底部的 `# Citations` 标题下，带编号：

```markdown
# Citations

[1] [BigQuery public dataset announcement](https://cloud.google.com/blog/products/data-analytics/...)
[2] [Internal data quality runbook](https://wiki.acme.internal/data/quality)
```

引用链接**可以**是绝对 URL、包相对路径，或映射到 `references/` 子目录中的路径，该子目录将外部材料镜像为一等 OKF 概念。

## 9. 一致性

一个包符合 OKF v0.1 如果：

- 树中每个非保留的 `.md` 文件包含可解析的 YAML frontmatter 块。
- 每个 frontmatter 块包含非空的 `type` 字段。
- 每个保留文件名（`index.md`、`log.md`）在存在时分别遵循 §6 和 §7 中描述的结构。

消费者**应该**将所有其他约束视为软指导。特别是，消费者**不得**因以下原因拒绝包：

- 缺少可选的 frontmatter 字段。
- 未知的类型值。
- 未知的额外 frontmatter 键。
- 断开的交叉链接。
- 缺少 `index.md` 文件。

这种宽松的消费模型是有意为之的：OKF 旨在当包增长、重构和部分由 Agent 生成时仍然保持有用。

## 10. 与其他格式的关系

OKF 有意地与几种既有模式接近：

- 使用 Markdown + frontmatter 作为 Agent 可读知识库的 LLM "wiki" 仓库。
- 如 Obsidian 和 Notion 等个人知识工具，使用带交叉链接的层次化 Markdown。
- "元数据即代码" 方法，将目录元数据存储在源代码旁边而非单独的注册中心。

OKF 的主要区别在于它是被规范的——确定了互操作性所需的那一小部分规则，而不指定工具链。

## 11. 版本控制

本文档指定 OKF 版本 0.1。未来的修订将以 `<major>.<minor>` 的形式版本化：

- 次版本号升级引入向后兼容的新增内容（新的可选字段、新的约定章节标题）。
- 主版本号升级可能进行破坏性变更（重命名必需字段、更改保留文件名）。

包**可以**通过在包根目录的 `index.md` frontmatter 块中包含 `okf_version: "0.1"` 来声明它们目标的 OKF 版本（这是 `index.md` 中唯一允许 frontmatter 的地方）。不理解声明版本的消费者**应该**尝试尽力消费而非拒绝包。

## 附录 A — 最小示例包

```
my_bundle/
├── index.md
├── datasets/
│   ├── index.md
│   └── sales.md
└── tables/
    ├── index.md
    ├── orders.md
    └── customers.md
```

`datasets/sales.md`：

```yaml
---
type: BigQuery Dataset
title: Sales
description: All sales-related tables for the retail business.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales
tags: [sales]
timestamp: 2026-05-28T00:00:00Z
---
```

```markdown
The sales dataset contains transactional tables, including
[orders](/tables/orders.md) and [customers](/tables/customers.md).
```

`tables/orders.md`：

```yaml
---
type: BigQuery Table
title: Orders
description: One row per completed customer order.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders]
timestamp: 2026-05-28T00:00:00Z
---
```

```markdown
# Schema

| Column        | Type      | Description                  |
|---------------|-----------|------------------------------|
| `order_id`    | STRING    | Unique order identifier.     |
| `customer_id` | STRING    | FK to [customers](/tables/customers.md). |
| `total_usd`   | NUMERIC   | Order total in USD.          |

Part of the [sales dataset](/datasets/sales.md).
```
