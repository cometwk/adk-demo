# NanoClaw Skills 架构

## Skills 的用途

NanoClaw 的核心设计故意保持最小化。Skills 是用户扩展它的方式：添加频道、集成、跨平台支持，或完全替换内部实现。例如：在 WhatsApp 之外添加 Telegram、从 Apple Container 切换到 Docker、添加 Gmail 集成、添加语音消息转录。每个 skill 修改实际代码库，添加频道处理器、更新消息路由器、更改容器配置、添加依赖项，而不是通过插件 API 或运行时钩子工作。

## 为什么选择这种架构

问题在于：用户需要对共享代码库组合多个修改，在核心更新时保持这些修改正常工作，而且所有这些都不需要成为 git 专家或丢失自定义更改。插件系统会更简单，但会限制 skills 能做的事情。给 skills 完整的代码库访问权限意味着它们可以更改任何内容，但这会造成合并冲突、更新破坏和状态跟踪的挑战。

这种架构通过使用标准 git 机制使 skill 应用完全程序化来解决这些问题，AI 作为 git 无法解决冲突时的后备方案，以及共享解决方案缓存使大多数用户永远不会遇到这些冲突。结果是：用户可以组合他们想要的确切功能，自定义设置在核心更新时自动保持，系统始终可恢复。

## 核心原则

Skills 是通过标准 git 合并机制应用的自包含、可审计的包。Claude Code 编排整个过程 — 运行 git 命令、读取 skill 清单，只在 git 无法解决冲突时介入。系统使用现有的 git 功能（`merge-file`、`rerere`、`apply`）而不是自定义合并基础设施。

## 三级解决方案模型

每个操作都遵循这个升级流程：

1. **Git** — 确定性的。`git merge-file` 合并，`git rerere` 重放缓存的解决方案，结构化操作无需合并即可应用。无 AI。处理绝大多数情况。
2. **Claude Code** — 读取 `SKILL.md`、`.intent.md` 和 `state.yaml` 来解决 git 无法处理的冲突。通过 `git rerere` 缓存解决方案，这样相同的冲突不需要解决两次。
3. **Claude Code + 用户输入** — 当 Claude Code 缺乏足够的上下文来确定意图时（例如，两个功能在应用层面真正冲突），它会询问用户做出决定，然后使用该输入执行解决方案。Claude Code 仍然做实际工作 — 用户提供方向，而不是代码。

**重要**：干净的合并不保证代码能工作。语义冲突可能产生干净的文本合并但在运行时失败。**每次操作后都会运行测试。**

## 备份/恢复安全机制

在任何操作之前，所有受影响的文件都会被复制到 `.nanoclaw/backup/`。成功时，备份被删除。失败时，备份被恢复。对于不使用 git 的用户也能安全工作。

## 共享基础

`.nanoclaw/base/` 保存核心代码库的干净副本。这是所有三方合并的单一共同祖先，只在核心更新时更新。

## 两种类型的更改

### 代码文件（三方合并）
Skills 编织逻辑的源代码。通过 `git merge-file` 与共享基础合并。Skills 携带完整的修改文件。

### 结构化数据（确定性操作）
像 `package.json`、`docker-compose.yml`、`.env.example` 这样的文件。Skills 在清单中声明需求；系统以编程方式应用它们。多个 skills 的声明被批量处理 — 依赖项合并，`package.json` 写入一次，`npm install` 运行一次。

```yaml
structured:
  npm_dependencies:
    whatsapp-web.js: "^2.1.0"
  env_additions:
    - WHATSAPP_TOKEN
  docker_compose_services:
    whatsapp-redis:
      image: redis:alpine
      ports: ["6380:6379"]
```

结构化冲突（版本不兼容、端口冲突）遵循相同的三级解决方案模型。

## Skill 包结构

一个 skill 只包含它添加或修改的文件。修改的代码文件携带**完整文件**（干净的核心 + skill 的更改），使 `git merge-file` 直观且可审计。

```
skills/add-whatsapp/
  SKILL.md                    # 这个 skill 做什么以及为什么
  manifest.yaml               # 元数据、依赖项、结构化操作
  tests/whatsapp.test.ts      # 集成测试
  add/src/channels/whatsapp.ts          # 新文件
  modify/src/server.ts                  # 用于合并的完整修改文件
  modify/src/server.ts.intent.md        # 用于冲突解决的结构化意图
```

### 意图文件
每个修改的文件都有一个 `.intent.md`，包含结构化标题：**这个 skill 添加什么**、**关键部分**、**不变量**和**必须保留的部分**。这些为 Claude Code 在冲突解决期间提供具体指导。

### 清单
声明：skill 元数据、核心版本兼容性、添加/修改的文件、文件操作、结构化操作、skill 关系（冲突、依赖、测试配合）、应用后命令和测试命令。

## 自定义和分层

**一个 skill，一条最佳路径** — skill 为 80% 的用户实现合理的默认值。

**自定义是更多的补丁。** 应用 skill，然后通过跟踪的补丁、直接编辑或额外的分层 skills 进行修改。自定义修改记录在 `state.yaml` 中并可重放。

**Skills 通过 `depends` 分层。** 扩展 skills 构建在基础 skills 之上（例如，`telegram-reactions` 依赖于 `add-telegram`）。

## 文件操作

重命名、删除和移动在清单中声明，并在代码合并**之前**运行。当核心重命名文件时，**路径重映射**在应用时解决 skill 引用 — skill 包永远不会被修改。

## 应用流程

1. 预检查（兼容性、依赖项、未跟踪的更改）
2. 备份
3. 文件操作 + 路径重映射
4. 复制新文件
5. 合并修改的代码文件（`git merge-file`）
6. 冲突解决（共享缓存 → `git rerere` → Claude Code → Claude Code + 用户输入）
7. 应用结构化操作（批量）
8. 应用后命令，更新 `state.yaml`
9. **运行测试**（强制性，即使所有合并都是干净的）
10. 清理（成功时删除备份，失败时恢复）

## 共享解决方案缓存

`.nanoclaw/resolutions/` 附带预计算、验证的冲突解决方案，带有**哈希强制** — 缓存的解决方案只有在基础、当前和 skill 输入哈希完全匹配时才应用。这意味着大多数用户永远不会遇到常见 skill 组合的未解决冲突。

### rerere 适配器
`git rerere` 需要 `git merge-file` 不会创建的未合并索引条目。适配器在 `merge-file` 产生冲突后设置所需的索引状态，启用 rerere 缓存。这要求项目是 git 仓库；没有 `.git/` 的用户失去缓存但不会失去功能。

## 状态跟踪

`.nanoclaw/state.yaml` 记录：核心版本、所有已应用的 skills（带有基础/skill/合并的每个文件哈希）、结构化操作结果、自定义补丁和路径重映射。这使得漂移检测即时且重放确定性。

## 未跟踪的更改

直接编辑通过哈希比较在任何操作之前被检测。用户可以将它们记录为跟踪的补丁、继续未跟踪或中止。三级模型始终可以从任何起点恢复一致状态。

## 核心更新

大多数更改通过三方合并自动传播。**破坏性更改**需要**迁移 skill** — 一个保留旧行为的普通 skill，针对新核心编写。迁移在 `migrations.yaml` 中声明并在更新期间自动应用。

### 更新流程
1. 预览更改（仅 git，不修改文件）
2. 备份 → 文件操作 → 三方合并 → 冲突解决
3. 重新应用自定义补丁（`git apply --3way`）
4. **更新基础**到新核心
5. 应用迁移 skills（自动保留用户的设置）
6. 重新应用更新的 skills（仅版本更改的 skills）
7. 重新运行结构化操作 → 运行所有测试 → 清理

用户在更新期间不会看到任何提示。要稍后接受新的默认值，他们删除迁移 skill。

## Skill 移除

卸载是**不包含该 skill 的重放**：读取 `state.yaml`，移除目标 skill，使用解决方案缓存从干净的基础重放所有剩余 skills。备份以确保安全。

## 变基

将累积的层扁平化为干净的起点。更新基础，重新生成差异，清除旧补丁和过时的缓存条目。用单独的 skill 历史换取更简单的未来合并。

## 重放

给定 `state.yaml`，在没有 AI 的情况下在全新机器上重现精确安装（假设有缓存的解决方案）。按顺序应用 skills、合并、应用自定义补丁、批量结构化操作、运行测试。

## Skill 测试

每个 skill 包含集成测试。测试**始终运行** — 应用后、更新后、卸载后、重放期间、CI 中。CI 测试所有官方 skills 单独和共享修改文件或结构化操作的 skills 的成对组合。

## 设计原则

1. **使用 git，不要重新发明它。**
2. **三级解决方案：git → Claude Code → Claude Code + 用户输入。**
3. **干净的合并不够。** 每次操作后运行测试。
4. **所有操作都是安全的。** 备份/恢复，没有半应用状态。
5. **一个共享基础**，只在核心更新时更新。
6. **代码合并 vs 结构化操作。** 源代码被合并；配置被聚合。
7. **解决方案被学习并共享**，带有哈希强制。
8. **一个 skill，一条最佳路径。** 自定义是更多的补丁。
9. **Skills 分层和组合。**
10. **意图是一等公民且结构化的。**
11. **状态是显式且完整的。** 重放是确定性的。
12. **始终可恢复。**
13. **卸载即重放。**
14. **核心更新是维护者的责任。** 破坏性更改需要迁移 skills。
15. **文件操作和路径重映射是一等公民。**
16. **Skills 被测试。** CI 按重叠测试成对组合。
17. **确定性序列化。** 没有噪声差异。
18. **需要时变基。**
19. **通过迁移 skills 逐步精简核心。**
