# 数据库表结构分析与关联关系

> 本文档描述支付系统核心表的设计意图和关联关系，供 Code Agent 参考。

---

## 1. 表清单

| 表名 | 职责 | 类型 |
|------|------|------|
| `agent` | 代理商主表 | 实体表 |
| `agent_closure` | 代理商层级闭包表 | 关系表 |
| `agent_rel` | 代理商与商户/通道绑定关系 | 关系表 |
| `merch` | 商户主表 | 实体表 |
| `apply` | 进件申请记录 | 商户进件登记表 |
| `order_daily` | 商户日交易统计 | 统计表 |
| `profit_daily` | 代理商日分润统计 | 统计表 |

  ---
  3. 设计意图分析

  3.1 代理商层级设计 (agent + agent_closure)

  采用**闭包表(Closure Table)**模式存储层级结构：

  agent.parent_id     → 直接父节点（辅助查询）
  agent_closure       → 存储所有祖先-后代关系，支持快速查询整棵子树

  优点：
  - 查询所有子孙节点：WHERE ancestor_id = ?
  - 查询所有祖先节点：WHERE descendant_id = ?
  - 支持 depth 字段快速获取层级深度

  3.2 代理关系设计 (agent_rel)

  这是多态关联设计，通过 agent_type 区分：

  ┌────────────┬─────────────┬─────────────┐
  │ agent_type │ obj_no 含义 │ obj_id 含义 │
  ├────────────┼─────────────┼─────────────┤
  │ MERCH      │ 商户编号    │ 商户ID      │
  ├────────────┼─────────────┼─────────────┤
  │ CHAN       │ 通道编号    │ 通道ID      │
  └────────────┴─────────────┴─────────────┘

  关键字段：
  - rate: 实际参与分润计算的费率（十万分比）
  - mode: 费率模式（固定/占比）
  - apply=1: 标识该代理商是商户的进件人

  3.3 分润计算流程

  order_daily (商户日交易)
        ↓
  按 merch_no 聚合，关联 agent_rel
        ↓
  profit_daily (代理商日分润)

  profit_daily 双重统计：
  - total_*: 该代理商名下所有商户的分润
  - own_*: 仅统计 apply=1（自己进件的商户）的分润

  3.4 进件流程

  apply (申请) → status=SUCCESS → merch (创建商户)
                             ↓
                      agent_rel (创建代理关系)

  ---
  4. 关键关联关系

  agent.agent_no ──────┬──► agent_rel.agent_no
                        │
                        └──► apply.agent_no
                              └──► merch.merch_no (申请成功后)
                                    └──► order_daily.merch_no
                                          └──► profit_daily (计算分润)

  agent_rel.obj_no ────► merch.merch_no (当 agent_type='MERCH')

  ---
  5. 数据流向

  进件流程:
    apply → merch → agent_rel (apply=1)

  交易统计:
    外部导入 → order_daily

  分润结算:
    order_daily + agent_rel → profit_daily

  这套设计支持：
  1. 多级代理：通过闭包表实现任意层级代理体系
  2. 分润计算：通过 agent_rel.rate 存储实际分润比例
  3. 进件追踪：apply 标志区分是否为直接进件商户
  4. 日级结算：profit_daily 支持 T+1 结算流程
