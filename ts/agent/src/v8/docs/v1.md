# Graph Query 设计文档

> 本文档描述 V6 Graph 层的**查询能力设计**：声明式 `graph_query`、存储抽象 `GraphStore`、以及四个 Agent 图工具的分工原则。  
> 本体与渐进式披露的基础设计见 [1-graph-design.md](../../../docs/design/1-graph-design.md)

---

## 1. 问题背景

### 1.1 命令式 API 的局限

V6 最初提供三个**命令式单跳**图工具：

| 工具 | 能力 |
|------|------|
| `inspect_node` | 按 ID 查看单个节点 |
| `query_neighbors` | 查看某节点的 1 跳邻居 |
| `search_nodes` | 按类型/ID 子串搜索节点 |

它们适合「已知入口、逐跳探索」的简单场景，但面对**多跳 + 属性过滤 + 聚合**时会失控：

```text
需求：找出「借了科学类书、且有 overdue」的 gold 卡 Reader

命令式路径（约 10+ 轮 tool call）：
  search_nodes({ type: "Reader" })
  → 对每个 Reader: inspect_node（看 membershipLevel）
  → 对每个 gold Reader: query_neighbors({ relation: "borrows" })
  → 对每本书: inspect_node（看类目）
  → 再 query_neighbors({ relation: "overdue" })
  ...
```

**Context 消耗为 O(节点数 × 跳数)**，且中间结果（大量 ID）对 Agent 几乎无用。

### 1.2 信息梯度断裂（改造前）

| | 发现节点 | 看到属性 | 属性过滤 | 多跳 |
|---|:---:|:---:|:---:|:---:|
| `search_nodes` | ✓（仅 ID） | ✗ | ✗ | ✗ |
| `query_neighbors` | ✓（邻居 ID） | ✗ | ✗ | 1 跳 |
| `inspect_node` | ✗（需已知 ID） | ✓ | ✗ | ✗ |
| （缺失） | — | — | — | N 跳 |

前三个工具各管一块、每块不完整，**逼迫 Agent 做 N+1**：先拿 ID，再逐个 `inspect_node` 看属性。

### 1.3 存储层假设

早期实现将整张图预加载进内存（`seedGraph()` → `Map` + `Edge[]`），查询引擎直接 `for...of` 全表扫描。这在 demo 可行，但无法对接数据库，且与「渐进披露」在大图上的目标矛盾。

## 2. 设计原则

| 原则 | 含义 | 落地 |
|------|------|------|
| **声明式优先于命令式拼接** | 复杂查询用一条 JSON 表达意图，由引擎一次执行 | `graph_query`：MATCH → TRAVERSE → RETURN |
| **过滤与投影下推** | `where` / `fields` 在存储层执行，不把全量数据交给 Agent 再筛 | `GraphStore.findNodes` / `getNeighbors` |
| **信息梯度连续** | 简单查询用轻量工具，复杂查询用声明式工具，能力不重叠、不断层 | 四工具分工（见 §4） |
| **存储与查询解耦** | Agent 工具依赖 `GraphStore` 接口，不依赖具体 `Map` 实现 | `InMemoryGraphStore` / 未来 `SqlGraphStore` |
| **边摘要代替边枚举** | 不把 10 万条邻居 ID 塞进 context | `inspect_node` 返回 `edgeSummary`（count only） |
| **类型必填防全表扫** | 无类型的全局搜索在大图上无意义 | `search_nodes.type` 必填 |
| **策略贯穿** | 查询每一步都走 `PolicyContext` | 工具层 + `GraphQueryEngine` 内检查 |
| **分页防爆** | 列表结果默认分页，工作集有上限 | 默认 limit=20/50，MAX_WORKING_SET=1000 |

## 3. 架构总览

```text
                    Agent（LLM Executor）
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
   inspect_node      query_neighbors      search_nodes
   （单节点详情）      （1 跳邻居）        （按类型搜索）
         │                 │                 │
         └─────────────────┼─────────────────┘
                           │
                    graph_query
              （多跳 / 过滤 / 聚合，1 次 tool call）
                           │
                           ▼
                  GraphQueryEngine
                  （MATCH → TRAVERSE → RETURN）
                           │
                           ▼
                     GraphStore 接口
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
     InMemoryGraphStore           SqlGraphStore
     （demo / 测试）              （未来：数据库）
```

### 3.1 模块文件

```
src/v6/runtime/
├── graph-store.ts      — GraphStore 接口 + NodeData / NeighborData / EdgeSummary
├── graph-filters.ts    — PropertyFilter 求值、fields 投影（store 与 engine 共用）
├── graph.ts            — InMemoryGraphStore、BaseNode
├── query-types.ts      — GraphQuery Zod schema（MATCH / TRAVERSE / RETURN）
└── query-engine.ts     — GraphQueryEngine.execute()

src/v6/agent/tools/
└── graph.ts            — createGraphTools(store, policy, facts?)
```

## 4. 四工具分工：何时用什么

改造后的信息梯度：

| 工具 | 适用场景 | 带回属性 | where 过滤 | 跳数 |
|------|----------|:---:|:---:|:---:|
| `inspect_node` | 已知 nodeId，看详情/方法/边摘要 | 全量 | — | 0 |
| `search_nodes` | 按类型发现节点（如「所有 gold Reader」） | 可选 `fields` | ✓ | 0 |
| `query_neighbors` | 从已知节点展开 1 跳（如「小明借的书」） | 可选 `fields` | ✓ | 1 |
| `graph_query` | 2+ 跳、存在性断言、聚合统计 | ✓ | ✓ | N |

**选择规则（写给 Agent / Prompt）：**

1. **已有 nodeId** → `inspect_node`（不要为看属性先 `search_nodes`）
2. **知道类型 + 条件，不知 ID** → `search_nodes` + `where` + `fields`
3. **从某节点看邻居** → `query_neighbors` + `where` + `fields`（不要邻居 ID 再 N 次 `inspect_node`）
4. **2 跳及以上 / 要聚合 / 要 exists 过滤** → `graph_query` 一次完成

## 5. GraphStore：存储抽象

查询不直接依赖 `Map<string, BaseNode>`，而依赖 **`GraphStore` 接口**。节点以 **`NodeData` DTO** 返回，与 `BaseNode` 类实例解耦，便于对接 SQL/REST。

```typescript
interface GraphStore {
  getNode(id: string): Promise<NodeData | undefined>

  findNodes(opts: {
    type: string              // 必填
    where?: PropertyFilter[]
    fields?: string[]
    limit?: number
    offset?: number
  }): Promise<Paginated<NodeData>>

  getNeighbors(nodeId: string, opts?: {
    relation?: string
    direction?: 'out' | 'in' | 'both'
    targetType?: string
    where?: PropertyFilter[]
    fields?: string[]
    limit?: number
    offset?: number
  }): Promise<Paginated<NeighborData>>

  getEdgeSummary(nodeId: string): Promise<EdgeSummary[]>
}
```

| DTO | 用途 |
|-----|------|
| `NodeData` | `{ id, type, properties }` |
| `NeighborData` | `{ nodeId, type, relation, direction, properties? }` |
| `EdgeSummary` | `{ relation, direction, targetType, count }` — **不含邻居 ID 列表** |

**实现：**

- **`InMemoryGraphStore`**：当前默认；`seedGraph()` 仍 `addNode` / `addEdge`，测试与 demo 使用。
- **`getBaseNode(id)`**（仅内存实现）：`call_method` 等仍需在 `BaseNode` 实例上执行业务方法，与 `GraphStore.getNode()` 并存。
- **未来 `SqlGraphStore`**：将 `PropertyFilter` 译为 SQL `WHERE`，`getNeighbors` 走索引，接口不变。

## 6. 命令式工具接口（改造后）

### 6.1 `search_nodes`

```typescript
{
  type: string                    // 必填
  where?: PropertyFilter[]
  fields?: string[]               // 提供则结果含 properties
  limit?: number                  // 默认 20
  offset?: number
}
```

- 已删除：`query`（ID 子串，UUID 场景无用）、`relatedTo`（与 `query_neighbors` 重复）。

**示例：**

```json
{
  "type": "Reader",
  "where": [{ "property": "membershipLevel", "op": "eq", "value": "gold" }],
  "fields": ["name"]
}
```

### 6.2 `query_neighbors`

```typescript
{
  nodeId: string
  relation?: string
  direction?: 'out' | 'in' | 'both'
  targetType?: string
  where?: PropertyFilter[]
  fields?: string[]
  limit?: number
  offset?: number
}
```

**示例：**

```json
{
  "nodeId": "xiao_ming",
  "relation": "borrows",
  "fields": ["title", "daysOnShelf"]
}
```

### 6.3 `inspect_node`

```typescript
{
  nodeId: string
  fields?: ('type' | 'properties' | 'edgeSummary' | 'methods')[]
  at?: string   // ISO 8601，诊断模式时间旅行
}
```

- **`edgeSummary`** 替代原 `outEdges` / `inEdges`：只返回 `{ relation, direction, targetType, count }`，避免大图下 ID 列表撑爆 context。
- 需要邻居详情时，Agent 应使用 `query_neighbors` 分页获取。

## 7. 声明式查询：`graph_query`

### 7.1 为何用 JSON 而非 Cypher

| 方案 | 优点 | 缺点 |
|------|------|------|
| 字符串 DSL（Cypher 类） | 简洁 | 需 parser；LLM 易写错语法 |
| GraphQL | 生态成熟 | 偏 REST 拉取，不适合图遍历语义 |
| **JSON + Zod** | 零 parser；LLM 擅长 JSON；schema 即文档 | 略冗长 |

原型阶段采用 **JSON 结构化查询**：`GraphQuerySchema` 即语法，`graph_query` 的 `inputSchema` 即 Agent 可见文档。

### 7.2 查询管线：MATCH → TRAVERSE → RETURN

```text
MATCH
  type + where + alias
       │
       ▼
TRAVERSE[]  （可重复多步）
  from? / relation / direction / targetType / where / alias / require
       │
       ▼
RETURN
  alias / fields / limit / offset / aggregate
```

#### MATCH — 起点集合

```typescript
{
  type: "Reader",
  where?: [{ property: "membershipLevel", op: "eq", value: "gold" }],
  alias?: "reader"    // 默认 "_start"
}
```

引擎调用 `store.findNodes()`，得到初始节点 ID 集合。

#### TRAVERSE — 边遍历步骤

```typescript
{
  from?: "reader",           // 从哪个 alias 出发；省略则用上一阶段
  relation: "borrows",
  direction?: "out",         // 默认 out
  targetType?: "Book",
  where?: [{ property: "daysOnShelf", op: "lt", value: 7 }],
  alias?: "book",
  require?: "exists" | "none"
}
```

| `require` | 语义 |
|-----------|------|
| `exists` | 源节点**至少有一个**满足条件的目标，否则从源集合剔除 |
| `none` | 源节点**没有**满足条件的目标，否则剔除（反向过滤） |
| （省略） | 只扩展目标集合，不反向过滤源 |

多步遍历时用 **`alias`** 命名工作集，后续步骤可通过 `from` 引用任意历史 alias。

#### RETURN — 输出

```typescript
{
  alias?: "reader",
  fields?: ["name", "membershipLevel"],
  limit?: 50,
  offset?: 0,
  aggregate?: {
    groupBy?: "isRestricted",
    metrics: [{ field: "*", fn: "count", as: "cnt" }]
  }
}
```

| 模式 | 返回 |
|------|------|
| 普通 | `{ mode: "nodes", rows: QueryRow[], total, truncated }` |
| 聚合 | `{ mode: "aggregate", rows: QueryAggregateRow[] }` |

聚合函数：`count` | `sum` | `avg` | `min` | `max`；`field: "*"` 表示计数。

### 7.3 PropertyFilter

```typescript
{
  property: string
  op: 'eq' | 'ne' | 'gt' | 'gte' | 'lt' | 'lte' | 'contains' | 'in'
  value: unknown    // op 为 in 时传数组
}
```

多个条件为 **AND** 关系。求值逻辑在 `graph-filters.ts`，`GraphStore` 与 `GraphQueryEngine` 共用。

### 7.4 执行限制（防爆）

| 常量 | 值 | 含义 |
|------|-----|------|
| 默认 RETURN limit | 50 | 单次返回行数 |
| MAX_LIMIT | 200 | RETURN 上限 |
| MAX_WORKING_SET | 1000 | 中间工作集 / 匹配扫描上限 |

超限时 `truncated: true`，Agent 应收紧 `where` 或缩小 `match` 范围。

## 8. 示例（图书馆场景）

图数据见 `src/v6/tests/1-graph/library/seed.ts`（4 Reader、9 Book、多跳关系）。

### 8.1 简单：gold 卡读者

等价于 `search_nodes`，但也可用于与其他 TRAVERSE 组合：

```json
{
  "match": {
    "type": "Reader",
    "where": [{ "property": "membershipLevel", "op": "eq", "value": "gold" }]
  },
  "return": { "fields": ["name", "membershipLevel"] }
}
```

### 8.2 存在性：有逾期书的 Reader

```json
{
  "match": { "type": "Reader", "alias": "reader" },
  "traverse": [{ "relation": "overdue", "direction": "out", "require": "exists" }],
  "return": { "alias": "reader" }
}
```

→ 仅 `xiao_li`。

### 8.3 两跳：读者 → 借阅书 → 作者

```json
{
  "match": {
    "type": "Reader",
    "where": [{ "property": "name", "op": "eq", "value": "小明" }],
    "alias": "reader"
  },
  "traverse": [
    { "relation": "borrows", "direction": "out", "targetType": "Book", "alias": "book" },
    { "from": "book", "relation": "written_by", "direction": "out", "targetType": "Author", "alias": "author" }
  ],
  "return": { "alias": "author" }
}
```

→ `author_liu`。

### 8.4 聚合：Book 总数

```json
{
  "match": { "type": "Book" },
  "return": {
    "aggregate": {
      "metrics": [{ "field": "*", "fn": "count", "as": "total" }]
    }
  }
}
```

→ `total: 9`。

### 8.5 对比：改造前后轮次

| 任务 | 改造前 | 改造后 |
|------|--------|--------|
| gold Reader + 属性 | `search_nodes` + N×`inspect_node` | `search_nodes` 一步（带 `fields`） |
| 有 overdue 的 Reader | 全量 Reader + 逐个查 overdue 边 | `graph_query` 1 步（`require: exists`） |
| 读者→书→作者 | 约 6+ 步 | `graph_query` 1 步 |

## 9. 与 FactStore / Policy 的集成

- **FactStore 叠加**：`inspect_node` / `graph_query` 的 RETURN 在返回属性前，将 `FactStore` 中已绑定事实覆盖到图属性（与 [1-graph-design.md](./1-graph-design.md) §10 一致）。
- **PolicyContext**：`findNodes` / `getNeighbors` / `getNode` 结果经 `checkEntityAccess`、`checkTypeAccess`；属性经 `redactProperties` 脱敏。
- **审计**：各工具 `execute` 开头调用 `maybeLogToolCall`（若 policy 开启）。
