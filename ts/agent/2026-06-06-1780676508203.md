- **日期**: 2026年6月3日
- **阅读时间**: 5 分钟

正如许多数据科学和数据工程团队所深知的，实现自助式业务分析历来是一项苦差事。

通过宽表和反范式化表让技术背景较弱的同事更容易访问数据模型，往往会随着业务扩展而产生定义不一致的重叠视图（而且对于那些不想学习 SQL 的员工来说，这几乎无济于事）。另一方面，为用户创建更多隔离环境通常无法覆盖业务问题的长尾需求，并导致指标和仪表板的膨胀，因为各团队各自为战。

LLM 的兴起为自助式分析提供了一条避免上述挑战的额外路径。然而，将 Claude 指向一个数据仓库并让 agent 自由执行，可能会产生一种虚假的精确感。

最初从临时需求中解放出来的兴奋，很快转变为恐惧——因为人们意识到这种设置将利益相关者与底层基础设施、文档和专业知识隔离开来，而正是这些基础设施、文档和专业知识此前引导他们使用精心策划的数据集。

在 Anthropic，**95% 的业务分析查询通过 Claude 自动完成，总体准确率约为 95%**。通过将这些通常枯燥、重复的工作交给 Claude，我们的数据科学团队可以专注于更具战略性的工作，如因果建模、预测和机器学习。

在与数十位 Anthropic 顶级 Claude Code 用户交流，并见识了分析 agent 的众多设计模式之后，我们总结了一些最佳实践，供其他与 LLM 合作的数据团队参考。在这篇文章中，我们将分享这些技巧和方法，以最大化 Claude 驱动自助式业务洞察的能力，包括：

- 为什么分析准确性是一个上下文和验证问题，而非代码生成问题；
- 导致大多数错误的三大失败模式；
- 我们为解决这些错误而构建的 Agent 分析技术栈；
- 我们如何衡量效果；以及
- 我们创建大多数 skill 的基础模板（见附录）

## 数据不是软件

LLM 的生成能力是一把双刃剑：使模型能够创造性地解决复杂问题的机制，同样也可能产生错误的输出。要充分理解分析 agent 的挑战，将它们与编码 agent 进行比较是有益的。

编码是一个开放式的解决方案空间，奖励模型的创造力，而文档和测试则提供了防止幻觉的自然护栏。相比之下，对于分析用例，通常只有一个使用单一正确数据源的正确答案，而且没有确定性的方法来证明其正确性。

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a20480bedac32484c00d6b9_4a7645b6.png)

对于自助式 Agent 业务分析，复杂性主要在于数据的歧义性。核心问题归结为我们**将用户问题映射到数据模型中特定且最新的实体的能力，以及知道正确使用它们的方法**。如果我们能做到这一点，那么后续的执行和 SQL 就变得微不足道了。

我们发现了导致绝大多数不准确回答的三个因素：

1. **概念与实体的歧义** (**Concept <> entity ambiguity**)：数据模型中有数百个可行选项（在潜在的数百万个字段中），agent 无法选择最能回答用户问题的正确字段。例如，在衡量活跃用户数量时：什么行为构成"活跃"？是否包括欺诈用户？使用什么回溯窗口？

2. **数据过期**：数据源、业务定义和模式不断变化；资产和 agent 知识会过时，开始返回微妙错误的答案。

3. **检索失败**：正确的信息实际上可能就在数据模型中并已正确标注，但由于搜索空间的广阔，agent 就是找不到它。

## 我们的 Agent 分析技术栈

在 Anthropic，我们最小化这三种错误的主要方式是通过我们的 Agent 数据技术栈。每一层主要针对一个或多个问题：

1. **实体歧义**：数据基础层和真相源缩小了合理实体的空间，直到只有一个受治理的答案。
2. **过期问题**：维护和验证流程确保一切不会随着业务变化而腐化。
3. **检索失败**：skill 确保 agent 可靠地找到并正确使用该答案。

在本节中，我们将讨论如何构建每一层。

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a2049920443016925a3ef72_74528df2.png)

### 数据基础层

确保分析 agent 准确的最重要方面是通过强大的数据基础，包括数据仓库中的数据模型、转换、测试和表，以及描述它们的元数据。标准的数据工程和数据质量实践，如[维度建模](https://en.wikipedia.org/wiki/Dimensional_modeling)、左移测试、关键管道的新鲜度和完整性检查，仍然适用（我们不会在这里重新讨论这些）。

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a2049920443016925a3ef75_98412372.png)

确实发生变化的是数据模型的最终用户不再是数据专家（例如数据科学家），而是代表具有不同程度数据专业知识或对底层基础设施理解的用户行事的 agent。这种转变带来了一个挑战：结果不能要求用户验证底层正确性，因为最终用户并不了解。

数据基础层主要针对歧义问题：例如，如果 *revenue* 解析到一个受治理的数据集而不是四十个合理候选项，问题在很大程度上在 agent 必须搜索之前就消失了。这也是第一道过期防线所在，因为定义规范模型的同一个仓库是确保它们保持最新的天然场所。

我们发现以下几种实践效果特别好：

- **创建规范数据集**：最常见的失败是 agent 无法将概念（"产品 X 的收入"）映射到唯一正确的表、列和指标定义，通常是因为有多个具有微妙不同实现的合理候选项。解决方案是更少、更严格治理的逻辑模型：策划一小组规范的、单一真相源的数据集，明确归属、即用且可发现，然后积极弃用近似重复项。物理汇总和缓存对成本和性能仍然重要，但它们应该从规范模型机械地派生，而不是作为替代方案与其并列。目标是当 agent 搜索一个概念时，它找到一个受治理的答案。

- **强制执行标准**：我们发现基础只有在规范模型和指标定义由 *工具*（agent 在结构上被优先路由到它们；下面更多详情）、由 *CI*（绕过它们的更改无法通过审查）和由 *制度*（下游团队在受治理层上构建或解释为什么不）强制执行时才能维持。没有执行的治理会迅速退化回多候选项问题。

- **同地存放制品**：我们对抗不断变化的数据模型和业务逻辑的主要防线是同地存放。几乎所有数据代码（即建模、语义层、参考文档、规范仪表板定义）都存在于同一个仓库中，CI 检查保护跨层完整性。如果建模更改会破坏下游仪表板或使文档化的指标失效，CI 会标记它，修复在同一个 PR 中发布。（我们将在下面的 **Skills** 部分回到这个机制。）

- **将元数据视为一等公民产品**：编码 agent 之所以表现良好，部分原因是代码库是 *可读的*：README、类型签名、文档字符串等。你的数据仓库也可以同样可读，但前提是列和表描述、规范指标定义、粒度文档、有效值范围、血缘、归属和模型分级以与转换本身相同的严谨度进行维护。虽然这不是新见解，但良好的治理提供了帮助 agent 选择正确数据集的关键上下文。

### 真相源

如果数据基础层是数据仓库本身，那么真相源就是 agent 用来导航它的参考层面。这一层减少了概念与实体的歧义，将利益相关者问题中的"周活跃用户"转化为数据模型中特定的、受治理的实体。按信任度降序排列大致如下：

- **语义层**：编译的指标和维度定义。如果一个问题清晰地映射到一个已定义的指标，agent 调用一个函数并得到一个数字——与公司其他层面产生的数字相同。我们的 agent 被 *结构性要求*（通过 skill 指令）优先利用语义层（见附录）。我们尝试过但 *没有* 效果的一个想法：通过让 LLM 从原始表和查询日志自动生成指标定义来引导语义层。它产生了看似合理的定义，但编码了我们试图消除的歧义，对我们的评估来说是净负面的。因此我们建议用 Claude 生成 *文档*，但由人类拥有 *定义*。

- **血缘和转换图**：当语义层不覆盖某个问题时，血缘和表排名（基于引用数量）让 agent 推理哪些上游模型供给一个概念，哪些已弃用，哪些共享粒度。这把"我不知道这个指标"转化为"我知道从哪个受治理的模型聚合"。它也是我们在下面的 **在线验证** 中提供的新鲜度和来源信号的骨干。

- **查询语料库**：来自仪表板、笔记本和先前分析的历史 SQL。直觉上，这应该是高价值的：它是每个已正确回答问题的记录。*在实践中，我们发现给 agent 对数千个先前查询的原始检索访问权限，准确率提升不到一个百分点*（我们在后面的小节中详述该消融实验）。非结构化检索无法将新问题映射到正确的先例。有效的方法是将该语料库提炼为每个领域的结构化参考文档和 **skill** 中描述的可重用分析模式。将查询历史视为策划的原材料，而不是 agent 直接阅读的真相源。

- **业务上下文**：大多数团队跳过的层，也是我们低估最久的层。不了解你业务的 agent 会回答用户问的问题，而不是用户想问的问题。它不会知道"Q2 发布"指的是一个特定产品，不会知道两个团队对同一个术语有不同的定义，也不会知道一个问题是因为周四有董事会会议而被提出的。我们输入一个公司知识图谱，由索引文档、路线图、决策日志和组织结构组成，以便 agent 可以解析环境引用并提出更好的澄清问题。

这四个方面的共同失败模式与数据基础层相同：**糟糕或过期的文档**。Claude 在弥合差距方面非常有用（起草列描述、从查询模式提出指标文档、在 CI 中标记未文档化的模型），但策划和归属由人类管理。

在接下来的两节中，我们讨论如何使这种归属足够低成本以至于它真正被执行。

### Skills

如果真相源是 agent 的 *声明式* 知识（即指标的含义），那么 skill 就是它的 *过程式* 知识：以什么顺序查阅哪些来源，如何导航有歧义的数据，以及完成的分析应该是什么样子。

在 Claude Code 中，[skill](https://docs.claude.ai/skills) 是 agent 按需读取的 markdown 文件夹。在 Anthropic，我们开发的 skill 极具价值。没有 skill，Claude 准确回答分析问题的能力在我们的评估中不超过 21%。添加 skill 后，这些数字在总体上持续超过 95%，在某些领域中定期达到约 99%。有关我们用来创建大多数 skill 的骨架，请参见附录。

一些最佳实践：

**创建成对的 skill**：一个 **knowledge** skill 充当薄的顶层路由器，允许按需加载额外的领域细节。它说"先尝试语义层，但如果没有覆盖，这里有该领域的约 30 个参考文件，描述相关的表、列、连接和陷阱。"这个路由器实际上是我们对检索失败的回答：不是让 agent 搜索一个百万字段的数据仓库，而是在写入查询之前将空间缩小到几十个策划文件。**unbook** skill 编码了高级分析师会遵循的流程：澄清问题、查找来源（通过 knowledge skill）、运行查询，然后将结果通过对抗性审查子 agent 进行循环。它还捆绑了十几个可重用的分析模式（留存曲线、速率分解、漏斗分析），以便常见请求不需要每次重新发明。

**创建适当的参考文档**：为 LLM 检索而编写。我们的参考文档描述表（粒度、范围和排除项）、陷阱的机制（例如"排除已知的免费邮箱域名，但保留像 anthropic.com 这样的自定义域名"）和显式路由触发器（例如"如果问题是关于实验提升的……请勿用于原始事件计数"），而没有会过时的规定性方案。以下是我们用来创建参考文档的骨架：

```markdown
# [领域] 表

## 快速参考

### 业务上下文 — [用通俗语言描述这个领域的含义]

### 实体粒度 — [一行代表什么]

### 标准卫生过滤 — [该领域每个查询都应用的过滤器]

## 维度
- [关键维度如何编码，以及同一概念在不同表中如何命名]

## 关键表

### [table_name]
- **粒度**: [...] · **范围/排除项**: [...]
- **用法**: [何时使用，何时不使用，连接键，必需过滤器]
[... 每个受治理表一个小节 ...]

## 陷阱
- [高级分析师会警告你的错误答案模式]

## 最佳实践 / 常见查询模式
- [默认选择、标准切分、已验证的模式——其中确切的查询形式是难点]

## 交叉参考
- [拥有相邻问题的邻近领域文档]
```

**将 skill 维护视为一等公民**：Skill 文档描述的是一个每天都在变化的数据模型，因此如果没有主动维护，它们几周内就会出错。我们目睹了离线准确率从发布时的约 95% 在一个月内下降到约 65%，之后我们将其视为工程问题。这意味着将 skill markdown 文件与转换模型放在同一个仓库中，因此更改模型的 PR 也是更新描述它的文档的 PR。代码审查钩子标记任何不触及 skill 文件的报告模型更改。我们约 90% 的数据模型 PR 现在在同一 diff 中包含 skill 更改。我们还随着模型改进和之前的失败模式不再适用而定期修剪 skill 脚手架。

**在所有界面上创建一致且无缝的体验**：同一个 skill *必须* 在 Slack、IDE、仪表板工具和独立 agent 会话中对问题提供相同的答案。我们通过确保一个规范来源（数据仓库）以及 skill 更改自动同步来实现这一点。合并时，skill 同步到插件市场（供 IDE 用户）、云存储 blob（供读取单个文件的托管应用），并通过 MCP 直接作为资源提供。我们还从一开始就设计可移植性，避免硬编码的仓库路径和特定界面的命名空间。

### 验证

最后，验证是发现三种失败模式中哪一种仍在泄漏的方式。

#### 离线评估

我们看到的一个常见模式是数据团队会建立精心设计的分析环境，却没有任何流程来了解其分析 agent 的准确性。

解决这一差距的一种方式是通过离线评估，即简单的问答对。你可以将离线评估类比于 ML 模型的离线测试——它们不能告诉你在线 agent 的性能，但确实能让你了解是否存在关键差距。

我们在 Anthropic 部署两种离线评估。**基于仪表板的评估**由 Claude 自动生成（然后由人工验证），覆盖最常见的利益相关者问题。**长尾评估**是我们向 Claude 提供业务上下文（路线图、表文档）并让它在其余领域生成合理问题。每当利益相关者在对话中纠正 agent 时，我们也持续收集该纠正作为候选评估。

其他最佳实践包括：

- **锚定基准真相使其不会漂移**：针对实时数据编写的评估会在底层数字变化的瞬间过时。将每个评估固定到快照日期，针对稳定的事实表编写，或让评分器评判 agent 的 *查询* 而非其数字。将测试套件接入 CI，以便触及依赖项的 PR 重新运行受影响的评估。

- **像存储遥测数据一样存储结果，而非测试日志**：每次运行都落入数据仓库表，包含 skill 版本、git SHA、模型 ID、每个断言的通过/失败、token 数量和挂钟时间。"那个更改有帮助吗？"变成了一个查询，你可以获得时间序列来捕获单次 CI 运行无法发现的缓慢回归。

- **按领域门控发布**：领域负责人不能向其利益相关者宣布 agent，直到他们的评估集切片通过某个阈值（我们最初使用约 90%）。它迫使参考文档修复在 *用户* 看到失败 *之前* 进行。

- **创建适当数量的评估**：你应该拥有的评估数量取决于业务领域的复杂性和底层数据模型的复杂性。通过跟踪离线准确率预测在线准确率的程度来校准：我们发现每个主题（例如"增长"）超过几十个后收益递减，而且随着每一代新模型的出现，这个上限会下降。

- **离线评估准确率应该约为 100%**；每个正确答案也应该命中你的语义层（如果有的话）。同样，这种准确率水平并不能告诉你你的系统不会产生错误答案，只是没有明显的差距——假设你有适当的评估覆盖。

#### 消融技术

关于 skill 的每个结构性决策（例如，暴露哪些来源、子 agent 是否值得其延迟、是否将两个 skill 合并为一个）都是通过固定我们的离线评估集来进行的。

我们只改变一个组件并比较通过率。每次运行只需一小时，但可以替代大量争论。方法论比任何单一结果更重要：

- **设计空结果实验**：我们最有用的消融实验是一个阴性结果。我们给了 agent 对整个仪表板、转换和分析师笔记本 SQL（数千个文件）的直接 grep 访问权限。然后我们在转录中验证它在每次回答前确实阅读了这些内容。准确率在任一方向上变动不到一个百分点。我们随后检查了明显的混淆因素：对于它答错的问题，答案是否确实在语料库中？大约 80% 的情况下是的。"答案存在"是否预测"现在答对了"？不，翻转率是平坦的。信息就在那里，agent 看到了，但它仍然没有使用它。那个单一实验告诉我们，我们的瓶颈不是对先前工作的 *访问*，而是 *结构*（即将问题映射到正确的实体）。那个洞察重新规划了数月的路线图。

- **在 PR 粒度上进行消融**：每个有意义的 skill 编辑都会在相关评估切片上进行前后运行，PR 描述中包含差异。它让"我改进了文档"保持诚实，并捕捉到令人惊讶的常见情况——一个善意的添加反而让事情变糟。

- **保持一个简短的失败清单**：我们的两个例子：超过某个临界点后叠加额外的文档细化轮次（我们遇到了连续三个净负面迭代：文档变得更长而不是更好），以及将对抗性审查器换成更便宜的模型以减少延迟（它失去了大部分准确率收益，没有真正的加速）。记录阴性结果成本很低，而且可以防止下一个人重新运行相同的实验。

#### 在线验证

最后一步是确保实际的在线系统性能尽可能准确。我们采取的一些步骤包括：

- **对抗性审查**：我们发现使用 Claude skill 激进地质疑潜在最终答案的所有底层假设，在我们的评估集中将准确率提高了 6%，但代价是多消耗 32% 的 token 和 72% 更高的延迟。

- **来源页脚**：每个回答都带有一个页脚，包含其来源层级（语义层 › 策划参考 › 原始表）、底层数据的新鲜度以及模型所有者。它不会使答案更正确，但确实帮助消费者判断他们可以信任该回答的程度。"原始表，新鲜度未知"的页脚是在转发上游之前进行验证的信号，也是我们对静默失败的少数缓解措施之一。

- **数据质量检查**：你的 agent 可能以适当的方式使用了正确的字段，但数据本身是错误的。添加基本的数据质量检查以确保引用的字段是最新的、完整的且没有异常，通常是良好的卫生习惯。

- **被动监控**：我们持续跟踪的两个生产信号是：通过语义层解析的 agent 查询份额，以及使用纠正语言（"那是错误的表"、"你遗漏了欺诈过滤器"）的回答份额。两者都输入到与离线通过率一起每周审查的仪表板中。

- **主动纠正收集**：闭环的部分。一个定时 agent 每隔几小时扫描利益相关者频道寻找类似的纠正语言，起草对相关参考文档的一行修复，并打开一个标记给领域负责人的 PR。修复路径故意很无聊——编辑一个 markdown 文件，合并，自动同步到所有地方——这样领域负责人不会在这个任务上花太多时间。相同的纠正反馈到离线评估集中。

这些都没有完全捕获的失败模式是 **静默** 的那种。答案是错误的，但看起来合理并被无异议地使用。我们的缓解措施是来源页脚、对任何面向领导层的内容的明确人工签字，以及每个领域顶级 KPI 的常设评估，每天对照权威仪表板进行健全性检查，尽管我们还没有一个稳健的解决方案。

## 入门指南

如果你从零开始，少量规范数据集、几十个离线评估和一个薄薄的 knowledge skill 将捕获大部分收益；这篇文章中的其他所有内容都是在我们构建了这些之后添加的。

我们也分享了许多最佳实践，并非所有都适合每个数据团队。通过以下问题与你的组织对齐几个影响你方法的原则：

- **今天正确的答案有多重要，对比未来？** AI 模型正在快速进步。我们经常看到公司构建大量基础设施来应对当前模型的不足，而这些不足在模型改进后就变得无关紧要。了解模型在哪里不足，并等待模型改进来填补差距，开销显著更少，但可能不适合你公司的风险承受能力。

- **你预计业务的复杂性随时间如何变化？** 如果你产生不了多少数据，只有少数输出消费者，或者你的数据模型可能保持简单，我们讨论的一些流程可能是过度的。

- **输出的目标受众技术程度如何？** 换句话说，如果你是为能够识别错误答案的数据科学家构建这个分析系统，你可能对错误的容忍度更高，相比之下受众对底层数据模型不熟悉的情况。

- **你愿意为提高准确率花费多少？** 我们发现某些流程如对抗性验证可以显著提高准确率，但通常以更高的成本和延迟为代价。

- **你对访问控制和内部数据隐私的舒适度如何？** Agent 拥有的上下文越多，通常表现越好；然而，广泛的数据访问与大多数公司的治理姿态相悖。这决定了你是构建一个 agent 还是多个范围限定的 agent。

无论你选择哪条路线，我们最大的收益都来自解决三种失败模式中的每一种：将歧义折叠为单一受治理的答案，使答案易于发现，并在任一出问题时标记。
