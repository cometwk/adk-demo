# 第21章 AI 中间件模式：语义层

本章深入探讨企业数据架构中最棘手的领域——**ChatBI（对话式数据分析）**。

在 2023 年，我们天真地以为 Text-to-SQL 可以解决一切。但在 2025 年，架构师们痛定思痛，发现让 LLM 直接写 SQL 去查询原始数据库是一个**"高风险、低良率"**的死胡同。

**本章将引入"语义层"**这一关键中间件，重构 AI 与数据库的对话方式，实现从"猜表名"到"查指标"的范式跃迁。


*图21.0 Agent作为通用数据翻译器*

在企业级 RAG 系统中，除了查文档（非结构化数据），用户最强烈的需求是查数（结构化数据）。

- "上个季度华东区的毛利是多少？"
- "对比一下 A 产品和 B 产品的库存周转率。"

最直观的做法是 **Text-to-SQL**：把数据库 Schema 喂给 LLM，让它生成 SQL。
但在 2025 年的实战中，这种架构已被证明是**不可扩展**的。

**为什么直接 Text-to-SQL 会失败？**

- **幻觉 (Hallucination)**：数据库可能有 5000 张表，列名缩写晦涩难懂（如 t_x_123）。LLM 经常猜错字段名。
- **口径不一 (Business Logic Gap)**："毛利"怎么算？是 收入 - 成本，还是 收入 - 成本 - 税费？这个逻辑通常不在数据库表结构里，而在 BI 系统的报表公式里，或者老员工的脑子里。LLM 不知道这些隐性知识，算出来的数永远对不上。
- **安全失控 (Security Risk)**：一旦开放了 SQL 生成权限，很难防止 LLM 写出全表扫描甚至 DROP TABLE（尽管有只读权限，但性能杀手依然存在）。

2025 年的架构共识是：**禁止 LLM 直接触碰原始数据库 (Raw Tables)。**

我们需要在中间加一层翻译器——**语义层 (Semantic Layer)**。

## 21.1 什么是语义层？数据的"菜单"

如果把原始数据库比作**"后厨的生鲜仓库"**（这也有一堆面粉，那也有一堆肉，乱七八糟），那么语义层就是**"餐厅的菜单"**。

- **仓库（DB）**：table_orders, table_order_items, table_users.
- **菜单（Semantic Layer）**：Total Revenue (总收入), Active Users (活跃用户), Monthly Churn (月流失率).

**语义层的作用**：它预先定义好了复杂的 JOIN 关系和计算公式。
Agent 不需要知道"总收入"涉及哪三张表、怎么关联。它只需要"点菜"——告诉语义层："我要 **Total Revenue**，按 **Month** 分组。"

## 21.2 架构模式：Headless BI + Agent

在 2025 年，**Headless BI**（无头 BI）工具（如 **Cube.js**, **dbt Semantic Layer**, **Looker Modeler**）成为了 AI 的最佳拍档。

### 21.2.1 架构流程

**1. User**: "上个月华东区的销售额是多少？"

**2. Agent**:
- 不生成 SQL。
- 生成 **语义查询 API**（通常是 JSON 格式）。
- `{"metric": "sales_amount", "filter": {"region": "East China"}, "time": "last_month"}`

**3. Semantic Layer (Middleware)**:
- 接收 JSON 请求。
- 查找"销售额"的定义（公式：sum(order_total)）。
- 自动生成对应数据库方言的复杂 SQL（Postgres / Snowflake / ClickHouse）。
- 执行查询，返回结果。

**4. Agent**: 解释结果。

## 21.3 核心实现：Context Injection (上下文注入)

虽然有了语义层，但 LLM 怎么知道有哪些指标可用？
我们不能把几千个指标全塞进 Prompt。我们需要 **RAG for Metadata**。

### 21.3.1 元数据 RAG 流程

- **Indexing**: 将语义层定义的所有指标（Metrics）和维度（Dimensions）及其描述（Description），存入向量数据库。
  - Metric: Revenue -> Desc: "销售总收入，不含税，已扣除退款。"
- **Retrieval**: 用户提问时，先检索 Top-10 最相关的指标定义。
- **Generation**: 将这 10 个指标定义注入 System Prompt。

### 21.3.2 实战代码 (伪代码)

```python
# 假设使用 Cube.js 作为语义层

def query_data_agent(user_query):
    # 1. RAG 检索元数据 (Metadata Retrieval)
    relevant_metrics = vector_db.search(user_query, type="metric", k=5)
    relevant_dimensions = vector_db.search(user_query, type="dimension", k=5)

    # 2. 构建 Prompt
    system_prompt = f"""
    你是一个数据分析师。请基于以下可用的指标和维度，生成 Cube.js 的查询 JSON。

    可用指标: {relevant_metrics}
    可用维度: {relevant_dimensions}

    不要编造指标。如果用户问的指标不存在，请回复无法回答。
    """

    # 3. LLM 生成 API Payload (不是 SQL)
    query_json = llm.generate(system_prompt, user_query)

    # 4. 调用语义层 API
    result = cube_api.load(query_json)

    return result
```

## 21.4 语义层的三大红利

引入这一层中间件，不仅仅是为了防幻觉，它带来了额外的架构红利。

### 21.4.1 统一口径 (Single Source of Truth)

以前，财务部用 Excel 算毛利，销售部用 SQL 算毛利，结果永远打架。
现在，所有的 Agent（无论是销售助手还是 CEO 助理）都调用同一个语义层的 gross_margin 指标。**口径在代码层面被统一了。**

### 21.4.2 细粒度权限控制 (Row-Level Security)

LLM 很难理解复杂的权限（"张三只能看华东区，但李四能看全国"）。
如果在 SQL 里拼 WHERE 条件，很容易漏掉。
语义层通常内置了 **Security Context**。Agent 调用 API 时带上 user_id，语义层在生成 SQL 时会自动注入 WHERE region = 'East'，物理隔离数据风险。

### 21.4.3 性能优化与缓存

语义层（如 Cube.js）通常自带 **Pre-aggregation (预聚合)** 和 **Caching** 机制。
Agent 查询"去年总营收"，语义层直接返回缓存结果（10ms），而不需要去扫描 10 亿行数据库。这解决了第 7 章提到的成本和延迟问题。

## 21.5 边界与挑战：语义层不是银弹

虽然语义层很美好，但在落地时有巨大阻力。

- **建设成本高 (High Setup Cost)**：你需要专门的数据工程师（Analytics Engineer）去写那些 YAML 或 Python 定义文件，把数仓里的几千张表梳理成指标体系。这是一项巨大的**数据治理**工程。
- **灵活性受限 (Flexibility Trade-off)**：语义层只能回答"定义好"的问题。如果用户问一个从未定义的、极其刁钻的临时查询（Ad-hoc Query），语义层可能不支持，这时又得退回到 Raw SQL 模式。

**架构建议**：采用 **80/20 原则**。

- **80% 的高频核心指标**（营收、DAU、库存）：通过语义层由 Agent 访问，保证极高的准确率和速度。
- **20% 的长尾探索性分析**：允许高级数据分析师使用 **Code Interpreter** (Python Agent) 直接操作数据子集，但必须在沙箱中进行。

## 本章小结

本章我们为 AI 加上了一个"翻译官"。

- **Text-to-SQL** 在企业级场景下是危险的。
- **语义层** 将数据抽象为"指标"和"维度"，为 AI 提供了稳定、安全、统一的 API。
- **Headless BI** 是连接 Agent 与 Data Warehouse 的标准化桥梁。

通过这一层，我们终于可以放心地让 CEO 问 Agent："我们公司现在的现金流状况如何？"，而不用担心它幻觉出一个错误的数字。

至此，我们的 AI 系统已经能看文档（RAG）、能查数（Semantic Layer）、能操作旧系统（Strangler Pattern）。
但是，并不是所有的计算都需要在云端的大模型上完成。
在下一章，我们将探讨 **边缘智能：SLM (Small Language Models)**，看看如何把 AI 塞进用户的手机里，实现极致的隐私与低延迟。

## 思考与面试角

**Q1：为什么说 SQL 对于 LLM 来说不是一种好的"语言"？**

> **参考答案**：SQL 是**声明式**的，但它与底层的 Schema 强耦合。Schema 中的表名、字段名往往是缩写，缺乏语义（如 col_1, ts）。且 SQL 的 Join 逻辑极其复杂，稍微错一个 ON 条件，数据就会发散或丢失。LLM 擅长自然语言与逻辑推演，不擅长记忆成千上万个无规律的物理字段名。

**Q2：如果公司没有预算上 Cube/dbt 这种重型语义层，有什么轻量级替代方案？**

> **参考答案**：可以使用 **View (数据库视图)** 作为轻量级语义层。数据团队预先写好复杂的 SQL，封装成宽表视图 view_sales_summary，把字段名重命名为中文或清晰的英文（total_revenue）。然后只把这些**视图**的 Schema 喂给 LLM。这相当于手动构建了一个简易的语义层。

**Q3：Text-to-SQL 在 2025 年还有生存空间吗？**

> 参考答案：有，但局限于辅助生成 (Copilot) 场景。即：Agent 生成 SQL，但不自动执行，而是展示给数据分析师看，由人确认无误后点击运行。在全自动 (Autopilot) 场景下，Text-to-SQL 正在被 Semantic Layer 取代。
