# 为 Agent 编写有效的工具——借助 Agent 自身

**作者：** Ken Aizawa，及来自 Research、MCP、Product Engineering、Marketing、Design 和 Applied AI 等团队的同事贡献

**发布日期：** 2025 年 9 月 11 日

---

Agent 的效果取决于我们赋予它们的工具。本文分享了提升 Agent AI 系统性能的技术，涵盖如何构建和测试工具原型、创建全面的评估体系，以及与 Agent 协作以自动提升工具性能。

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro) 可以为 LLM Agent 提供数以百计的工具来解决现实任务。本文介绍了在各种 Agent AI 系统中提升性能的有效技术。

## 什么是工具？

确定性系统在给定相同输入时总是产生相同的输出，而非确定性系统——如 Agent——即使在相同的起始条件下也可能生成不同的响应。

传统软件在确定性系统之间建立契约（例如 `getWeather("NYC")` 总是以相同方式工作）。工具代表了一种新型软件，反映了确定性系统与非确定性 Agent 之间的契约。当用户问"今天需要带伞吗？"时，Agent 可能会调用天气工具、根据通用知识回答，或先提出澄清问题。偶尔，Agent 可能会产生幻觉或无法理解如何使用工具。

这意味着我们需要从根本上重新思考方法：我们不能像为其他开发者编写函数和 API 那样编写工具和 MCP 服务器，而是需要为 Agent 进行设计。目标是"增加 Agent 能够有效发挥作用的表面积"，通过支持多样化的成功策略来实现。对 Agent 最"符合人体工程学"的工具，最终也出人意料地适合人类使用。

## 如何编写工具

这个过程涉及与 Agent 协作来编写和改进工具：先快速建立原型，运行全面评估，然后重复评估与改进的循环，直到 Agent 达到良好的性能。

### 构建原型

不经过实际测试，很难预判哪些工具对 Agent 来说是符合人体工程学的。先搭建一个快速原型。如果使用 [Claude Code](https://www.anthropic.com/claude-code) 编写工具，请提供任何软件库、API 或 SDK（包括 [MCP SDK](https://modelcontextprotocol.io/docs/sdk)）的文档。对 LLM 友好的文档通常可以在官方文档站点的 `llms.txt` 文件中找到。

将工具封装在[本地 MCP 服务器](https://modelcontextprotocol.io/docs/develop/connect-local-servers)或 [Desktop Extension](https://www.anthropic.com/engineering/desktop-extensions)（DXT）中，可以在 Claude Code 或 Claude Desktop 应用中连接和测试。

- 在 Claude Code 中连接本地 MCP 服务器：`claude mcp add <name> <command> [args...]`
- 在 Claude Desktop 中连接：导航到 `Settings > Developer` 或 `Settings > Extensions`

工具也可以直接传入 [Anthropic API](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/overview) 调用进行编程测试。

亲自测试工具以发现粗糙之处。收集用户反馈，建立对预期用例和提示的直觉。

### 运行评估

衡量 Claude 使用工具的效果需要运行评估。生成大量基于真实世界用例的评估任务。与 Agent 协作分析结果并确定改进方向。完整流程参见[工具评估 cookbook](https://platform.claude.com/cookbook/tool-evaluation-tool-evaluation)。

**生成评估任务**

Claude Code 可以快速探索工具并创建数十个提示-响应对。提示应来源于真实世界用例，并基于真实的数据源和服务。避免使用过于简单的"沙盒"环境，这类环境无法以足够的复杂度对工具进行压力测试。强有力的评估任务可能需要多次工具调用——潜在地需要数十次。

**强任务示例：**

- 安排下周与 Jane 的会议，讨论最新的 Acme Corp 项目，附上上次规划会议的笔记，并预约一个会议室
- 客户 ID 9182 报告对一次购买被收取了三次费用——查找所有相关日志条目并确定是否有其他客户受到影响
- 客户 Sarah Chen 提交了取消请求——通过确定她为什么离开、什么优惠最有吸引力以及任何风险因素来准备一个挽留方案

**弱任务示例：**

- 安排下周与 jane@acme.corp 的会议
- 在支付日志中搜索 `purchase_complete` 和 `customer_id=9182`
- 按客户 ID 45892 查找取消请求

每个评估提示都应配对一个可验证的响应或结果。验证器可以从精确字符串比较到委托 Claude 来评判响应。避免使用过于严格的验证器，因为格式差异或有效的替代表述可能会拒绝正确响应。

可选择指定你期望 Agent 调用哪些工具，但避免过度指定，因为可能存在多条有效路径。

**运行评估**

通过直接的 LLM API 调用以编程方式运行评估。使用简单的 Agent 循环（`while` 循环包装交替的 LLM API 和工具调用）：每个评估任务一个循环。每个评估 Agent 应接收一个任务提示和你的工具。

在评估 Agent 的系统提示中，指示 Agent 输出推理和反馈块——而不仅仅是结构化的响应块。在工具调用和响应块*之前*输出这些内容，可能通过触发链式思考行为来提高 LLM 的有效智能。

使用 Claude 时，你可以开箱即用地启用 [interleaved thinking](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#interleaved-thinking) 来获得类似功能。这有助于探查 Agent 为什么调用或不调用某些工具，并突出需要改进的领域。

除了顶层准确率之外，还应收集指标，如工具调用的总运行时间、任务的总运行时间、工具调用总数、总 token 消耗和工具错误数。跟踪工具调用可以揭示常见工作流和整合机会。

**分析结果**

Agent 在发现问题方面是有用的伙伴——从矛盾的工具描述到低效的实现和令人困惑的 schema。然而，"Agent 在反馈和响应中遗漏的内容往往比包含的内容更重要"。LLM 并不总是[说出它们的真实想法](https://www.anthropic.com/research/tracing-thoughts-language-model)。

观察 Agent 在哪里卡住或困惑。仔细阅读评估 Agent 的推理和反馈（或 CoT）以识别粗糙之处。审查包括工具调用和响应在内的原始记录，以捕捉 Agent 的 CoT 中未明确描述的行为。读出言外之意——评估 Agent 不一定知道正确的答案和策略。

分析工具调用指标。冗余的工具调用可能暗示需要调整分页或 token 限制参数；无效参数导致的工具错误可能暗示需要更清晰的描述或更好的示例。在推出 Claude 的[网络搜索工具](https://www.anthropic.com/news/web-search)时，他们发现 Claude 不必要地在 `query` 参数中追加 `2025`，导致搜索结果偏向性并降低性能——通过改进工具描述修复了这个问题。

### 与 Agent 协作

Agent 可以分析结果并为你改进工具。将评估记录拼接起来并粘贴到 Claude Code 中。Claude 擅长分析记录并一次性重构多个工具——例如，在做出更改时确保实现和描述保持自洽。

本文中的大部分建议来自使用 Claude Code 反复优化内部工具实现。评估是在内部工作区之上创建的，反映了内部工作流的复杂性，包括真实的项目、文档和消息。

保留的测试集确保不会对"训练"评估过拟合。这些测试集揭示了超出"专家"工具实现所达到的额外性能提升——无论是由研究人员手动编写的还是由 Claude 自动生成的。

## 编写有效工具的原则

### 为 Agent 选择正确的工具

更多工具并不总是带来更好的结果。一个常见错误是工具仅仅封装现有软件功能或 API 端点——无论它们是否适合 Agent。Agent 与传统软件有着不同的"可供性"——不同的感知潜在行动的方式。

LLM Agent 的上下文有限，而计算机内存便宜且充裕。考虑在通讯录中搜索联系人：传统软件可以高效地逐一处理列表，但使用返回所有联系人的工具的 LLM Agent 会将有限的上下文浪费在无关信息上（就像通过暴力方式从头到尾阅读每一页）。更好的方法是先跳到相关页面。

构建少量针对特定高影响力工作流的有针对性的工具，与评估任务匹配，然后逐步扩展。对于通讯录的情况，实现 `search_contacts` 或 `message_contact` 而不是 `list_contacts`。

工具可以整合功能，在底层处理多个离散操作（或 API 调用）。工具可以用相关元数据丰富响应，或在单次调用中处理频繁链式执行的多步任务。

**整合示例：**

- 不使用 `list_users`、`list_events` 和 `create_event`，而是考虑 `schedule_event`，它会查找空闲时间并安排事件
- 不使用 `read_logs`，而是考虑 `search_logs`，它只返回相关的日志行及周围上下文
- 不使用 `get_customer_by_id`、`list_transactions` 和 `list_notes`，而是实现 `get_customer_context`，一次性编译所有相关客户信息

每个工具应有清晰、独特的目的。工具应使 Agent 能够像人类一样细分和解决任务，同时减少中间输出消耗的上下文。过多或重叠的工具可能使 Agent 偏离高效策略。

### 工具命名空间

AI Agent 可能获得对数十个 MCP 服务器和数百个工具的访问权限——包括其他开发者创建的工具。当工具在功能上重叠或目的模糊时，Agent 会困惑于使用哪一个。

命名空间（在公共前缀下对相关工具进行分组）有助于划定边界。MCP 客户端有时默认这样做。例如，按服务命名空间（`asana_search`、`jira_search`）和按资源命名空间（`asana_projects_search`、`asana_users_search`）有助于 Agent 在正确的时间选择正确的工具。

在前缀式和后缀式命名空间之间做出选择对工具使用评估有非平凡的影响。影响因 LLM 而异，因此请根据你自己的评估选择命名方案。

Agent 可能调用错误的工具、用错误的参数调用正确的工具、调用过少的工具，或错误地处理响应。通过选择性地实现名称反映自然任务细分的工具，你同时减少了加载到上下文中的工具和描述数量，并将 Agent 计算从 Agent 的上下文卸载回工具调用本身，从而降低整体错误风险。

### 从工具返回有意义的上下文

工具实现应仅向 Agent 返回高信号信息。优先考虑上下文相关性而非灵活性，摒弃低级技术标识符（如 `uuid`、`256px_image_url`、`mime_type`）。像 `name`、`image_url` 和 `file_type` 这样的字段更有可能直接指导 Agent 的后续行动和响应。

Agent 处理自然语言名称、术语或标识符的成功率显著高于处理隐晦标识符。仅仅将任意的字母数字 UUID 解析为更有语义的语言（甚至是零索引的 ID 方案），就能通过减少幻觉显著提高 Claude 在检索任务中的精度。

有时 Agent 需要灵活地与自然语言和技术标识符输出进行交互，特别是触发下游工具调用时（例如 `search_user(name='jane')` → `send_message(id=12345)`）。通过暴露 `response_format` 枚举参数来启用两者，允许 Agent 控制工具返回 `"concise"` 还是 `"detailed"` 响应。

你可以添加更多格式以获得更大的灵活性，类似于 GraphQL 中你选择要接收的信息。示例 ResponseFormat 枚举：

```go
enum ResponseFormat {
   DETAILED = "detailed",
   CONCISE = "concise"
}
```

一个详细的工具响应可能使用约 206 个 token，而相同数据的简洁响应可能使用约 72 个 token（约 1/3 的 token）。Slack 线程使用获取线程回复所需的唯一 `thread_ts` 标识符；这些和其他 ID（`channel_id`、`user_id`）可以从"详细"响应中获取以启用进一步的工具调用，而"简洁"响应仅返回线程内容并排除 ID。

甚至工具响应结构——XML、JSON 或 Markdown——也会影响评估性能。没有一刀切的解决方案，因为 LLM 基于下一个 token 预测进行训练，倾向于与其训练数据匹配的格式表现更好。最佳响应结构因任务和 Agent 而异。

### 优化工具响应的 token 效率

优化上下文的质量很重要，优化工具响应中返回的上下文*数量*同样重要。

对于可能消耗大量上下文的工具响应，实现分页、范围选择、过滤和/或截断的某种组合，并设置合理的默认参数值。对于 Claude Code，工具响应默认限制为 25,000 个 token。虽然 Agent 的有效上下文长度会随时间增长，但对上下文高效工具的需求将持续存在。

截断响应时，用有用的指导引导 Agent。直接鼓励 Agent 追求更节省 token 的策略，比如进行多次小范围的定向搜索而不是单次广泛搜索来检索知识。类似地，如果工具调用引发错误（例如输入验证期间），通过提示工程使错误响应清晰地传达具体和可操作的改进建议，而不是不透明的错误代码或堆栈跟踪。

工具截断和错误响应可以引导 Agent 朝着更节省 token 的工具使用行为（使用过滤器或分页）或给出正确格式化工具输入的示例。

### 通过提示工程优化工具描述

改进工具最有效的方法之一：对工具描述和规范进行提示工程。由于这些内容会被加载到 Agent 的上下文中，它们共同引导 Agent 采取有效的工具调用行为。

编写工具描述和规范时，想想你会如何向新员工描述你的工具。考虑你可能隐含带入的上下文——专门的查询格式、小众术语的定义、底层资源之间的关系——并将其明确化。通过清晰描述（并用严格的数据模型强制执行）预期的输入和输出来避免歧义。输入参数应明确命名：用 `user_id` 代替 `user`。

通过评估，你可以更有信心地衡量提示工程的影响。即使对工具描述的小幅精炼也能带来显著的改进。"Claude Sonnet 3.5 在我们对工具描述做出精确优化后，在 SWE-bench Verified 评估中达到了最先进水平的性能"，大幅降低了错误率并提高了任务完成率。

工具定义的其他最佳实践可在[开发者指南](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#best-practices-for-tool-definitions)中找到。对于 Claude，特别推荐了解工具如何动态加载到 Claude 的[系统提示](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#tool-use-system-prompt)中。对于 MCP 服务器，[工具注解](https://modelcontextprotocol.io/specification/2025-06-18/server/tools)有助于披露哪些工具需要开放世界访问或进行破坏性更改。

## 展望未来

为 Agent 构建有效工具需要将软件开发实践从可预测的确定性模式重新导向非确定性模式。

通过上述迭代的、评估驱动的过程，成功的工具中涌现出一致的模式：有效的工具被有意且清晰地定义，审慎使用 Agent 上下文，可在多样化的工作流中组合，并使 Agent 能够直观地解决现实任务。

未来，Agent 与世界交互的具体机制将不断演进——从 MCP 协议的更新到底层 LLM 本身的升级。通过系统化、评估驱动的方法来改进 Agent 的工具，我们可以确保随着 Agent 变得更加强大，它们使用的工具也将随之演进。
