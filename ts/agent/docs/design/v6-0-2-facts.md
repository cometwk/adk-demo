# facts 收集

## 要点

在 V6 的设计中，对 **FactBinding** 和 **FactStore** 的重新设计是系统从“工具型 Agent”演进为“决策助手系统”的核心。
其核心目标是解决 V5 中由于使用**扁平键值对（KV）**导致的事实无归属、不可追溯以及无法自动化评估的问题。

以下是 V6 重新设计 FactBinding 和 FactStore 的核心要点：

### 1. 从扁平变量到结构化绑定（Structure of FactBinding）
V5 的事实存储是一个简单的字典 `Record<string, any>`（例如：`{workload_alice: 85}`），这在处理大规模实体时会造成命名空间冲突。V6 将每一条事实重构为包含丰富元数据的 **FactBinding** 对象：
*   **`entityId` 与 `property`**：建立了明确的**实体命名空间**，将事实与具体的图节点绑定。
*   **`source`**：记录事实的来源（如来自哪个工具调用或聚合函数），防止模型凭空编造数字。
*   **`confidence`**：不再由 LLM 自行上报，而是由系统根据来源类型计算基础置信度，模型仅能做有限的微调。
*   **`validUntil`**：引入有效时间，使系统能识别陈旧事实并触发重新拉取或降权处理。

### 2. 事实有归属 (Fact-with-binding) 的设计哲学
V6 强调“事实有归属”，这一原则带来了以下关键收益：
*   **自动化的规则评估**：系统可以根据规则声明的 `appliesTo`（适用实体）和 `requiredFacts`（所需事实），自动从 FactStore 中拉取对应实体的属性进行评估。这消除了 V5 中模型必须手动拼凑变量名（如 `workload_alice`）来喂给规则的负担。
*   **消除误报**：在 V5 的 Demo 中，常因为模型拼错变量名而导致 `missingFacts` 误报，V6 的结构化绑定彻底修复了这一结构性缺陷。

### 3. 工具链的协同演进
为了支撑新的事实库架构，V6 调整并新增了一系列工具：
*   **`bind_fact`**：显式让 LLM 将发现的事实写入 FactStore，而不是散落在对话背景中。
*   **`lookup_fact`**：允许从 FactStore 中查询已绑定的事实及其来源。
*   **`call_method` 的增强**：增加了 **precondition assertion（前置断言）**。系统会检查方法参数是否在 FactStore 中有合法来源，阻止模型用“语义荒谬”（如全零）的捏造数据进行盲调。

### 4. 支撑“反事实”与“重放” (Trace as a Artifact)
FactStore 是 V6 实现决策可重放性的基础。
*   **反事实推理 (Counterfactuals)**：由于事实是结构化绑定的，系统允许在已绑定的事实库上做局部替换（如“如果 Alice 的负载降到 60”），并重跑**确定性规则流（Critic）**，从而快速给出不同假设下的结论变化。
*   **决策重放**：FactStore 记录了决策时刻的数据快照，支持后续的审计、校准和学习。

### 总结：V5 vs V6 的对比
| 维度 | V5 的事实处理 | V6 的 FactBinding / FactStore |
| :--- | :--- | :--- |
| **数据结构** | 扁平 KV (`Record<string, any>`) | 结构化对象 `(entityId, property, value, ...)` |
| **归属关系** | 无，由模型在 Prompt 中硬编码变量名 | **事实有归属**，系统自动关联实体属性 |
| **来源可追溯性** | 弱，无法验证数据真伪 | **强**，每条事实必须标记 `source` |
| **规则评估方式** | LLM 手动匹配事实与规则 | 系统按命名空间自动绑定并评估 |
| **置信度来源** | LLM 随口报告（如 0.95） | 系统根据来源计算基础值 |

这种重新设计确保了事实在复杂的推理长链中保持**可定位、可追溯、可反事实**，是 V6 从处理小规模 Demo 走向生产级决策系统的底层保障。

## 分析

你感到“不对头”是非常准确的直觉。这段代码展示的 `seedLibraryFactStore` 在 V6/V7 的设计语境下确实存在严重的**逻辑错位**。

这种“不合理”主要体现在它违反了系统设计的核心原则，将 **“图数据库（底层数据）”** 与 **“事实存储（推理状态）”** 这两个概念混淆了。以下是详细的深度分析：

### 1. 违反了“渐进式实体披露”原则 (E)
在 V5 和 V6 的设计中，核心原则是 **Progressive Entity Disclosure**。
*   **错误点**：这个 `seed` 函数一次性把“小明”、“图书馆设置”、“三本书”的所有细节都塞进了 `FactStore`。
*   **后果**：这让 Agent 失去了“探索”的意义。如果所有事实都已经预存在 `FactStore` 中，系统就退化回了 V4 之前的状态——模型不需要调用 `inspect_node` 或 `query_neighbors` 去发现证据，因为它“出生”就已知晓一切。
*   **正确做法**：初始化时，`FactStore` 应该几乎是**空的**，或者只包含极少数的“入口实体”（Entry Entities）。其余的事实应该由 Agent 在推理过程中通过 `bind_fact` 动态写入。

### 2. 混淆了“数据快照”与“工作记忆”
`FactStore` 的本质是 **Agent 的“工作笔记”**，而不是数据库的副本。
*   **错误点**：代码中手动指定 `source: { kind: 'graph_property' }` 是一种“伪造”。
*   **后果**：在真实的 V6 链路中，`source` 必须由系统自动填充，以证明这条事实确实是通过某个工具调用（如 `inspect_node`）从图中获取的。手动硬编码 `source` 会导致**证据链无法审计**。
*   **设计冲突**：如果这些数据本来就在图里，应该通过工具查出来；如果这些数据不在图里，那它们就不应该标记为 `graph_property`。

### 3. 开发体验（DX）的崩塌
正如你观察到的，手动编写这些结构化绑定非常痛苦且冗余。
*   **冗余性**：每一个属性（name, title, daysOnShelf）都要写一遍 `confidence`, `validFrom`, `observedAt`。
*   **维护困难**：这种硬编码方式使得本体（Ontology）发生变化时，所有的 seed 代码都要重写。
*   **V6 的初衷**：这种繁琐的结构化数据是给**系统层（Critic/MCDA）**消费的，而不是让人类手写的。人类开发者应该定义的是“图数据”或“规则函数”，而 `FactBinding` 对象应该是工具执行的**副产品（Side-effect）**。

### 4. 绕过了“盲调”防护机制
V6 引入 `FactStore` 的一个关键目的是防止模型在没有证据的情况下“盲调”方法（例如参数全为 0）。
*   **逻辑漏洞**：如果你在初始化时就喂饱了 `FactStore`，那么 `call_method` 的前置断言（Precondition Assertion）就失去了意义。
*   **后果**：系统无法验证 Agent 是否具备“通过调查获取事实”的能力，因为事实是开发者“喂”到嘴里的，这会导致系统在面对真实生产数据（数据可能缺失或变动）时表现脆弱。

### 总结：哪里不对头？
这个 `seed` 函数把 **“给 Agent 考试的考卷（事实库）”** 直接变成了 **“开卷考试的答案”**。

**合理的架构应该是：**
1.  **Graph**：存储所有的原始数据（小明借了什么书、书的标题等）。
2.  **Ontology**：定义规则（借书上限、逾期准则）。
3.  **FactStore**：初始化时为**空**。
4.  **推理时**：Agent 搜索到“小明” -> 调用 `inspect_node` -> 系统**自动**产生 `FactBinding` 并存入 `FactStore` -> 此时 `FactStore` 才有了内容，并支持后续的规则评估。

**你的感觉是对的：** 如果需要这样大规模手写 `seed`，说明当前的实现并没有真正利用“工具驱动发现”的逻辑，而是把 `FactStore` 当成了一个带格式的内存数据库在用。
