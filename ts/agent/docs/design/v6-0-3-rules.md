在 V5 至 V7 的设计演进中，**Rules（规则，在本体中称为“C”）** 的工作机制经历了从“文本提示”到“结构化对象”，再到“自动化评估系统”的转变。

以下是规则系统如何工作的详细机制：

### 1. 规则的结构化定义
在 V5 之后，规则不再是隐含在代码里的逻辑，而是**一等公民（First-class Object）**。一条典型的规则包含以下核心字段：
*   **基本属性**：包括 `id`、`description` 和 `kind`（硬约束 `hard_constraint` 或软准则 `soft_criterion`）。
*   **适用范围 (`appliesTo`)**：定义规则在哪些类型的实体（如 `Engineer` 或 `Project`）上生效。
*   **所需事实 (`requiredFacts`)**：声明评估此规则必须从事实库中获取哪些属性（如 `workload`）。
*   **V6 增强字段**：引入了 `weight`（权重）、`severity`（严重程度）、`direction`（打分方向）以及 `subsumedBy`（蕴含关系，防止重复计数）。

### 2. 规则拓扑与执行顺序 (Rule DAG)
为了保证逻辑严密，V6 引入了 **Rule DAG（有向无环图）** 对规则进行拓扑排序。执行过程遵循以下顺序：
1.  **推理规则 (Inference Rules)**：首先运行，用于根据基础事实产生推导事实（`derived facts`）。
2.  **硬约束 (Hard Constraints)**：检查是否命中“一票否决”项（Veto）。如果命中，候选方案可能直接出局。
3.  **软准则 (Soft Criteria)**：最后进行加权打分，用于区分方案的优劣。

### 3. 事实绑定与自动评估 (Fact Binding)
这是 V6/V7 的核心改进，解决了模型“盲调”和变量名混乱的问题：
*   **自动取值**：系统根据规则定义的 `appliesTo` 和 `requiredFacts`，自动从 **FactStore** 中拉取对应实体的属性。
*   **显式绑定**：不再让 LLM 手动拼凑扁平的键值对，而是通过 `evaluate_rule` 工具在已绑定的事实空间内进行精确评估。
*   **来源追溯**：规则评估所依赖的每一条事实都带有 `source`（如 `graph_property` 或 `aggregation`），确保结果可审计。

### 4. 判决双轨制 (Dual-track Verdict)
规则的最终产出不再仅由 LLM 说了算，而是采用**双轨制**：
*   **系统判决 (System Verdict)**：由确定性规则系统（Critic）运行 MCDA（多准则决策分析）算法得出。它不依赖 LLM，具有可重放和高度一致性的特点。
*   **模型判决 (Model Verdict)**：由 LLM 综合语义信息给出判断。
*   **冲突处理**：如果两份判决不一致，系统会通过 **Reconciler（协调器）** 显式暴露冲突，由用户做最终裁决，而不是让模型静默覆盖系统结论。

### 5. 校准与解释
*   **得分计算**：V6 舍弃了 V5 简单的“触发数/总数”逻辑，改为基于权重、严重程度和方向的**校准评分**。
*   **证据链条**：规则触发后会引用具体的证据 ID（Evidence ID），使得最终输出的推荐建议包含完整的“规则+证据”支撑。
*   **反事实测试**：支持在不改变图数据的情况下，通过局部替换 FactStore 中的事实（如“如果负载降到 60”）来重跑规则，观察结论的变化。

**总结来说：** 规则系统的工作方式是从 **FactStore** 获取受控事实，按照 **Rule DAG** 的拓扑顺序进行自动化评估，并通过**双轨判决**确保最终决策的科学性与可信度。
