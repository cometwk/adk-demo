rule 简化设计

# 1

V6 中的 **Rule（规则）设计确实非常复杂**，这种复杂性是系统从“演示 Demo”向“工业级决策支持系统”转化的必然结果。

### 1. 为什么说它是一个“好”设计？
尽管复杂，但在 V6 的语境下，它是为了解决 V5 架构中的三个致命缺陷而存在的：

*   **防止模型“篡改”结论**：在 V5 中，评分逻辑过于幼稚（触发数/总数），模型在发现评分不合理时会直接在最终答案中“强行修正”，导致系统判决形同虚设。V6 通过 **MCDA（多准则决策分析）** 引入权重、严重程度和方向，让系统评分符合业务直觉。
*   **解决“变量名大爆炸”**：V5 使用扁平 KV 存储事实（如 `workload_alice`），当实体成千上万时，模型无法记住所有变量名。V6 的 **FactStore** 采用结构化绑定（`entityId + property`），规则评估时可以自动按命名空间拉取事实。
*   **对抗模型偏置**：V6 坚持使用 **确定性评价器（Deterministic Critic）**，即不调用 LLM 运行规则。这提供了一个**不可篡改、可重放**的对照组，用于发现模型的幻觉或偏见。

### 2. 能简化吗？
可以。这种复杂性主要源于对“极端严谨性”和“大规模数据”的支持。如果你的场景规则数量较少且逻辑简单，可以进行如下降级：

*   **弱化 Rule DAG 依赖**：如果规则之间没有相互推导关系，可以取消**拓扑排序**，将规则评估简化为单层列表扫描。
*   **合并推理阶段**：将 Planner 和 Executor 合并。在数据量小的情况下，不需要 Planner 预判子图，直接让 Agent 边查边走。
*   **简化 MCDA 模型**：取消复杂的权重动态校准，使用固定权重。甚至可以暂时不实现 `subsumedBy`（蕴含合并）逻辑，只要保证输入数据的唯一性即可。

### 3. 推荐的简化/实施方案
如果你觉得 V6 全量实现太重，建议按照以下**“三步走”**的简化路径进行：

#### 第一步：核心资产——FactStore（必做）
**这是从 V5 升级到 V6 最关键的一步，不能省略**。
*   **做法**：将扁平的 `Record<string, any>` 替换为带 `entityId` 和 `property` 的 **FactStore**。
*   **收益**：解决了命名空间冲突，且由于事实有了 `source`（来源），你的推理链条立刻变得可审计。

#### 第二步：评价逻辑——确定性 Critic（简化版）
*   **做法**：建立一个独立的 `scoreCandidates` 函数，仅在后台运行。它不使用 LLM，只读取 FactStore 里的事实，并根据固定权重相加。
*   **关键点**：**不要让 LLM 在推理过程中看到中间分**，防止它绕过系统。

#### 第三步：冲突显显式化——Reconciler（基础版）
*   **做法**：当系统自动评分的结果与模型给出的结论（ID）不一致时，在 UI 上打一个**警告标记**。
*   **收益**：不需要复杂的冲突原因分析，只要让用户知道“系统和模型打架了”就能极大地提升系统的可信任度。

**总结建议**：保留 **FactStore（事实绑定）** 和 **确定性 Critic（双轨判决）** 这两个核心骨架，因为它们是“可工程化”的基石。其余的 DAG 拓扑排序、动态权重校准、子图剪枝等功能，可以等规则规模超过 50 条后再考虑引入。

# 2

> 上述简化方案中，Direction 映射（risk_up/risk_down）是否保留？TODO 提到它属于 MCDA 复杂性，但它解决了 V5 的评分荒谬问题。

在建议的简化方案中，**Direction 映射（如 `risk_up` / `risk_down`）应当被保留**。

虽然它增加了 MCDA 的实现复杂性，但它是解决 V5 “评分荒谬”问题的核心机制，是系统从“盲目计数”转向“语义对齐”的关键。以下是基于源文件的详细分析：

### 1. 为什么它是“必须保留”的？
在 V5 的设计中，评分逻辑仅仅是“触发规则数 / 总规则数”，这导致系统无法识别规则的**打分方向**。
*   **V5 的缺陷**：如果一条规则是“人员负载过高”，系统无法自动理解它是支持 “HIGH 风险” 还是支持 “LOW 风险”，只能依靠代码中硬编码的字符串过滤（例如看到 "HIGH" 就加分，看到 "LOW" 就减分）。
*   **评分荒谬的根源**：由于缺乏方向感，系统会给出极其不符合直觉的分数（如 LOW 风险得分 0.67，但实际上触发的都是增险规则），导致模型（LLM）不得不强行改判，使规则系统形同虚设。

### 2. Direction 在 V6 中的核心价值
*   **解耦模型与分值**：`directionMapping` 让规则显式地描述自己“在推哪个候选方案”。例如，“风险上升”规则被触发时，会自动为 HIGH 候选加分，同时为 LOW 候选扣分。
*   **确保打分符合业务直觉**：通过方向映射，当所有“风险上升”规则触发时，系统判决（System Verdict）能确保 HIGH 排在第一位，从而通过回归测试中的 `direction-aware` 校验。
*   **解决“负面证据”缺失**：V5 很难处理“支持 LOW 风险”的正面证据。有了 Direction，系统可以区分“让风险变高的因素”和“让风险变低的因素”。

### 3. 简化实施方案建议
如果你希望在简化 MCDA 的同时保留这一核心功能，可以采取以下轻量化做法：

1.  **标签化映射**：不实现复杂的加权算法，只需在规则（Rule）定义中增加一个 `targetCandidateId` 或 `impact` 字段（例如 `impact: "positive_to_HIGH"`）。
2.  **简单极性计算**：
    *   如果规则触发且方向为 `up`，则目标候选方案得分 +1。
    *   如果规则触发且方向为 `down`，则目标候选方案得分 -1。
3.  **避免动态校准**：保留 Direction 映射，但暂时不引入 V6.4 中的 `weightOverrides`（基于用户反馈的动态权重调整），使用固定初始权重即可。

**总结**：`Direction` 映射不是“可选的复杂点”，而是**评分系统具备逻辑合理性的前提**。如果移除它，系统将退回到 V5 那个 LLM 必须频繁“越权修正”的不可靠状态。

# 3

> Direction 在 V6 中是如何工作的，举例说明

在 V6 架构中，**Direction（打分方向）** 是 **MCDA（多准则决策分析）评分器**的核心组成部分。它的引入主要是为了解决 V5 评分系统由于只看“规则触发数”而导致的打分与业务直觉脱节（如“风险上升”的规则触发却给“低风险”候选打高分）的严重漏洞。

### 1. 工作原理：从“盲目计数”到“语义对齐”
在 V6 中，评分不再是简单的加总，而是通过 `directionMapping` 明确每条规则在被触发时，应该支持哪个候选方案：

*   **方向映射 (`directionMapping`)**：在本体中定义规则（C）时，会显式标注该规则对不同候选（如 HIGH、MEDIUM、LOW）的影响方向。
*   **候选偏向**：当一条“风险上升”类规则触发时，它会为 **HIGH** 候选方案增加权重分，同时可能对 **LOW** 候选方案进行扣分。
*   **确定性计算**：这一过程在 **Deterministic Critic（确定性评价器）** 内部完成，不依赖 LLM 的即时判断，从而产生一个可重放、符合逻辑的“系统判决”（System Verdict）。

### 2. 举例说明
假设用户询问“项目 Portal 的风险大吗？”，系统中存在以下候选答案和规则：

*   **候选答案**：`cand_HIGH` (高风险), `cand_LOW` (低风险)。
*   **规则案例**：**"工程师过载阈值"**（如 workload > 85）。
    *   **V6 中的 Direction 定义**：该规则被标记为 `direction: "risk_up"`。
    *   **映射逻辑**：当此规则触发时，`cand_HIGH` 的得分增加（权重 × 严重程度），而 `cand_LOW` 的得分由于方向相反会被削弱。

**推理链对比：**
*   **V5 行为（无 Direction）**：如果 Agent 发现 Alice 工作量过载（触发了 1 条规则），由于 V5 只计算“触发数/总数”，系统可能会给 `cand_LOW` 打出 0.67 的高分（因为它可能没触发其他负面规则），导致结果极其荒谬。
*   **V6 行为（有 Direction）**：由于系统知道过载规则是推向 `risk_up` 的，得分会自动流向 `cand_HIGH`。最终系统判决会显示 `cand_HIGH` 的分值显著高于 `cand_LOW`，这与业务直觉（负载高 = 风险高）完全一致。

### 3. 关键收益
*   **防止模型越权修正**：在 V5 中，LLM 常因为看到系统给出的分数太幼稚而自行修改结论；引入 Direction 后，系统判决本身就具备了合理性，模型更倾向于与系统保持一致（Agree）。
*   **支持回归测试**：V6 包含专门的 `direction-aware` 测试：断言当所有“风险上升”规则触发时，HIGH 候选必须排在第一位。
*   **证据链清晰**：系统可以清楚地向用户解释：“由于触发了方向为‘风险上升’的过载规则，因此系统建议为高风险”。

# 4

> MCDA 评分器如何结合 Direction 映射来计算最终得分？
>
> 结合上面的例子，以步骤和数据描述

在 V6 架构中，**MCDA（多准则决策分析）评分器**取代了 V5 简单的“触发数/总数”逻辑。它通过 **Direction（打分方向）** 映射，将规则触发转化为对特定候选方案（Candidate）的加权支持或削弱。

以下结合“项目风险评估”的例子，通过步骤和数据描述 MCDA 评分器如何结合 Direction 计算最终得分：

### 核心背景数据
*   **候选方案**：`cand_HIGH`（高风险）、`cand_LOW`（低风险）。
*   **事实（FactStore）**：`Engineer(Alice).workload = 90`。
*   **规则（Rule）**：`rule_overload`（负载 > 85），`weight: 0.8`，`severity: 1.0`，`direction: "risk_up"`。
*   **方向映射逻辑**：`risk_up` 映射为：支持 `cand_HIGH`（正向），削弱 `cand_LOW`（反向）。

---

### 计算步骤与数据流转

#### 第一步：自动化事实提取与规则触发
系统判决器（Deterministic Critic）根据规则要求的 `appliesTo: ["Engineer"]` 和 `requiredFacts: ["workload"]`，自动从 **FactStore** 抓取 Alice 的事实（90）。
*   **逻辑检查**：`90 > 85`。
*   **状态**：规则 `rule_overload` 被标记为 **Triggered**。

#### 第二步：识别打分方向（Direction Mapping）
评分器查询规则定义，发现该规则属于 `risk_up` 方向。它会查找预设的方向语义映射表：
*   **对于 `cand_HIGH`**：该规则是“正向证据”（Impact = +1）。
*   **对于 `cand_LOW`**：该规则是“负向证据”（Impact = -1）。

#### 第三步：计算各候选方案的原始加权分
系统对每个候选方案独立计算其得分总和：
*   **`cand_HIGH` 的加权计算**：
    *   计算公式：`score += weight (0.8) * severity (1.0) * impact (+1)`。
    *   结果：**+0.8**。
*   **`cand_LOW` 的加权计算**：
    *   计算公式：`score += weight (0.8) * severity (1.0) * impact (-1)`。
    *   结果：**-0.8**。

#### 第四步：检查硬约束（Veto 机制）
如果存在 `hard_constraint` 类规则被触发（如“项目必须有负责人”），且候选方案违反了该约束，该候选方案会被直接执行 **Veto（否决）**，不论其加权得分多高，分值立即归零或标记为不可选。

#### 第五步：置信度校准（Confidence Scaling）
最终得分会乘以基于事实来源计算出的置信度。
*   **数据来源**：如果 Alice 的负载来自 `graph_property`（数据库直接读取），基础置信度为 1.0；如果是 `user_input`（用户随口说的），置信度可能只有 0.6。
*   **最终输出**：`cand_HIGH` 最终得分 = `0.8 * 1.0 = 0.8`。

---

### 结果对比：V5 vs V6/V7

| 维度 | V5 实现（无 Direction） | V6/V7 实现（有 Direction） |
| :--- | :--- | :--- |
| **计算逻辑** | 仅统计触发了多少规则。 | 根据 Direction 区分正负向影响。 |
| **打分表现** | 可能给 `cand_LOW` 打出 0.67 的高分（因为它没触发其他规则）。 | `cand_HIGH` 获得显著高分，`cand_LOW` 被扣分。 |
| **模型一致性** | 模型发现系统评分荒谬，会强行覆盖系统结论。 | 系统评分符合业务直觉，模型更倾向于同意（Agree）。 |

**总结**：MCDA 通过 **Direction** 映射解决了“事实与目标不一致”的问题。它确保了“过载”这一事实能够精准地为“高风险”加分，并为“低风险”扣分，从而产出一个具有业务逻辑合理性的确定性对照组。


# 5

> 上述简化方案中，Veto 机制（硬约束否决）是否保留？方案没明确提及，但它与 Direction 无关。

在针对 V6 架构的简化实施方案中，**Veto 机制（硬约束一票否决）应当被保留**。

虽然它与 Direction（打分方向）在数学逻辑上是独立的，但它在业务安全和系统可解释性方面具有不可替代的价值。以下是保留 Veto 机制的原因及其实施建议：

### 1. Veto 与 Direction 的本质区别
*   **Direction（打分方向）** 解决的是“**好与坏**”的相对评价，确保分数符合业务直觉（如负载高则风险增）。
*   **Veto（一票否决）** 解决的是“**行与不行**”的绝对边界，用于处理那些不可逾越的硬性准则（Hard Constraints）。
*   **保留原因**：即便有了正确的打分方向，如果一个方案虽然在软指标上得分很高，但违反了关键红线（如：项目没有负责人），它在现实业务中依然是不可接受的。

### 2. 在简化版 Critic 中的实施方式
在简化的 **确定性评价器（Deterministic Critic）** 中，Veto 的实现其实比完整的 MCDA 加权评分更简单：
*   **逻辑入口**：在进行加权求和之前，先检查所有标记为 `hard_constraint` 类型的规则。
*   **处理逻辑**：一旦任何一条硬约束被触发（Triggered），直接将该候选方案标记为 `vetoed: true`，其最终推荐分值可直接归零或降至最低，不再受后续加权评分的影响。
*   **技术优势**：这种“开关式”的逻辑不需要复杂的权重调优，非常适合作为系统初期的安全底线。

### 3. Veto 对“冲突协调（Reconciliation）”的关键意义
在 V6 设计中，Veto 是发现模型幻觉最有效的“警报器”：
*   **识别硬冲突**：如果系统（Critic）否决了一个模型（Executor）极力推荐的方案，这通常意味着模型忽略了某个关键的硬性准则。
*   **冲突解释**：在向用户展示冲突时，系统可以给出非常明确的理由：“**系统认为该方案命中了一票否决项（如：安全合规未通过），而模型未识别到此约束**”。

### 4. 推荐的“简化版评分组合”
将 Veto 纳入之前的简化路径后，你的 **确定性 Critic** 核心模块应包含：
1.  **事实读取**：从 `FactStore` 自动按需获取属性。
2.  **Veto 检查（保留）**：优先拦截触碰 `hard_constraint` 的方案。
3.  **加权打分（简化）**：保留 `Direction` 以确保正负向分值正确，但使用固定权重。
4.  **冲突暴露**：若 Veto 状态与模型建议不符，强制提醒用户。

**总结**：Veto 机制是决策系统的“**安全闸门**”。在简化方案中保留它，不仅能显著提升系统在极端情况下的可靠性，还能为用户提供最清晰的决策边界说明。

# 6

> Direction 映射配置方式？当前 scoring.ts 为每个候选（HIGH/MEDIUM/LOW）单独配置 risk_up/risk_down 映射值。简化后如何处理？

在 V6 的设计中，**Direction 映射**是解决 V5 “评分荒谬”问题（即无法区分正面与负面证据）的核心。针对你提出的三种简化配置方式，结合源代码中的 `ScoringProfile` 设计理念，以下是详细分析与推荐方案：

### 1. 三种方案对比分析

*   **方案 1：全局配置（基于 Intent/Profile）**
    *   **做法**：在 `ScoringProfile` 中定义一套全局映射表。例如：`risk_up` 统一对应 `HIGH: +1, LOW: -1`；`risk_down` 统一对应 `HIGH: -1, LOW: +1`。
    *   **优点**：极大减少了重复配置，规则定义只需关注业务属性（是增险还是减险），不需要关心具体的候选 ID。
    *   **缺点**：灵活性稍弱，难以处理某些只针对特定候选生效的特殊规则。

*   **方案 2：候选级配置（当前 V6 做法）**
    *   **做法**：每个 Candidate 对象内部自带一套针对不同方向的权重调整系数。
    *   **优点**：最严谨，能精准表达“规则 A 对 MEDIUM 是中性，对 HIGH 是强正向”的语义。
    *   **缺点**：配置极其冗余，随着候选数量增加，维护成本呈指数级上升，不符合“简化”目标。

*   **方案 3：简化为数值（+1/-1）**
    *   **做法**：规则直接声明 `direction: 1` 或 `-1`。
    *   **优点**：实现最简单。
    *   **缺点**：**风险最高**。它剥离了语义上下文，系统无法得知这个 `+1` 到底是加给哪个候选的。这会导致系统退回到 V5 的状态，即无法自动化执行“双轨判决”，因为系统失去了“方向感”。

---

### 2. 推荐方案：全局 Intent 映射（方案 1 的进化版）

为了兼顾简化与逻辑严密性，建议采用 **“基于意图（Intent）的全局映射方案”**。

#### 实现细节：
将方向映射逻辑从具体的规则或候选对象中抽离，放入 **`ScoringProfile`** 中：

1.  **定义规则方向（语义化）**：规则只声明业务方向，例如 `direction: "risk_up"` 或 `direction: "priority_down"`。
2.  **配置全局映射表**：在系统中按意图配置映射矩阵：
    ```ts
    // 简化后的全局配置示例
    const directionImpacts = {
      risk_assessment: {
        risk_up:   { HIGH: 1,  LOW: -1, MEDIUM: 0.5 },
        risk_down: { HIGH: -1, LOW: 1,  MEDIUM: 0.5 }
      },
      prioritization: {
        prio_up:   { P0: 1, P3: -1 },
        prio_down: { P0: -1, P3: 1 }
      }
    };
    ```
3.  **计算逻辑**：
    *   当 Critic 执行时，识别当前任务的 `Intent`（如风险评估）。
    *   计算公式：`candidateScore = Σ (rule.weight * directionImpacts[intent][rule.direction][candidateId])`。

### 3. 为什么这样处理最合适？

*   **消除冗余**：开发者在编写 100 条规则时，只需要写字符串 `"risk_up"`，而不需要为每条规则去配置三个候选的加减分。
*   **保留核心价值**：它依然保留了 V6 要求的 **“方向感”**。系统可以确切地知道“工程师过载”这条规则是在推高 `HIGH` 的分数并拉低 `LOW` 的分数。
*   **支持“双轨判决”**：因为有明确的方向贡献，`System Verdict`（系统判决）可以生成逻辑自洽的排名，从而在与 `Model Verdict`（模型判决）冲突时，能够给出明确的解释（如：“系统因方向映射判定为高风险，而模型由于语义幻觉判定为低风险”）。
*   **易于校准**：如果业务专家认为“增险规则”对 `MEDIUM` 的影响太大了，只需要改动全局映射表中的一个系数，全量规则的评分逻辑就会随之更新，实现了 **“可演化的校准”**。

**总结**：推荐采用 **全局映射（方案 1）** 并将其与 **Intent（意图）** 绑定。这既保持了实现的轻量化，又守住了 V6 防止评分荒谬的工程底线。

# 7

> missingFacts（缺失事实）处理？当前 scoring.ts 根据缺失比例降低 confidence。简化方案 没提及，但它属于“可审计性”。

保留并影响 confidence
保留 missingFacts 字段，confidence = 1 - missingRatio

在 V6 和 V7 的架构设计中，**不建议选择“选项 3：删除 `missingFacts`”**。

虽然删除它可以极大地简化代码，但它会直接破坏系统核心的**“可审计性（Accountability）”**和**“置信度校准（Calibrated Scoring）”**底座。以下是基于源代码和设计逻辑的深度分析：

### 1. 为什么不能假设 Executor 已收集完整？
在 V6 的 **“判决双轨制”** 逻辑下，Executor（LLM 驱动）和 Critic（确定性逻辑）是相互独立且互不信任的。
*   **Executor 的局限性**：LLM 在长链推理中可能由于 Token 限制或注意力偏移，漏查某些关键属性。
*   **Critic 的职责**：Critic（评价器）的作用就是作为一个“冷酷的裁判”，通过 `evaluate_rule` 自动检查 `FactStore`。如果 Critic 发现规则所需的 `requiredFacts` 缺失，它必须如实报告，而不是默认 Executor 已经尽力了。

### 2. 删除 `missingFacts` 的风险（针对选项 3）
*   **混淆“真阴性”与“无数据”**：如果删除该字段，系统将无法区分“项目风险低是因为所有指标都合格（全证据）”和“项目风险低是因为没查到任何风险证据（数据缺失）”。
*   **破坏置信度（Confidence）的客观性**：V6 明确规定置信度是**计算出来的**，而不是模型随口报的。置信度的重要组成部分就是**完整度（Completeness）**。如果删除了缺失事实的感知，系统输出的置信度将永远是 1.0，这是一种伪造的安全感。

### 3. 为什么选项 1 是推荐方案？
**保留并让其影响置信度（Option 1）** 是最符合 V6/V7 工程化决策助手目标的方案。

*   **实现透明决策**：在最终的决策输出中，系统必须显式列出 `uncertainties`。如果缺少高影响事实，系统必须告诉用户：“因为缺少 Alice 的工作量数据，该风险评估的置信度从 1.0 降至 0.7”。
*   **引导后续行动**：`missingFacts` 字段不仅是为了扣分，它还是 `next_best_queries` 的来源。系统可以据此告诉用户或 Agent 接下来应该去补查哪些信息。
*   **支持 Rule DAG 测试**：在 V6 的回归测试中，有一项关键测试就是“部分事实缺失时，硬约束必须触发不确定性，软准则必须降低置信度”。

### 总结建议
在简化实现时，你可以简化 `missingFactImpact` 的计算公式（例如简单的按比例扣分），但**必须保留该字段及其对置信度的惩罚逻辑**。

**推荐做法：**
1.  **数据层**：保留 `evaluate_rule` 返回的 `missingFacts: string[]`。
2.  **逻辑层**：采用 **$confidence = 1 - missingRatio$**（或者更精确地根据缺失规则的 `weight` 加权扣分）。
3.  **展示层**：若置信度低于阈值，在 UI 上显式标注“由于证据不足，结论仅供参考”。

这样做能保住 V6 架构中最重要的资产：**让决策过程变得可问责、可解释，而不是另一个不可控的黑盒。**

# 8

评分输出结构？简化后 SystemVerdict.ranking 的每个元素应该保留哪些字段？

在简化 V6 评分系统的实施中，推荐保留 **选项 1：保持不变**。

虽然你简化了底层的计算逻辑（例如将加权公式简化为简单的 `Σ(weight × directionContrib)`），但在 **`SystemVerdict.ranking`** 的输出结构中保留完整的字段，是确保系统具备“可解释性”和“双轨判决能力”的底线。

以下是保留这些字段的深度原因分析：

### 1. 为什么 `confidence`（置信度）不可删除？
即使数学公式简单了，置信度依然是 V6 的核心资产。
*   **区分“无证据”与“好证据”**：正如我们在上一轮关于 `missingFacts` 的讨论中提到的，`confidence` 告诉你这个分数是基于完整数据得出的（1.0），还是因为由于数据缺失而“凑巧”得到的（例如 0.5）。
*   **审计价值**：如果删除置信度，用户看到低分时无法判断是“真的没风险”还是“系统根本没去查相关的风险指标”。

### 2. 为什么 `triggeredRuleIds`（触发规则 ID）是灵魂？
V6 的设计目标是“决策支持”而非“单纯给分”。
*   **证据链条**：没有触发规则列表，评分就是一个黑盒数字。`triggeredRuleIds` 允许 UI 渲染出：“系统判定为高风险，是因为触发了‘工程师过载’和‘关键路径依赖’这两条规则”。
*   **冲突协调 (Reconciliation)**：当系统判决与模型判决不一致时，`Reconciler` 需要通过对比 `triggeredRuleIds` 与模型引用的证据来分析冲突原因（例如：模型看到了系统没识别到的证据）。

### 3. `rawScore` vs `normalizedScore` 的必要性
*   **`rawScore`（原始分）**：反映了加权求和的直接结果，便于开发者调试逻辑和权重是否合理。
*   **`normalizedScore`（归一化分）**：对最终用户极度友好。用户不需要理解 5.6 分代表什么，但他们能理解在 0 到 1 的区间内，0.8 代表风险极高。在多任务场景下，归一化分是实现横向对比的前提。

### 4. 字段保留建议（基于简化后的实现）

| 字段 | 保留建议 | 在简化方案中的含义 |
| :--- | :--- | :--- |
| **`candidateId`** | **必须** | 标识得分归属哪个方案（如 HIGH/LOW）。 |
| **`rawScore`** | **必须** | 简化公式 $Σ(weight \times directionContrib)$ 的直接结果。 |
| **`normalizedScore`**| **推荐** | 将 `rawScore` 映射到 区间，便于 UI 展示。 |
| **`confidence`** | **必须** | 基于 $1 - missingRatio$ 计算，反映数据完整度。 |
| **`triggeredRuleIds`** | **必须** | 用于解释“为什么是这个分”，支撑证据链显示。 |

### 总结
**Option 1** 虽然看起来数据量大，但它定义的每一项都是为了让决策过程**可解释（Interpretable）**和**可审计（Accountable）**。

如果你选择了 **Option 2 或 3**，虽然代码变短了，但系统会退化回 V5 的状态：用户只能看到一堆冷冰冰的数字，而当系统给出荒谬结论时（例如因为数据缺失导致的低分），用户将完全失去发现并修正幻觉的机会。
