MCDA（Multi-Criteria Decision Analysis，多准则决策分析）评分器在 V6 架构中被引入，是为了彻底解决 V5 阶段决策评价体系中“算法过于幼稚”和“模型静默胜出”这两个核心痛点。

以下是 MCDA 评分器的来龙去脉及其在系统中的演进分析：

### 一、 起源：V5 评分逻辑的崩塌 (The Problem)
在 V5 版本中，系统虽然引入了显式规则（C），但其评价工具 `evaluate_candidates` 的逻辑非常简单，即**“触发规则数 / 总规则数”**。

这种设计导致了几个结构性问题：
*   **脱离业务直觉**：由于没有权重（Weight）和严重程度（Severity）的概念，系统给出的分数往往无法反映真实的风险。例如，一个命中了“关键风险”的候选方案，可能因为没命中小规则而得到高分。
*   **模型“越权”修正**：当 LLM 发现系统给出的 `LOW=0.67` 打分极其荒谬时，它会通过最终答案直接推翻系统结论（硬改判为 HIGH）,。这导致确定性规则系统形同虚设，决策重新回到了“只看 LLM 自由发挥”的黑盒状态。
*   **缺乏“方向感”**：V5 的打分函数无法识别规则是“加分项”还是“减分项”，只能靠 label 字符串硬编码处理，扩展性极差。

### 二、 核心设计：MCDA 的引入 (The Solution)
为了实现“可校准”的评分，V6 将简单的计数逻辑替换为成熟的 **MCDA 评分框架**,。

其核心组件包括：
1.  **规则属性升级**：每条规则现在拥有 **Weight（权重）**、**Severity（严重程度）** 和 **Direction（打分方向）**,。
    *   **Direction Mapping**：显式描述规则是在推哪个候选（例如：“风险上升”规则对 HIGH 候选加分，对 LOW 候选扣分）,。
2.  **否决权机制 (Veto)**：不再只是“扣分”。当一个候选方案命中了标记为 `hard_constraint` 的否决条款时，该方案将直接出局，不再参与加权计算。
3.  **置信度计算**：分数的置信度不再由模型随口报告（如 0.95），而是基于 **FactStore** 中事实的来源、有效期及缺失事实的影响力（missingFactImpact）自动计算得出,。

### 三、 运行机制：确定性 Critic 与 Rule DAG
MCDA 评分器并不是独立运作的，它运行在 **Deterministic Critic（确定性评价器）** 内部。

其工作流程如下：
*   **拓扑排序**：通过 **Rule DAG**，系统首先运行“推理规则”生成推导事实，接着检查“硬约束”执行否决，最后才是“软准则”的加权打分,。
*   **自动化绑定**：评分器会自动从 **FactStore** 中拉取属性值进行评估，模型不再需要手动拼凑变量名给规则。
*   **双轨判决 (Dual-track Verdict)**：系统生成 MCDA 评分（System Verdict），LLM 生成感性判断（Model Verdict）。如果两者发生冲突，系统会显式向用户暴露冲突，而不是让模型静默覆盖结论,。

### 四、 演进与迁移：从 V5 到 V6.5
1.  **V6.1 迁移**：引入 Rule DAG 和 MCDA 评分，核心目标是修正 V5 中打分荒谬的问题。
2.  **V6.4 反馈闭环**：MCDA 的权重不再是死板的。系统通过用户反馈（如“系统对”或“模型对”）生成 `calibration log`，并据此调整 `weightOverrides`（权重覆盖），使评分系统具备自我进化的能力,。
3.  **V6.5 诊断模式扩展**：在诊断（Diagnostic）模式下，MCDA 进化为基于**归因评分（Causal Attribution）**的机制。它不再仅仅是加权和，而是基于 **But-for 测试**（即“如果没有 X，结果还会发生吗”）来评估原因的必要性和充分性,。

**总结来说：**
MCDA 评分器将决策评价从“模糊的概率猜想”提升到了**“可解释的工程化评估”**。它充当了系统中的“定海神针”，确保即使 LLM 产生幻觉，系统也能依据确定的业务准则给出可靠的对照基准,。
