在 V5 到 V7 的演化过程中，**EventStore** 的引入是系统从“前向预测”向“后向归因”进化的里程碑。它的出现解决了单一快照无法处理时间线和因果关系的局限性。

以下是 EventStore 的来龙去脉分析：

### 1. 起源：从 V5/V6 的“静态快照”说起
在 V5 和 V6 的主体设计中，系统主要关注 **前向（Predictive）推理**。
*   **V5 的局限**：事实（Facts）是扁平的 KV 对，没有时间概念。
*   **V6 的改进**：引入了 **FactStore**，为事实增加了 `validUntil`（有效期）和 `source`（来源），实现了事实的结构化绑定。
*   **遗留问题**：FactStore 本质上是“现在”这一时刻的投影。它能回答“现在风险大吗？”，但无法回答“为什么上周项目延期了？”。

### 2. 核心动因：V6.5 “双模式”扩展的需求
为了支持 **诊断（Diagnostic/Abductive）模式**，系统必须具备回顾过去的能力。
*   **推理方向的改变**：诊断模式需要从已发生的“结果（Outcome）”反推“原因（Causes）”。
*   **时间维度的必要性**：后向推理必须建立在时间线和事件历史之上。
*   **架构升级**：为了不破坏 V6 的现有结构，设计者提出了 **EventStore**，将 FactStore 定义为 `EventStore.factsAt(now)` 的一个切片投影。

### 3. EventStore 的核心功能设计
EventStore 不仅仅是一个数据库，它是具备“时间旅行”能力的推理底座：
*   **时间维度（Time-travel）**：通过 `validFrom`、`validUntil` 和 `observedAt` 等字段，EventStore 可以还原图中任何实体在历史任意时刻的状态。
*   **反事实推理（Counterfactuals）的支持**：
    *   **eraseEvent**：这是 EventStore 的关键能力。在进行“But-for”测试（即：如果没有 X，事情还会发生吗？）时，系统会从 EventStore 中“擦除”某个事件，然后重放因果链。
    *   **归因评分**：基于擦除事件后的重放结果，计算原因的必要性和充分性。

### 4. 演进关系对比
从架构演进上看，三者的关系如下：

| 阶段 | 存储组件 | 核心能力 | 场景 |
| :--- | :--- | :--- | :--- |
| **V5** | `Record<string, any>` | 简单的 KV 存储 | 处理小规模 Demo |
| **V6** | **FactStore** | 结构化绑定、实体命名空间 | 前向决策辅助（Predictive） |
| **V6.5/V7** | **EventStore** | **时间窗口、事件流、反事实重放** | 后向事故归因（Diagnostic/RCA） |

### 5. 实现现状（V7 阶段）
在 V7 的具体实现中，EventStore 已经成为了支撑 **判决双轨制** 和 **可校准评分** 的底层基础设施：
*   **冲突解决**：在构造 EventStore/FactStore 时，系统会依据置信度（Confidence）优先原则解决事实冲突。
*   **工具集成**：新增了 `query_events` 和 `record_event` 等工具，允许 Agent 直接操作时间轴上的事件。

**总结来说**，EventStore 的产生是为了让 Agent 拥有**“记忆”和“追溯”**的能力。它将事实从静态的数字变为了动态的事件流，使得系统能够跨越时间维度，在复杂的因果网络中寻找问题的根源。
