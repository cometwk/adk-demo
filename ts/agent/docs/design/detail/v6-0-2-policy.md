# 语义校验与事实绑定防御机制

在 V6 及更高版本的架构中，**PolicyContext（策略上下文）** 是通过中段推理层与工具层的深度集成来实现的。其核心逻辑在于将权限意识（Policy Awareness）从 AI 的“自律”转变为系统的“强约束”。

以下是其具体工作机制的详细分析：

### 1. 工具层装饰器强制执行 (`withPolicy`)
系统在 `policy/filters.ts` 中提供了一个名为 **`withPolicy(tool, ctx)`** 的装饰器。
*   **注册与拦截**：所有与图相关的工具（如 `inspect_node`, `query_neighbors`）在注册到 Executor 时，都会被这个装饰器包装。
*   **运行时过滤**：每当 Agent 调用这些工具的 `execute` 方法时，装饰器会首先检查当前的 `PolicyContext`。这意味着 Agent 无法通过直接调用工具来绕过权限规则。

### 2. 敏感字段的遮蔽 (Masking) 与过滤
针对不同的图操作工具，`PolicyContext` 实施了差异化的过滤策略：
*   **`inspect_node`（查看节点）**：
    *   **实体级过滤**：直接剪掉（Clip）当前用户无权访问的 `deniedEntityIds`。
    *   **字段级遮蔽**：对本体中标记为 `sensitiveProperties`（如薪资、私人电话等 PII 字段）执行 **Masking（遮蔽）** 处理，返回脱敏后的内容。
*   **`query_neighbors`（查询邻居）**：
    *   邻居列表会根据用户的 **`scope`（授权范围）** 自动过滤，Agent 根本看不到范围外的连接关系。
*   **`aggregate_facts`（聚合事实）**：
    *   系统在进行聚合计算（如计算平均负载）前，会先过滤掉 Agent 无权访问的实体数据，确保聚合结果不包含越权信息。

### 3. 禁止脱敏数据进入证据链 (Evidence Protection)
为了防止敏感信息在推理过程中“泄露”到永久性的决策轨迹中，系统对 **`record_evidence`** 工具进行了严格限制：
*   **二次校验**：当 Agent 尝试将某个事实记录为证据时，系统会检查该字段是否已被标记为 `redacted`（已脱敏）。
*   **物理隔离**：系统**禁止**将任何被遮蔽或脱敏的数据写入证据链。这确保了即使是经过脱敏后的“占位符”也不会留在持久化的 `DecisionTrace` 中，从而彻底消除了数据泄露的风险。

### 4. 关键设计价值
这种设计的核心价值在于**职责分离**：
*   **Agent 无需感知权限逻辑**：Prompt 不再需要长篇大论解释“哪些能看哪些不能看”，简化了模型任务。
*   **系统级的安全底线**：通过在工具层拦截，系统确保了即使 LLM 产生幻觉尝试访问敏感数据，也会收到 `ok: false` 配合 `permission_denied` 的标准返回，保证了决策过程的可审计性与合规性。
