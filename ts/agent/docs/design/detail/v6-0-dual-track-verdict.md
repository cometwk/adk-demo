# 判决双轨（Dual-track Verdict）

> 注意：目前的实现，有点特化, `风险评估领域` 先这样
>  更干净的做法可能是：
>
>  1. 把 diagnoseConflict 拆分成两部分：
>    - 通用部分：判断 ordinal 高低（需要注入 scale 配置）
>    - 领域部分：将 ordinal 差异翻译成语义（higherRisk → "模型更激进"）
>  2. 或者更激进一点：承认每个领域有独立的 Reconciler 实现
>    - 通用的只有 agree 判断（ID 匹配）
>    - 冲突诊断和解释完全由领域模块负责


在 V6 的架构设计中，**判决双轨（Dual-track Verdict）** 是为了解决 V5 中“模型随意推翻系统打分”的问题而引入的核心机制,。它通过两条独立的评估路径并行运行，并显式对比两者的结果来确保决策的可靠性,。

以下是其工作方式的详细说明及具体示例：

### 1. 两条并行轨道的功能分工
*   **轨道一：系统判决 (System Verdict)**
    *   **性质**：**确定性 (Deterministic)** 的规则系统,。
    *   **逻辑**：基于 FactStore 中绑定的结构化事实（如数值、状态），运行多准则决策分析（MCDA）和规则 DAG（有向无环图）,。它严格遵循预设的硬约束和权重，不会受到模型幻觉的影响。
*   **轨道二：模型判决 (Model Verdict)**
    *   **性质**：**语义化 (Semantic)** 的综合判断,。
    *   **逻辑**：由 LLM 执行。它不仅看数据，还会考虑非结构化的背景信息、语义关联以及多条规则之间的复杂交织。

### 2. 核心组件：协调器 (Reconciler)
在两条轨道得出结果后，系统不会简单地取平均值，而是通过 **Reconciler** 进行比对：
*   **一致（Agree）**：如果系统和模型结论一致（例如都认为风险等级为 HIGH），系统输出**高置信度结论**,。
*   **冲突（Conflict）**：如果两者结论不同，系统**禁止模型静默覆盖系统结论**，而是必须显式向用户暴露冲突,。

---

### 3. 具体场景举例：评估“项目延期风险”

#### **场景设定**：
项目中有一位核心工程师 Alice，其当前负载（workload）为 90%，且该项目依赖的一个底层接口尚未交付。

#### **双轨执行过程**：
1.  **系统判决轨迹 (Track 1)**：
    *   **触发规则**：命中 `rule_workload_threshold` (硬约束：负载 > 80% 即为高风险)。
    *   **计算结果**：由于命中了“硬约束”的 Veto（否决）条款，系统判定结论为 **HIGH**,。
2.  **模型判决轨迹 (Track 2)**：
    *   **综合推理**：模型查到 Alice 虽然负载高，但她拥有该领域 10 年经验，且该项目目前的非关键路径任务较多。
    *   **提议结论**：模型认为通过调整优先级可以化解，给出结论为 **MEDIUM**。

#### **协调与输出 (Reconciliation)**：
*   **冲突暴露**：系统发现 System (HIGH) 与 Model (MEDIUM) 不一致。
*   **因果诊断**：Reconciler 自动推断冲突原因（likelyCause），例如：`system_veto_detected`（系统命中了硬性否决项，而模型未充分识别）,。
*   **用户界面呈现**：
    *   **提示**：“系统与模型判断不一”。
    *   **详情**：“系统依据负载规则（>80%）判定为高风险；模型认为专家经验可降低风险至中等。”。
    *   **决策入口**：系统保留 [我有不同看法] 或 [采纳系统/模型] 的按键，将最终裁决权交给人类专家,。

---

### 4. 为什么这样设计？（设计意图）
*   **对抗模型偏置**：在 V5 中，LLM 可能会因为觉得系统评分“不合理”而偷偷修改最终答案，导致系统规则形同虚设,。V6 通过双轨制强迫模型在“透明的规则框架”下工作。
*   **可解释性与安全性**：当发生冲突时，这种“显式呈现”为业务专家提供了一个校准样本。如果专家多次认为模型对而系统错，则可以据此调整系统规则的权重（ScoringProfile）,。
*   **判决透明化**：它将决策从一个黑盒变成了两个透明轨道的对抗与对齐，是实现“可工程化、可问责”决策系统的底层保障。
