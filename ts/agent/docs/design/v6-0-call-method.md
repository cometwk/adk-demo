在 V6 的架构设计中，针对“模型使用语义荒谬的参数（如全零）盲调方法”这一结构性问题，系统通过**工具层的前置校验**、**事实绑定机制**以及**职责分离策略**进行了彻底解决。

具体解决方案如下：

### 1. 增强 `call_method` 的前置条件断言 (Precondition Assertion)
在 V6 中，工具层不再只是机械地检查参数 Schema（类型是否正确），而是引入了**语义校验**。
*   **显式声明约束**：每个方法都可以声明 `preconditions`（前置条件）。例如，`evaluateRisk` 方法可以要求 `teamLoad` 必须是 `must_be_positive`（必须为正数），或者 `seniorCount` 必须是 `must_be_in_facts`（必须在事实库中存在）。
*   **工具层拦截**：当模型尝试以 `{teamLoad: 0, seniorCount: 0}` 调用时，工具层会在执行前根据这些断言进行检查，直接拒绝“语法对但语义错”的调用，从而防止“证据链中毒”。

### 2. 从“手动传参”转向“自动事实绑定” (Fact Binding)
V6 引入了 **FactStore**，将事实从模型随口说的变量变成了受控的结构化记录。
*   **强制来源检查**：任何用于调用的事实必须通过 `bind_fact` 写入 FactStore，并标注 `source`（来源）。
*   **禁止凭空捏造**：系统要求方法调用的参数必须来自已绑定的事实。如果模型没有通过 `inspect_node` 等工具获取真实的 `seniorCount` 并将其绑定，它就无法通过校验来调用相关方法。这迫使模型必须先去图中“寻找证据”，而不是在没有数据时胡乱猜测。

### 3. 限定 Executor 职责与引入 Planner 预判
V6 改变了对 LLM（Executor）的指令策略，从源头上减少了盲调的可能性：
*   **职责限定**：Executor 的职责被严格限定为“探索图”和“绑定事实”，而不再要求模型自己打分或处理复杂的逻辑评估。Prompt 会明确告知模型：“最终评分由系统完成，你的工作是将事实绑定齐全”。
*   **Planner 预判路径**：V6 增加了 **Planner** 阶段。Planner 会预先研读本体和意图，预判出“必须先检查子图负载，才能评估风险”。这避免了模型在没有收集齐必要信息时就匆忙调用下游方法的“反复”行为。

### 4. 确定性 Critic 的“判决双轨制”
即便模型绕过了某些限制给出结论，V6 的 **Deterministic Critic（确定性规则系统）** 也会作为对照组运行。
*   **逻辑兜底**：Critic 基于 FactStore 中的硬性数据（如真实的负载数值）运行确定性规则，其结果不会被模型编造的数字所干扰。
*   **冲突暴露**：如果模型因为盲调给出了错误的结论，而系统依据真实数据得出了不同的判决，**Reconciler（协调器）** 会将这种冲突显式暴露给用户，而不是让模型静默胜出。

**总结来说：**
V6 通过在工具调用前加 **“安检”（Preconditions）**、在调用时查 **“身份证”（Fact Binding）**、以及在调用后设 **“督查”（Critic）**，确保了模型无法再用荒谬的虚假数据毒害决策链条。
