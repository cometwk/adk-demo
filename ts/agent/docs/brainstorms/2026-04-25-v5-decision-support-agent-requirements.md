# V5：辅助决策 Agent 落地需求

## Problem Frame

V4 已经把手写 agent loop 替换为 AI SDK 原生 tool-calling loop，解决了历史丢失、畸形 JSON、手写 validator/executor 冗余等问题。但 V4 仍把系统定位为“图推理问答”：给定目标，模型探索图、调用方法、输出一个答案。

V5 的目标不同：系统要成为一个**面向模糊问题的辅助决策 Agent**。用户的问题可能没有唯一正确答案，例如“评估 project_portal 的综合交付风险”。系统需要基于本体 `G={E,R,T,C}` 生成候选判断、按需收集证据、应用规则和准则、表达不确定性，并帮助用户做决策。

当前 V4 到 V5 的主要差距是：

- `C` 没有一等建模，规则、约束、判断准则和解释模板藏在 method implementation 或自然语言描述里。
- `E` 仍以全量 node id 目录进入 System Prompt，不适合大图、隐私和权限场景。
- 输出偏单一结论，缺少候选答案、证据链、反证、不确定性和下一步建议。
- 方法参数 schema 暴露不稳定，模型可能先盲调再依赖错误恢复。

V5 应在保留 V4 的 AI SDK loop 基础上，补齐本体完整性和决策支持能力。

## Requirements

**本体完整性与 `C` 一等化**

- R1. 系统必须显式表达本体 `G={E,R,T,C}`，并区分类型/类集合 `T`、关系 schema 与关系事实 `R`、实体事实 `E`、约束/准则/规则集合 `C`。
- R2. `C` 必须支持至少五类规则：硬约束、软准则、推导规则、冲突处理规则、解释规则。
- R3. 每条规则/准则必须有稳定 ID、适用对象、描述、所需事实、规则类型，并能被工具查询。
- R4. 方法实现可以继续承载确定性计算，但其相关规则 ID、输入事实和解释依据必须能被查询或返回，避免规则完全黑盒化。

**渐进式实体披露**

- R5. System Prompt 不应注入全量实体目录；默认只暴露入口实体、类型 schema、关系 schema、规则摘要和工具说明。
- R6. 系统必须支持按需发现实体，包括基于入口实体查询邻居、按类型或关键词搜索节点、按关系方向过滤、分页返回。
- R7. 节点读取必须支持字段投影，允许只请求类型、属性、入边、出边、方法等部分信息。
- R8. 对高连接度节点或大图结果，系统必须能返回分页或摘要，而不是一次性返回全部邻居和属性。

**方法与工具契约**

- R9. 所有 tool result 必须使用统一结构区分成功、失败、空结果、参数错误和可重试错误。
- R10. 系统必须提供方法描述能力，让模型在调用方法前查询参数 schema、返回 schema、required facts 和相关规则。
- R11. 通用 `call_method` 可以保留，但不能作为模型理解方法参数的唯一入口；模型应先通过 schema/description 工具获得调用依据。
- R12. 工具错误必须可被模型和测试稳定识别，不能只依赖自然语言 error message。

**决策任务层**

- R13. 系统必须引入决策任务模型，用于表达问题、意图、入口实体、候选答案、证据、准则、不确定性和推荐判断。
- R14. 对模糊问题，系统应先识别问题意图，再生成多个候选答案，而不是直接收敛到单一结论。
- R15. 系统必须支持记录证据，每条证据应包含来源、相关实体、相关关系或规则、置信度和可解释陈述。
- R16. 系统必须支持根据显式准则比较候选答案，并保留支持证据、反对证据、触发规则和置信度。

**决策输出**

- R17. 最终输出必须包含推荐判断、备选判断、关键证据、触发规则/准则、不确定性和建议下一步补充的信息。
- R18. 当高影响事实缺失时，系统必须显式说明缺失项和可能影响，不能假装确定。
- R19. 输出应区分硬约束和软准则：硬约束用于边界，软准则用于比较和排序。
- R20. 默认输出应适合用户阅读；调试或审计模式可以展开完整 tool trace、evidence IDs 和 rule IDs。

**V4 兼容与演进**

- R21. V5 必须保留 V4 的核心运行方式：AI SDK `generateText({ maxSteps, tools })` 负责多轮 tool-calling loop。
- R22. V5 不应恢复 V3 的手写 loop、`NextAction` JSON 协议或手写 executor/validator 闭环。
- R23. V5 应允许现有 V4 图节点、属性、边和 `@agentMethod` 继续作为基础能力存在。
- R24. V5 的新增能力应能分阶段落地：先显式 `C` 和方法描述，再渐进式实体披露，再证据链和候选答案比较。

**验证与可审计性**

- R25. 系统必须支持 golden trace 验证：同一模糊问题的推理步骤可以不同，但结构化输出必须满足稳定 schema。
- R26. 测试必须覆盖 tool contract、本体规则查询、方法 schema 描述、渐进式实体披露和模糊问题输出。
- R27. 示例场景必须继续围绕 `project_portal` 综合交付风险，展示 V4 单答案推理与 V5 辅助决策输出的差异。

## Success Criteria

- System Prompt 不再列出所有 node id，只包含入口实体、类型/关系 schema、规则摘要和工具策略。
- 模型在调用 `evaluateRisk` 之前能查询到 `teamLoad`、`seniorCount` 等 required params，而不是先盲调失败。
- 对“评估 project_portal 的综合交付风险”这类模糊问题，输出包含推荐判断、备选判断、证据、不确定性和下一步建议。
- `C` 中的规则/准则可以通过工具查询，并能在最终回答中被引用。
- 高连接度或大图查询不会一次性展开全部实体，工具结果支持分页、过滤或摘要。
- 测试能断言结构化 tool result 和结构化 decision output，而不依赖自然语言逐字匹配。

## Scope Boundaries

- 不把项目改造成完整生产级知识图谱平台；V5 仍是 demo 原型验证。
- 不引入远程图数据库、持久化存储、权限系统或真实组织数据。
- 不实现多 Agent 协作；多 Agent 只有在单 Agent 的规则、证据和候选答案结构稳定后再考虑。
- 不把所有业务逻辑迁移成规则 DSL；需要解释、查询和组合的规则一等化，复杂确定性计算仍可留在方法中。
- 不追求全自动决策；系统定位是辅助用户判断，而不是替用户承担最终决策责任。
- 不要求自然语言输出完全确定；要求结构化输出和关键证据稳定。

## Key Decisions

- **完整 V5 范围**：本轮落地需求覆盖显式 `C`、渐进式 `E`、证据链、候选答案评分和测试策略，而不是只做 V5.0 MVP。
- **保留 AI SDK loop**：V4 已解决运行时闭环问题，V5 只增加本体和决策层，不回退到手写 loop。
- **`C` 作为可查询对象**：规则不仅是 prompt 文本，也不完全藏在 method 中；规则 ID、适用对象和解释依据必须可见。
- **入口实体优先**：V5 不再通过全量节点目录提供规划空间，而是通过入口实体和工具逐步发现相关事实。
- **辅助决策输出优先于唯一答案**：V5 的价值在于帮用户比较判断、理解依据和识别缺失信息。

## Alternatives Considered

- **只实现 V5.0 MVP**：优点是实现风险低；缺点是不能验证渐进式实体披露和完整决策输出，无法回答完整 V5 的核心假设。
- **直接做多 Agent**：看起来符合长期扩展方向，但会在 `C`、证据、候选答案结构未稳定前放大不确定性，因此暂不采用。
- **把所有规则做成 DSL**：可解释性强，但会把 demo 变成规则引擎项目，超出当前目标。

## Dependencies / Assumptions

- V4 的 `inspect_node`、`query_neighbors`、`call_method` 和 AI SDK tool-calling loop 已作为基础方向成立。
- 项目仍允许按版本目录演进，例如后续实现可放在 `src/v5`，并保留 `src/v4` 作为对照。
- 当前 demo 的主要验证场景仍是项目交付风险评估。
- 规划阶段需要检查实际代码结构后再决定具体文件路径、测试框架和实现顺序。
