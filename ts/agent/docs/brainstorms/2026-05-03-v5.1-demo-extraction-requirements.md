# v5.1 Demo 场景独立化重构

## Problem Frame

当前 `src/v5.1` 目录混合了通用框架代码和场景特定代码（工程组织交付风险评估），导致：
- 新场景难以复用框架，需要复制大量代码
- 框架代码和场景代码耦合在一起，难以区分边界
- 未来添加新场景（如 demo/ex2）时无法清晰复用

需要将场景特定代码独立到 `demo/ex1` 目录，保留通用框架在 `src/v5.1`，使新场景可以直接复用框架。

## Requirements

**框架层保留（src/v5.1）**
- R1. `runtime/` 目录保持不变（types.ts, graph.ts, decorator.ts, registry.ts）
- R2. `agent/` 目录保持不变（run.ts, prompt.ts, output.ts, tools/）
- R3. `ontology/decision.ts` 保持不变
- R4. `ontology/schema.ts` 仅保留通用类型定义（Ontology, TypeSchema 等），移除 `projectOntology`
- R5. `ontology/constraints.ts` 仅保留基础设施代码（类型定义、registerConstraint、evaluateConstraint 等），移除 `registerProjectPortalConstraints`

**场景层独立（src/v5.1/demo/ex1）**
- R6. 创建 `demo/ex1/` 目录，作为"工程组织交付风险评估"场景
- R7. `demo/ex1/schema.ts` 包含 `projectOntology` 场景本体定义
- R8. `demo/ex1/constraints.ts` 包含 `registerProjectPortalConstraints()` 和场景特定约束定义
- R9. `demo/ex1/seed.ts` 包含 Engineer/Team/Project 实体类和 `seedGraph()` 函数
- R10. `demo/ex1/index.ts` 作为场景入口，导入框架模块和场景模块，运行决策 Agent

**导入路径更新**
- R11. 场景代码导入框架代码时使用相对路径向上引用（如 `../../ontology/schema`）
- R12. 移除原 `src/v5.1/index.ts`（入口已移动到 demo/ex1）

## Success Criteria
- 运行 `demo/ex1/index.ts` 功能与原 `src/v5.1/index.ts` 一致
- 框架层代码不包含任何场景特定引用
- `demo/ex2` 可以直接复用 `src/v5.1` 框架层（通过复制 ex1 结构作为模板）

## Scope Boundaries
- 不改变框架层代码的功能逻辑
- 不添加新的场景功能（ex2 是下一个话题）
- 不优化或重构现有代码实现

## Key Decisions
- 框架代码保留在 `src/v5.1`（而非移动到 `src/lib`）：用户选择保留在 v5.1 目录
- demo 目录放在 `src/v5.1/demo`（而非项目根目录）：与框架版本绑定更紧密
- 约束基础设施和场景配置分离：保留 registerConstraint 等在框架层，场景约束移到 demo

## Dependencies / Assumptions
- TypeScript 相对路径导入可正确解析
- `tsconfig.json` 已配置适当的路径映射（如需要）
