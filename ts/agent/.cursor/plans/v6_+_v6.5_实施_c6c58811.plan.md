# V6 + V6.5 同时实施计划

## 总体策略

同时实施的核心优势：`FactBinding` 从第一行就含 `validFrom/validUntil/observedAt`，`EventStore` 作为唯一底层存储，`FactStore` 是 `EventStore.factsAt(now)` 的投影。predictive 和 diagnostic 共享同一份基础，mode-specific 逻辑在独立文件里。

全部新代码落在 `src/v6/`，V5 完全不动。

## Phase 1：共享基础层（串行，其余各 Phase 依赖于此）

**目标**：建好两种模式共同依赖的所有类型和运行时。

- [`src/v6/runtime/types.ts`](src/v6/runtime/types.ts)
  - 复用 V5 的 `ToolResult<T>` / `PageInfo`
  - `FactBinding`：含 `entityId, property, value, source, confidence, validFrom, validUntil, observedAt`（V6.5 时间字段从第一版就建入）

- [`src/v6/runtime/eventStore.ts`](src/v6/runtime/eventStore.ts)
  - `Event`：`id, type, occurredAt, actorId, affectedEntities, payload, derivedBindings`
  - `EventStore`：`addEvent / timelineFor / factsAt(entityId, t) / eraseEvent(id): EventStore / snapshot()`
  - `FactStore = EventStore.factsAt(now)` 的适配器（predictive 路径用这个入口）

- [`src/v6/ontology/schema.ts`](src/v6/ontology/schema.ts)
  - 复用 V5 的 `TypeSchema / RelationSchema / Ontology`，加 `version: string`

- [`src/v6/ontology/rules.ts`](src/v6/ontology/rules.ts)
  - `Rule`：`id, version, kind, appliesTo, requiredFacts[{property, scope}], direction, weight, severityFn, veto, dependsOn, subsumedBy, evaluator, explanation`
  - `RuleContext = { entityId?, facts: FactStore, graph: Graph }`
  - `RuleResult = { triggered, severity?, derivedFacts?, missingFacts? }`

- [`src/v6/ontology/ruleDag.ts`](src/v6/ontology/ruleDag.ts)
  - 拓扑排序：inference → hard → soft
  - `evaluateAll(facts, rules, graph): Map<ruleId, RuleResult>`
  - 蕴含合并：subsumedBy 规则不参与打分（仍报 triggered=true 用于解释）

- [`src/v6/ontology/causal.ts`](src/v6/ontology/causal.ts)（V6.5）
  - `CausalEdge`：`cause/effect/mechanism/typicalLag/strength/counterEvidence/relatedRuleIds`
  - `CausalPath = { edges, rootCause, finalEffect }`
  - `CausalGraph`：`backwardChain(outcome, maxDepth) / forwardChain / potentialCauses(outcome)`
  - **必须与 `Ontology.relations` 分开**

- [`src/v6/ontology/decision.ts`](src/v6/ontology/decision.ts)
  - `DecisionMode = "predictive" | "diagnostic"`
  - `DecisionTask`：mode-aware（predictive 必填 entryEntities；diagnostic 必填 outcome + timeWindow）
  - `SystemVerdict / ModelVerdict / Reconciliation / DecisionResponse`
  - `CandidateCause / AttributionResult / DiagnosticVerdict`（V6.5 类型一并建入）

- [`src/v6/policy/context.ts`](src/v6/policy/context.ts) + [`src/v6/policy/filters.ts`](src/v6/policy/filters.ts)
  - `PolicyContext`：`principal, scope, redaction, audit`
  - `withPolicy(tool, ctx)` 装饰器：mask sensitiveProperties、过滤 deniedEntities、阻止 redacted 字段进入 evidence

## Phase 2：Predictive 核心（V6，可在 P1 完成后立即开始）

**目标**：完整的前向决策链路可端到端跑通（修复 V5 demo 的三个问题）。

- [`src/v6/ontology/scoring.ts`](src/v6/ontology/scoring.ts)
  - `ScoringProfile`：`aggregation, veto, severityWeights, directionMapping, calibration`
  - `scoreCandidates(input): ScoredCandidate[]`
  - **direction-aware**：不靠 label 字符串硬编码，用 `directionMapping[candidateLabel][rule.direction]`
  - hard_constraint veto 命中 → 候选直接出局

- [`src/v6/agent/criticPredictive.ts`](src/v6/agent/criticPredictive.ts)
  - `runPredictiveCritic(task, facts, ontology): SystemVerdict`（纯 deterministic，不调 LLM）
  - 内部走 `ruleDag.evaluateAll` + `scoring.scoreCandidates`

- [`src/v6/agent/tools/`](src/v6/agent/tools/)（predictive 工具集）
  - `graph.ts`：V5 工具全部保留，加 `withPolicy` 装饰；`inspect_node` 增加 `at?` 时间旅行参数
  - `method.ts`：V5 工具 + precondition assertion（阻止 `teamLoad:0` 盲调）
  - `facts.ts`：`bind_fact / lookup_fact`（写入/读取 FactStore，取代扁平 KV）
  - `rules.ts`：`inspect_rules` + `evaluate_rule(ruleId)`（细粒度，不打分）
  - `candidates.ts`：`propose_candidates`（保留）、`record_evidence`（confidence 改为 base + LLM modifier）
  - `counterfactual.ts`：`simulate({ mode: "what_if", overrides })`（snapshot override + 重跑 critic）
  - **移除** `evaluate_candidates`（V5 的问题根源）

- [`src/v6/agent/executor.ts`](src/v6/agent/executor.ts)
  - `generateText` + 上述工具集
  - Prompt 明确："你不需要知道中间分，你的职责是把 fact binding 集齐"
  - 输出 `{ facts: FactStore, candidates, modelVerdict, evidence }`

- [`src/v6/agent/reconciler.ts`](src/v6/agent/reconciler.ts)
  - 比较 `systemVerdict` vs `modelVerdict`
  - agree → `surfacedToUser: false`；conflict → `diagnoseDiff` 推断 `likelyCause`

- [`src/v6/agent/planner.ts`](src/v6/agent/planner.ts)（predictive planner）
  - 一次轻量 LLM 调用，**不能调用工具**
  - 输出 `ExplorationPlan`：`expectedSubgraphs, methodsToInvoke, rulesetOfInterest, estimatedSteps`

- [`src/v6/agent/prompt.ts`](src/v6/agent/prompt.ts)
  - `buildPredictiveSystemPrompt`：只给 schema + rules 摘要 + entry entities，不给中间分
  - `buildPlannerPrompt`

- [`src/v6/frontend/intent.ts`](src/v6/frontend/intent.ts)
  - 规则匹配（"风险" → risk_assessment，"优先" → prioritization）+ 小 LLM 兜底
  - mode 一级分类（predictive / diagnostic）
  - 返回 `{ mode, intent, confidence }`

- [`src/v6/frontend/entityLinker.ts`](src/v6/frontend/entityLinker.ts)
  - ID 模糊匹配 + 别名表 + 上下文最近实体

- [`src/v6/frontend/clarify.ts`](src/v6/frontend/clarify.ts)
  - 多候选 entity 时生成结构化选择题（不是 free-form chat）

## Phase 3：Diagnostic 扩展（V6.5，可在 P1 完成后与 P2 并行开始）

**目标**：同一架构下支持后向归因（RCA / post-mortem）。

- [`src/v6/ontology/attribution.ts`](src/v6/ontology/attribution.ts)
  - `scoreCauses(input): AttributionResult[]`
  - 四个维度：路径完整性 + but-for 必要性 + 充分性 + 时序合理性
  - `overdetermined` 检测：top-2 attribution 都 > 0.4 时标注
  - **attribution 不强制 sum=1**

- [`src/v6/agent/criticDiagnostic.ts`](src/v6/agent/criticDiagnostic.ts)
  - `runDiagnosticCritic(task, eventStore, causalGraph, candidates): DiagnosticVerdict`（deterministic）
  - 内部调用 `eventStore.eraseEvent` + 因果链重放做 but-for 测试（不暴露给 executor）

- [`src/v6/agent/tools/events.ts`](src/v6/agent/tools/events.ts)（diagnostic 专用工具）
  - `query_events(entityId, from, to, eventTypes?)`
  - `walk_causal_graph(seed, direction, maxDepth)`
  - `propose_causes([{label, causalPath, rationale}])`
  - `record_event({type, occurredAt, affectedEntities, payload})`

- [`src/v6/agent/plannerDiagnostic.ts`](src/v6/agent/plannerDiagnostic.ts)
  - 输出 `DiagnosticPlan`：`rootOutcome, backwardChains, eventsToReconstruct, candidateCauseSpace`
  - 沿 CausalGraph 反向遍历，不是向外展开依赖

- [`src/v6/agent/prompt.ts`](src/v6/agent/prompt.ts)（追加）
  - `buildDiagnosticSystemPrompt`：四条守则（outcome 已发生 / 相关≠因果 / 多因并存 / 警惕后见之明）

## Phase 4：集成与测试

**目标**：两条轨道打通，关键 success criteria 全部验证。

- [`src/v6/agent/critic.ts`](src/v6/agent/critic.ts)
  - 按 `task.mode` 路由：predictive → `criticPredictive`，diagnostic → `criticDiagnostic`

- [`src/v6/data/seed.ts`](src/v6/data/seed.ts)
  - 复用 V5 实体（alice/bob/project_portal/project_api/team_frontend/team_backend）
  - 新增事件时间线（`scope_added @2026-04-08`、`api_delivery_slip @2026-04-15`、`milestone_missed @2026-04-21`）
  - 新增因果图（5-10 条 CausalEdge：workload→burnout、dep_slip→blocked、scope_added→pressure、...）
  - `ruleFixtures.ts`：规则单测样例

- [`src/v6/index.ts`](src/v6/index.ts)
  - `runDecisionAssistant(userQuery, options)` —— 按 mode 路由到完整 pipeline
  - `runPredictiveSession` / `runDiagnosticSession` 内部函数

- 关键测试（按 success criteria）：
  - predictive：V5 demo 输入 → critic 输出 HIGH > MEDIUM > LOW
  - burnout 规则：alice/bob 各自独立评估，无 missingFacts 误报
  - precondition：`evaluateRisk({teamLoad:0})` 被拒绝
  - reconciliation：system/model 不一致时 conflict.surfacedToUser=true
  - diagnostic：用户问"portal 延期为什么" → mode=diagnostic，api_dep attribution 最高
  - but-for：eraseEvent(api_slip) 后 milestone_missed 概率下降
